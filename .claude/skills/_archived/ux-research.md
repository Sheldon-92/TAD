> 综合自 UX 研究最佳实践和用户体验方法论，已适配 TAD 框架

## TL;DR Quick Checklist

```
1. [ ] 研究目标清晰；选对方法（定性/定量）
2. [ ] 样本与偏差控制；知情同意
3. [ ] 模板化材料：访谈大纲/问卷/任务脚本
4. [ ] 记录与整理：逐字稿/可用性问题清单
5. [ ] 产出：研究计划/访谈笔记/洞察报告
```

**Red Flags:** 方法与目标不匹配、样本偏差、无任务脚本、仅轶事证据

## 触发条件

当用户需要进行用户研究、可用性测试、用户访谈、A/B测试分析或用户体验评估时，自动应用此 Skill。

## 用户访谈

### 访谈准备

```markdown
## 用户访谈准备清单

### 1. 研究目标定义
- [ ] 明确研究问题
- [ ] 确定目标用户群
- [ ] 定义成功指标

### 2. 访谈大纲

**开场 (5分钟)**
- 自我介绍
- 说明访谈目的
- 获得录音/录像许可
- 强调没有对错答案

**背景了解 (10分钟)**
- 基本信息
- 使用习惯
- 相关经历

**核心问题 (30分钟)**
- [主题1相关问题]
- [主题2相关问题]
- [主题3相关问题]

**总结 (5分钟)**
- 还有什么想补充的？
- 感谢参与
- 后续安排说明

### 3. 物料准备
- [ ] 访谈大纲
- [ ] 录音设备
- [ ] 笔记模板
- [ ] 原型/产品（如需要）
- [ ] 礼品/报酬
```

### 访谈问题设计

```markdown
## 访谈问题技巧

### 好问题的特征
✅ 开放式
✅ 中性、不引导
✅ 聚焦具体行为和经历
✅ 一次只问一个问题

### 问题类型

**行为问题** (了解实际行为)
- "上一次你[做某事]是什么时候？"
- "能描述一下你通常是怎么[做某事]的吗？"
- "你最近一次遇到[问题]时是怎么解决的？"

**态度问题** (了解观点和感受)
- "你对[某事]有什么看法？"
- "你觉得[某事]最重要的是什么？"
- "如果可以改变一件事，你会改变什么？"

**深挖问题** (获取更多细节)
- "能再详细说说吗？"
- "为什么会这样觉得？"
- "当时你是怎么想的？"
- "然后呢？"

### 避免的问题类型

❌ 引导性问题
- 不好: "你是不是觉得这个功能很有用？"
- 好: "你对这个功能有什么看法？"

❌ 假设性问题
- 不好: "如果我们加这个功能你会用吗？"
- 好: "你目前是怎么解决这个问题的？"

❌ 是非题
- 不好: "你喜欢这个设计吗？"
- 好: "这个设计给你什么感觉？"
```

### 启发式评估

```markdown
## Nielsen 十大可用性原则评估

### 1. 系统状态可见性
系统应该通过适当的反馈，在合理时间内让用户了解正在发生什么。

**检查点**:
- [ ] 加载状态是否清晰显示？
- [ ] 用户操作后是否有即时反馈？
- [ ] 进度条是否准确反映实际进度？

**发现**: [记录问题和位置]

### 2. 系统与现实世界匹配
使用用户熟悉的语言、词汇和概念，而非系统术语。

**检查点**:
- [ ] 术语是否易懂？
- [ ] 图标是否符合直觉？
- [ ] 信息组织是否符合用户心智模型？

**发现**: [记录问题和位置]

### 3. 用户控制与自由
用户经常误操作，需要"紧急出口"来离开不想要的状态。

**检查点**:
- [ ] 是否有撤销/重做功能？
- [ ] 是否可以取消操作？
- [ ] 是否可以轻松返回上一步？

**发现**: [记录问题和位置]

### 4. 一致性与标准
用户不应该困惑不同词语、情境或操作是否表示同一件事。

**检查点**:
- [ ] 视觉设计是否一致？
- [ ] 交互模式是否一致？
- [ ] 术语使用是否一致？

**发现**: [记录问题和位置]

### 5. 错误预防
比好的错误消息更重要的是预防问题发生。

**检查点**:
- [ ] 是否有输入验证？
- [ ] 危险操作是否有确认？
- [ ] 是否有智能默认值？

**发现**: [记录问题和位置]

### 6. 识别而非回忆
减少用户的记忆负担，让信息在需要时可见。

**检查点**:
- [ ] 操作选项是否可见？
- [ ] 是否提供足够的上下文？
- [ ] 帮助信息是否易于获取？

**发现**: [记录问题和位置]

### 7. 灵活性与效率
系统应同时满足新手和专家用户的需求。

**检查点**:
- [ ] 是否有快捷方式？
- [ ] 是否可以自定义常用操作？
- [ ] 是否支持高级用户加速操作？

**发现**: [记录问题和位置]

### 8. 美观与简约设计
界面不应包含不相关或很少需要的信息。

**检查点**:
- [ ] 是否有视觉噪音？
- [ ] 信息层级是否清晰？
- [ ] 重要内容是否突出？

**发现**: [记录问题和位置]

### 9. 帮助用户识别、诊断和恢复错误
错误消息应用简单语言表达，精确描述问题并提供解决方案。

**检查点**:
- [ ] 错误消息是否易懂？
- [ ] 是否说明错误原因？
- [ ] 是否提供解决方案？

**发现**: [记录问题和位置]

### 10. 帮助与文档
最好的系统无需文档即可使用，但必要时需提供帮助。

**检查点**:
- [ ] 帮助是否易于搜索？
- [ ] 帮助是否针对具体任务？
- [ ] 帮助步骤是否具体可执行？

**发现**: [记录问题和位置]

---

### 问题严重程度评级
| 级别 | 说明 |
|------|------|
| 0 | 不是可用性问题 |
| 1 | 表面问题，低优先级修复 |
| 2 | 次要问题，应该修复 |
| 3 | 主要问题，必须修复 |
| 4 | 灾难性问题，立即修复 |
```

## 最佳实践

```
✅ 推荐
□ 研究目标要明确、可验证
□ 用开放式问题深入挖掘
□ 观察行为而非只听态度
□ 用真实数据支持结论
□ 输出可执行的建议

❌ 避免
□ 引导性问题影响结果
□ 样本量太小做结论
□ 只报告正面发现
□ 研究发现没有后续行动
□ 假设用户会告诉你真相
```
