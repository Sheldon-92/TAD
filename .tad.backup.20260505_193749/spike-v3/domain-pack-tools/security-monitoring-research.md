### CLI Usage Examples

#### trivy

```bash
# Install
brew install trivy

# Scan a container image
trivy image python:3.11-slim

# Scan filesystem dependencies
trivy fs --scanners vuln,secret .

# Scan IaC (Terraform, CloudFormation, Dockerfile)
trivy config ./infrastructure/

# Generate SBOM in CycloneDX format
trivy image --format cyclonedx --output sbom.json python:3.11-slim

# Scan with severity filter (CI gate)
trivy image --severity HIGH,CRITICAL --exit-code 1 myapp:latest

# Output as SARIF for GitHub Security tab
trivy image --format sarif --output results.sarif myapp:latest
```

#### grype

```bash
# Install
brew tap anchore/grype && brew install grype

# Scan a container image
grype python:3.11-slim

# Scan a directory (filesystem)
grype dir:.

# Scan an existing SBOM (generated by syft or cdxgen)
grype sbom:sbom.json

# Filter by severity
grype python:3.11-slim --only-fixed --fail-on high

# Output as JSON
grype python:3.11-slim -o json > results.json

# Output as SARIF
grype python:3.11-slim -o sarif > results.sarif
```

#### nuclei

```bash
# Install
brew install nuclei

# Update templates to latest
nuclei -update-templates

# Scan a target URL with all templates
nuclei -u https://example.com

# Scan with specific severity
nuclei -u https://example.com -severity critical,high

# Scan with specific tags (e.g., CVE only)
nuclei -u https://example.com -tags cve

# Scan multiple targets from file
nuclei -l targets.txt -severity critical -o results.txt

# Scan with custom template
nuclei -u https://example.com -t custom-templates/

# Network service scanning
nuclei -u 192.168.1.0/24 -tags network
```

#### renovate

```bash
# Option 1: GitHub App (recommended for most users)
# Install from https://github.com/apps/renovate — creates onboarding PR

# Option 2: Self-hosted CLI
npx renovate --token $GITHUB_TOKEN --repositories owner/repo

# Option 3: Docker
docker run --rm -e RENOVATE_TOKEN=$TOKEN renovate/renovate owner/repo

# Configuration (renovate.json in repo root)
# {
#   "extends": ["config:recommended"],
#   "vulnerabilityAlerts": { "enabled": true },
#   "automerge": true,
#   "automergeType": "pr",
#   "packageRules": [
#     { "matchUpdateTypes": ["patch"], "automerge": true }
#   ]
# }
```

#### cdxgen

```bash
# Install
npm i -g @cyclonedx/cdxgen

# Generate SBOM for current project
cdxgen -o sbom.json

# Generate SBOM for a container image
cdxgen -t docker -i myapp:latest -o sbom.json

# Generate SBOM with specific type
cdxgen -t python -o sbom.json .

# Generate Cryptography BOM (CBOM)
cdxgen --type java --spec-version 1.6 -o cbom.json .

# Submit to Dependency-Track
cdxgen -o sbom.json --server-url http://dtrack:8081 --api-key $KEY
```

## 3. Best Practices (from GitHub Repos)

### 1. aquasecurity/trivy-action (GitHub Actions integration)

**Repo**: https://github.com/aquasecurity/trivy-action
**Pattern**: Image scan on every PR, SARIF upload to GitHub Security tab.
**Key lesson**: After the March 2026 supply chain compromise of trivy-action (75 of 76 tags hijacked), the community best practice is to **pin actions to full commit SHA**, never mutable tags. Example:
```yaml
- uses: aquasecurity/trivy-action@<full-sha>
  with:
    image-ref: 'myapp:${{ github.sha }}'
    format: 'sarif'
    output: 'trivy-results.sarif'
    severity: 'CRITICAL,HIGH'
```

### 2. anchore/scan-action (Grype + SBOM workflow)

**Repo**: https://github.com/anchore/scan-action
**Pattern**: Syft generates SBOM once, Grype rescans as new CVEs appear — no need to re-pull images. The "generate once, scan many" pattern reduces CI time and network load.
```yaml
- uses: anchore/sbom-action@v0
  with:
    image: myapp:latest
    artifact-name: sbom.spdx.json
- uses: anchore/scan-action@v5
  with:
    sbom: sbom.spdx.json
    fail-build: true
    severity-cutoff: high
```

### 3. renovatebot/renovate (Automated dependency updates at scale)

**Repo**: https://github.com/renovatebot/renovate
**Pattern**: Merge confidence badges aggregate CI data from millions of updates across the ecosystem to show how likely a given update is to break builds. Combined with automerge rules for patch-level updates, this creates a low-friction update pipeline.
**Key config**:
- `vulnerabilityAlerts: { enabled: true }` — prioritize security updates
- `automerge: true` for patch updates that pass CI
- `packageRules` to group related deps (e.g., all `@types/*` in one PR)

## 4. Capability Design Recommendations

### Capability 1: `container_scan`
- **Type**: B (Code/Tool)
- **tool_ref**: trivy, grype
- **Steps**:
  1. `select_scanner` — Choose trivy (comprehensive) or grype (SBOM-focused) based on context
  2. `execute_scan` — Run `trivy image --severity HIGH,CRITICAL --exit-code 1 <image>` or `grype <image> --fail-on high`
  3. `parse_results` — Extract CVE IDs, severity, fixed versions, EPSS scores
  4. `prioritize_findings` — Rank by EPSS exploit probability + KEV catalog status, not just CVSS
  5. `generate_report` — Output SARIF for GitHub Security tab or JSON for Dependency-Track ingestion
- **quality_criteria**:
  - Must scan both OS packages AND language-specific dependencies
  - Must distinguish between fixable and unfixable vulnerabilities
  - Must include base image upgrade recommendation when applicable
  - Exit code must reflect severity threshold for CI gating

### Capability 2: `dependency_monitor`
- **Type**: B (Code/Tool)
- **tool_ref**: grype, trivy, renovate
- **Steps**:
  1. `inventory_deps` — Generate SBOM via `cdxgen -o sbom.json` or `trivy fs --format cyclonedx`
  2. `scan_vulnerabilities` — Run `grype sbom:sbom.json` against latest advisory databases
  3. `check_staleness` — Flag dependencies with no release in 12+ months (CWE-1104 signal)
  4. `generate_update_plan` — Map each finding to renovate packageRule or manual action
  5. `verify_remediation` — Re-scan after update to confirm vulnerability resolved
- **quality_criteria**:
  - Must cover all lock files in repository (package-lock.json, uv.lock, go.sum, etc.)
  - Must differentiate direct vs transitive dependency vulnerabilities
  - Must track advisory source (NVD, GitHub Advisory, OSV)

### Capability 3: `sbom_inventory`
- **Type**: A (Document/Research)
- **tool_ref**: cdxgen, syft, trivy
- **Steps**:
  1. `discover_projects` — Identify all build manifests (package.json, requirements.txt, go.mod, Dockerfile, etc.)
  2. `generate_sboms` — Run cdxgen per project, output CycloneDX JSON
  3. `merge_inventory` — Aggregate component list across all project SBOMs
  4. `analyze_composition` — Identify shared dependencies, license distribution, component age
  5. `publish_inventory` — Submit to Dependency-Track or export as compliance artifact
- **quality_criteria**:
  - Must support CycloneDX spec version >= 1.5
  - Must capture both direct and transitive dependencies
  - Must include license information per component
  - Must be reproducible (same input = same SBOM, excluding timestamps)

### Capability 4: `update_automation`
- **Type**: B (Code/Tool)
- **tool_ref**: renovate
- **Steps**:
  1. `configure_renovate` — Set up renovate.json with extends, packageRules, and automerge policies
  2. `define_security_policy` — Enable vulnerabilityAlerts, set priority for security patches
  3. `establish_merge_rules` — Automerge patches that pass CI; require review for major/minor
  4. `monitor_pr_health` — Track merge confidence badges, identify consistently failing updates
  5. `audit_update_history` — Review merged update PRs for regressions or security improvements
- **quality_criteria**:
  - Must cover all 90+ supported package managers present in the repo
  - Must not auto-merge major version bumps without human review
  - Must group related dependency updates to reduce PR noise
  - Must integrate with CI to verify builds pass before merge

### Capability 5: `network_vuln_scan`
- **Type**: B (Code/Tool)
- **tool_ref**: nuclei
- **Steps**:
  1. `define_scope` — Identify target URLs, IPs, or CIDR ranges; confirm authorization
  2. `select_templates` — Choose template categories (cve, misconfig, default-login, exposure)
  3. `execute_scan` — Run `nuclei -l targets.txt -severity critical,high -tags cve,misconfig`
  4. `deduplicate_findings` — Remove duplicate findings across targets
  5. `correlate_with_inventory` — Map findings to SBOM components where possible
  6. `generate_remediation_plan` — Prioritize by CVSS + exploitability + asset criticality
- **quality_criteria**:
  - Must update templates before each scan (`nuclei -update-templates`)
  - Must respect rate limits and scope boundaries (no scanning unauthorized targets)
  - Must output machine-readable format (JSON/SARIF) alongside human-readable
  - Must flag KEV (Known Exploited Vulnerabilities) separately for urgent triage

## 5. Anti-Patterns & Pitfalls

### Anti-Pattern 1: "Scan and Forget"
**Problem**: Running vulnerability scans in CI but never reviewing or acting on results. Findings accumulate, alerts become noise, and critical CVEs get buried.
**Mitigation**: Set severity-based exit codes (`--exit-code 1 --severity CRITICAL`) to block builds. Schedule regular triage meetings. Use Dependency-Track for continuous monitoring, not just point-in-time scans.

### Anti-Pattern 2: "CVSS-Only Prioritization"
**Problem**: Treating all CRITICAL CVSS scores equally. A CVSS 9.8 with no known exploit in a dev-only dependency is less urgent than a CVSS 7.5 with active exploitation (KEV listed) in a production-facing component.
**Mitigation**: Use EPSS (Exploit Prediction Scoring System) + KEV catalog status for prioritization. Grype supports EPSS and KEV natively. Combine with asset criticality for risk-based triage.

### Anti-Pattern 3: "Pin to Tag, Not SHA"
**Problem**: Using mutable version tags (e.g., `@v2`) for security scanning GitHub Actions. The March 2026 trivy-action compromise demonstrated that tag hijacking can turn your security scanner into a credential stealer.
**Mitigation**: Always pin GitHub Actions to full commit SHA. Verify SHA against the official repository's release notes. Use tools like StepSecurity Harden-Runner to detect unexpected network calls.

### Anti-Pattern 4: "SBOM as Checkbox"
**Problem**: Generating SBOMs for compliance without integrating them into vulnerability management. The SBOM sits in an artifact store, never re-scanned as new CVEs emerge.
**Mitigation**: Use the "generate once, scan continuously" pattern — cdxgen/syft produce the SBOM, Dependency-Track or grype re-scan it on schedule. SBOM is a living document, not a one-time deliverable.

### Anti-Pattern 5: "Auto-Merge Everything"
**Problem**: Configuring renovate to auto-merge all dependency updates including major versions. Breaking changes slip through when test coverage is insufficient.
**Mitigation**: Auto-merge only patch updates that pass full CI. Require human review for minor and major bumps. Use renovate's merge confidence badges to gauge ecosystem-wide breakage risk. Follow the principle: wait 2-3 days before updating to a new version (ref: litellm 1.82.7/1.82.8 PyPI poisoning, 2026-03-24).

### Anti-Pattern 6: "Single Scanner Dependency"
**Problem**: Relying on one vulnerability scanner (e.g., trivy alone). Each scanner has different advisory sources and detection heuristics — single-scanner blind spots are real.
**Mitigation**: Use complementary scanners. Trivy for breadth (container + IaC + secrets), grype for depth (EPSS + risk scoring), nuclei for network-facing services. Cross-reference findings to reduce both false positives and false negatives.
