1. 一句话理解 TAD

AI 做所有体力活（设计、写代码、测试、审查）。
人类只在 三个关键点 把关：需求对不对？优先级对不对？最终结果对不对？
这三个点是 80/20 法则的人类端——用 20% 的人工介入避免 80% 的返工。

为什么不让 AI 全自动？因为 AI 越强，跑偏时方向越错、速度越快。三个检查站就是刹车。

2. 三角模型：谁做什么

Human（你） — 价值守护者
不用写代码，只做三件事：确认需求 → 定优先级 → 验收结果。
你是 Alex 和 Blake 之间的唯一信息桥梁。
Alex — 方案负责人 $alex
接收你的需求 → 提问澄清 → 设计方案 → 写 Handoff 文档 → 最终验收。
Alex 只设计，不写代码。
Blake — 执行负责人 $blake
读取 Handoff → 写代码 → 跑测试 → 调 AI 专家审查 → 交完工报告。
Blake 只执行，不做设计决策。

4. 核心工作流

一个功能从想法到上线的完整路径：

$alex → *analyze
需求提问 3-5 轮 → *design
技术方案 → *handoff
交接文档 + 专家审查
$blake → *develop
Ralph Loop 实现 → *gate 3
技术质量门 → *complete
完工报告
$alex → *accept
业务验收 + 归档 → Done!
什么时候用 TAD？

新功能（3+ 文件）、架构变更、多模块重构 → 用 TAD

单文件 bug、改配置、文档更新、紧急修复 → 直接让 Codex 干

5. 四道质量门

每道门都是强制检查站。不通过就不能往下走。

G1 需求清晰 Alex 负责
G2 设计完整 Alex + 专家
G3 代码质量 Blake + 专家
G4 业务验收 Alex + 人类
Gate 3 详解（最复杂的一道）
Layer 1 Blake 自检：build → test → lint（最多重试 15 次）
Layer 2 AI 专家审查：spec-compliance → code-reviewer → security-auditor

Layer 1 全过了才进 Layer 2。Layer 2 发现 P0 问题必须修复再重跑。

6. Ralph Loop：Blake 的迭代引擎

Blake 不是写完代码就交差。*develop 启动一个自动迭代循环：

Ralph Loop:
  ┌─ 实现代码
  │
  ├─ Layer 1: build + test + lint
  │   └─ 失败？自动修 → 重跑（最多 15 次）
  │
  ├─ Layer 2: 调 AI 专家审查
  │   └─ 有 P0？修复 → 回到 Layer 1
  │
  └─ 全过？→ 生成完工报告 → Gate 3

Circuit Breaker：Layer 1 连续 3 次同一个错误 → 自动停止，上报 Alex，不做无效重试。

9. 项目知识系统

核心思路：每次 Gate 通过后，agent 自问"这次有什么值得记住的？"有就存下来，下次自动读取。

三层结构

层内容例子
L1 原则TAD 方法论规则（很少改）"deny-list 比 allow-list 安全"
L2 模式可复用经验（按主题分文件）"macOS pwd -P 不规范化大小写"
L3 事件一次性事件记录"2026-03-24 litellm 投毒事件"

自动行为

  • Gate 3 通过 → Blake 评估是否有新发现 → 写入 L2
  • Gate 4 通过 → Alex 评估业务层发现 → 写入 L2
  • *design 时 → Alex 自动读取相关历史经验
  • *dream → 去重 + 合并 + 清理过时条目

10. Epic：大型项目管理

改 3 个文件是一个 Handoff。改 30 个文件跨 5 个模块？拆成 Epic。

Epic = 多个 Phase，每个 Phase 一个 Handoff 
Epic: 用户认证系统
├── Phase 1: 数据库 schema 设计        ← Handoff → Blake → Gate 3 → Gate 4 ✅
├── Phase 2: API 接口实现              ← Handoff → Blake → Gate 3 → Gate 4 ✅
├── Phase 3: 前端登录页面              ← 🔄 进行中
├── Phase 4: OAuth 集成                ← ⬚ 计划中
└── Phase 5: 安全审计 + 验收           ← ⬚ 计划中

关键规则

  • 同时只能有 1 个 Active Phase（避免混乱）
  • 每个 Phase 完成后更新 Epic 的 Phase Map
  • 所有 Phase 完成 → Epic 归档到 .tad/archive/epics/

YOLO 模式

明确知道要做什么？YOLO 模式让 Alex 自动驱动所有 Phase：设计 → 审查 → 调 Blake 实现 → 审查 → Gate → 下一个 Phase。人类只在开头和结尾介入。

11. Migration 引擎

v2.28.0 新增。解决一个老问题：升级 TAD 后旧文件留垃圾。

原理

每个版本附带一个 manifest（.tad/migrations/2.26.0-to-2.27.0.yaml），声明该版本"删了什么、改名了什么、合并了什么"。升级时引擎读取 manifest 链并执行。

五步安全流水线

每个文件操作（删除/重命名/合并）都过这五步检查：

步骤检查什么不过怎么办
1. 路径规范化去掉 ..、多余斜杠拒绝
2. 禁止字符反斜杠、控制字符、冒号拒绝
3. 前缀白名单只允许 .tad/ .claude/ 等拒绝
4. 物理路径验证cd + pwd -P，防 symlink 逃逸拒绝
5. ZERO_TOUCH 检查不碰用户数据目录拒绝

其他安全机制

  • 单一删除咽喉点：整个引擎只有一处 rm（guarded_remove），删前现场重验
  • 用户修改保护：你改过的框架文件不会被删，跳过 + 报告
  • 备份：删除前先备份到 .tad-backup/{from}-to-{to}/
  • 幂等：跑两次效果和一次一样
  • 发版门禁：删了文件但忘写 manifest → 发版自动阻断

12. Handoff：信息传递的唯一通道

Handoff = Alex 写给 Blake 的交接文档。这是 Blake 获取任务信息的唯一来源。

Handoff 包含什么
§1 任务概述做什么、为什么做、意图声明
§2 背景已有代码、依赖、之前的工作
§3 需求功能需求 + 非功能需求
§4 技术设计架构图、组件规格、数据模型
§5 强制问题历史搜索、函数验证、数据流
§6 实现步骤Blake 的具体任务清单
§7 文件结构要创建/修改的文件列表
§8 测试要求需要什么测试、怎么验证
§9 验收标准逐条可验证的 AC + bash 命令

关键规则：Handoff 创建前必须经过 2+ 位 AI 专家审查。P0 问题必须修复才能发给 Blake。

14. 十个实用技巧

1. *discuss 当产品顾问
不写代码也能用 TAD。*discuss 可以做产品方向讨论、技术选型、竞品分析。结束时不产出 handoff。
2. *idea 攒灵感
想到好点子不想现在做？*idea 存档。以后 *idea-list 回顾，*idea-promote 升级为正式任务。
3. *learn 学新技术
不理解某个概念？*learn 走苏格拉底式教学——通过提问引导你理解，比直接告诉答案学得深。
4. *bug 快速修 bug
不用走完整流程。*bug 快速诊断 → express handoff → Blake 修。适合范围明确的 bug。
5. *status 看全景
一个命令看到所有进行中的 Epic、待处理的 Handoff、存档的 Idea、Roadmap 主题。
6. 能力包自动加载
讨论 API 设计 → 自动加载 web-backend（43 条规则）。做 React → 自动加载 web-frontend。也可以手动指定。
7. YOLO 模式跑大任务
大型 Epic 可以用 YOLO 模式：Alex 自动设计 → 审查 → 调 Blake → 审查 → Gate → 下一 Phase。人类只在开头和结尾介入。
8. 知识自动积累
Gate 通过后 agent 自动评估学到了什么，存入项目知识库。下次做类似任务自动读取，不用手动管理。
9. CLAUDE.md marker 保护内容
在 CLAUDE.md 里加 <!-- TAD:PROJECT-CONTENT-BELOW -->，升级时你写在下面的内容不会被覆盖。
10. *dream 清理知识库
知识积累到 50+ 条后，*dream 自动去重、合并、清理过时条目。只生成候选等你确认，不直接改原文件。

16. 常见问题

Q: $alex 没反应？
检查 .agents/skills/alex/SKILL.md 是否存在。不存在就重装：
curl -sSL https://raw.githubusercontent.com/Sheldon-92/TAD/main/tad.sh | bash -s -- --platform codex --yes
Q: Gate 3 失败说没有 evidence？
Blake 必须调 AI 子 agent 做审查。检查 .tad/evidence/reviews/ 目录是否有审查文件。
Q: Blake 看不到 Alex 写的 Handoff？
TAD 设计上 Alex 和 Blake 在不同终端。你是两者之间的桥梁——需要手动把 Handoff 路径告诉 Blake。
Q: 升级后旧文件还在？
v2.28.0 之前没有 migration 引擎。首次升级到 2.28.0+ 后，后续升级自动处理。手动清理：
bash .tad/hooks/lib/migration-engine.sh --from <旧版本> --to <新版本> --target . --source <tad源目录>
Q: 能力包没加载？
Packs 按需加载。在 *design 时说"请加载 ai-guardrails pack"即可手动指定。
Q: TAD 占多少 context？
CLAUDE.md 路由层 ~2K tokens。SKILL 文件按需加载（progressive loading），只在激活角色时加载。能力包按关键词匹配动态加载。
