# Capability Pack Template

Use this template to build a new capability pack. Based on patterns from 13 production packs.

## Architecture Types

| Type | When to Use | Example |
|------|-------------|---------|
| **reference-based** | Domain has parallel judgment rules (facets of one framework) | web-backend (43 rules), code-security (36 rules) |
| **deep-skill** | Domain needs structured multi-step interaction with state | product-thinking (3 interconnected skills) |
| **orchestration-router** | Domain is a workflow with phase transitions and gates | research-methodology (5-phase pipeline) |

**Default to reference-based** — 11 of 13 packs use this pattern.

## File Structure

```
.tad/capability-packs/{{pack-name}}/
├── CAPABILITY.md          # Main router (installed as SKILL.md)
├── install.sh             # Cross-agent installer
├── LICENSE                # Apache 2.0
└── references/
    ├── {{capability-1}}-rules.md    # Judgment rules
    ├── {{capability-2}}-rules.md
    └── {{capability-N}}-workflow.md # Structured workflow (if Type A/Mixed)
```

## CONSUMES/PRODUCES Interface

Declare what your pack needs and produces for multi-pack chaining:

```yaml
# In pack-registry.yaml (auto-generated by scan-packs.sh)
consumes: "User task description + optional existing codebase"
produces: "Applied judgment rules + quality checklist results"
```

**Chaining example:** web-ui-design (PRODUCES DESIGN.md) → web-frontend (CONSUMES DESIGN.md)

## Quality Bar

| Metric | Target | Source |
|--------|--------|--------|
| CAPABILITY.md words | < 3,500 (< 5K tokens) | AKU research |
| Rules per reference file | ≥ 3 | Minimum actionable density |
| Rule format | "When X, do Y" (prescriptive) | AKU format |
| Tool bindings | CLI commands in every reference | Not just tool names |
| Research sources | ≥ 20 per pack (GitHub-First) | /capability-upgrade methodology |
| Frontmatter | `name:` + `description:` MANDATORY | Claude Code registration requirement |

## Build Process

Follow `/capability-upgrade` skill (5 stages):
1. **Assess** — read existing YAML, determine AI agent boundaries
2. **Research** — GitHub-First deep research (NotebookLM)
3. **Design** — structure + expert review
4. **Build** — write files + install + verify
5. **Validate** — test in real project

## Common Patterns

- **Cross-cutting rule** in CAPABILITY.md router (not buried in one reference)
- **Anti-Skip Table** counters common rationalizations for skipping rules
- **`<!-- capability: X -->` HTML comment** on line 2 of every reference file (grep anchor)
- **Quick Rule Index table** at top of every reference file
- **determinismLevel annotation** on rubric-type rules (deterministic / semi-deterministic / non-deterministic)
