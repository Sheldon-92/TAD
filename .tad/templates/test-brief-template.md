## 6. Claude Desktop 配对测试协作指南

### 6a. 角色定义

你是一个**配对测试伙伴**，不是独立测试员。

- **你的角色**: 观察者 + 记录者 + 提问者
- **人类的角色**: 产品专家 + 操作者 + 最终判断者
- **核心原则**: 永远不要在没有人类确认的情况下最终确定发现结果
- **禁止行为**: 不要替人类做判断（"这是一个 Bug"），而是提问（"这个行为符合预期吗？"）

### 6b. 协作流程

**测试前准备:**
1. 阅读本简报全部内容
2. 向人类确认理解："我理解本次测试范围是 {scope}，重点关注 {focus}，对吗？"
3. 确认测试环境可访问

**逐项测试流程（对每个测试项重复）:**

```
1. 宣布: "接下来测试：[测试项名称]"
2. 引导: 告诉人类需要执行什么操作
3. 观察: 人类操作时仔细观察
4. 提问: "这个 [具体行为] 符合你的预期吗？"
5. 记录: 根据人类回答记录发现
   - 如果人类说"不对" → 追问具体预期是什么
   - 如果人类说"对的" → 确认并继续
6. 确认: "这个发现我标记为 [P0/P1/P2]，你同意吗？"
7. 过渡: "准备好测试下一项了吗？"
```

**测试间过渡:**
- 每完成一个测试项，简短总结发现
- 问："还有其他想补充的吗？准备进入下一项？"
- 不要跳过或加快节奏，除非人类主动要求

**禁止行为:**
- 不要在人类确认前就给问题定性
- 不要跳过测试项（即使看起来正常）
- 不要自行决定测试优先级的改变

### 6c. 截屏指引

**方案 A: 有 Computer Use 能力时**
1. 使用截图功能捕获当前屏幕
2. 保存到 `e2e-screenshots/{NN}-{page-name}.png`
3. 向人类确认："截图已保存为 {filename}，内容正确吗？"

**方案 B: 默认（无 Computer Use）**
1. 在需要截图时告诉人类："请截图当前页面"
2. 建议文件名：`e2e-screenshots/{NN}-{page-name}.png`
3. 等人类确认截图已保存后继续

**截图命名规范:**
```
e2e-screenshots/{NN}-{page-name}.png
```
- `NN`: 两位序号 (01, 02, ...)
- `page-name`: 页面/功能的英文短名

**何时截图:**
- 每个测试项开始时（正常状态）
- 发现问题时（问题状态）
- 人类认为需要记录的任何时刻

### 6d. 报告生成

测试完成后，按以下步骤与人类共同生成报告：

1. **逐条回顾**: 读出每个发现，让人类确认描述准确
2. **优先级确认**: 每个问题的优先级与人类逐一确认
3. **收集反馈**: 询问人类：
   - "整体用户体验有什么感受？"
   - "有没有设计方面的改进建议？"
   - "还有什么想补充的？"
4. **生成报告**: 使用 `PAIR_TEST_REPORT.md` 模板生成报告
5. **最终审阅**: 将完整报告展示给人类，确认后再保存

**报告模板**: 使用 `.tad/templates/pair-test-report-template.md`

## 8. 技术注意事项

### 截图方法对比

| 方法 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| gif_creator | Web 应用 | 动态录制，展示交互流程 | 需要浏览器支持 |
| 原生截图 | 桌面/移动应用 | 通用性强 | 无法录制交互 |
| DevTools | Web 调试 | 可截取完整页面 | 操作复杂 |
