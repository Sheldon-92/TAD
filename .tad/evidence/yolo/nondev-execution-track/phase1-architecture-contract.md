## A. `task_type: deliverable` — Definition + Routing

### A.1 Definition

`deliverable` is a new value for the `task_type` frontmatter key (joining `code | yaml | research | e2e | mixed`).

A handoff is `task_type: deliverable` when the unit of work is a **non-code content artifact whose
quality is judged by a pack-specific rubric**, not by build/test/lint. Examples: a research report
(academic-research), an audiobook chapter (ai-voice-production), a short film cut (video-creation),
a product spec / PRD (product-thinking). The "Files to Modify" section is reframed as "Deliverables
to Produce" (artifact paths), and Gate 3 acceptance is "rubric weighted score ≥ pass_threshold,
scored by an independent judge."

### A.2 Routing touchpoints — four touchpoints (1 producer + 3 consumers) (AC1)

The value must be detected and routed at **four touchpoints: ONE producer (Touchpoint 0, which
CLASSIFIES the work and SELECTS the template) + THREE consumers (Touchpoints 1-3, which act on the
classification)**. Each consumer touchpoint is additive — a new deliverable path beside the existing
code path. (v1 incorrectly claimed "exactly three" and omitted the producer — P0-3 fix.)

**Touchpoint 0 — Alex handoff classification** (`.claude/skills/alex/SKILL.md`, modified Phase 2 — additive)
- This is the PRODUCER touchpoint: nothing in v1 actually set `task_type: deliverable` or selected the
  deliverable template. Without it the other three touchpoints never fire.
- Additive rule in `alex/SKILL.md` (in the handoff-drafting / template-selection step that follows
  `intent_router_protocol` + `adaptive_complexity_protocol`, around alex/SKILL.md:496-545): when the
  unit of work is a **pack-produced content artifact** (per the §A.4 rule-of-thumb — the artifact IS
  the product, not a feeder for a downstream build), Alex sets `task_type: deliverable` and selects
  `deliverable-handoff.md` instead of `handoff-a-to-b.md`.
- This is additive: it adds a classification branch; it does NOT change the existing intent_router
  signal-detection or the default `analyze`→code template selection for non-deliverable work.
- The corresponding `tad-handoff` template-selection step is updated to honor `task_type: deliverable`
  → `deliverable-handoff.md`.

**Touchpoint 1 — Handoff frontmatter** (`.tad/templates/deliverable-handoff.md`, created Phase 2)
- New frontmatter keys (in addition to the existing `task_type`):
  - `task_type: deliverable`
  - `rubric_ref:` — path to the rubric file (e.g. `.claude/skills/academic-research/references/scholar-eval.md`)
  - `pass_threshold:` — numeric 0-1 (e.g. `0.75`)
  - `deliverable_paths: []` — list of artifact paths the producer must create
  - `pack:` — the capability pack name (key into `deliverable-rubrics.yaml`)
- Detection: gate reads `task_type` from frontmatter (same mechanism the Phase-4 hook already uses to read `task_type`).

**Source-of-truth precedence (rubric_ref / pass_threshold) — resolves dual-source-of-truth (CR P1-3):**
1. If the handoff frontmatter sets `rubric_ref` and/or `pass_threshold` → **the frontmatter values WIN**
   (explicit per-handoff override; allows a one-off rubric/threshold without editing the registry).
2. Else, fall back to the `deliverable-rubrics.yaml` row keyed by `pack`.
3. If **BOTH** are absent (no frontmatter value AND no registry row / null) → **Gate 3 BLOCKS** with
   "deliverable handoff has no resolvable rubric_ref + pass_threshold (set in frontmatter or register
   the pack in deliverable-rubrics.yaml)". No silent default.

**Touchpoint 2 — Gate 3 branch** (`.claude/skills/gate/SKILL.md`, modified Phase 2 — see §B)
- The gate reads `task_type` from the active handoff frontmatter. `IF task_type == deliverable` →
  take the deliverable branch (§B). `ELSE` → existing code path, **byte-unchanged** (§E).

**Touchpoint 3 — Completion report** (`.tad/templates/deliverable-completion.md`, created Phase 2)
- Replaces the code-shaped "Layer 1 (Build/Tests/Lint/tsc)" table with a **Rubric Scores** table
  (8-dim weighted scores + weighted average + verdict) and an **Artifacts Produced** table (paths).
- Keeps `gate3_verdict:` frontmatter marker (value ∈ `pass|fail|partial`) so `post-write-sync.sh`
  trace emission continues to work unchanged.

### A.3 Routing summary (divergence = sibling gate sections + template shape; both gates branch)

```
                                Gate 3                                  Gate 4
task_type ∈ {code,yaml,e2e,mixed}  existing block (test-runner)  [UNCHANGED]   existing block (3 code subagents) [UNCHANGED]
task_type == research              existing path (no branch)     [UNCHANGED]   existing path                     [UNCHANGED]

                                                                                + human approval, §B.7)          [ADDED]
```

Both gate sections are **additive sibling sections** (own ```yaml fence) placed after the existing
blocks; the existing fenced blocks (Gate 3 = gate/SKILL.md:95-338; Gate 4 = the block starting line
341) stay byte-identical. The deliverable path is selected by a single guard line in each gate's prose
header (§B.1, §B.7).

### A.4 `deliverable` vs `research` (resolves D-open-3 — explicit)

- **`research`** = a research *spike* whose output is an input to a later code/yaml handoff. It never
  had a Gate-3 branch and does not get one. It remains a documentation/feeder type. **KEPT, untouched.**
- **`deliverable`** = the research/content artifact IS the delivery; acceptance = rubric score. This is
  the new lane. An academic-research report shipped to a user is `deliverable` (rubric = scholar-eval,
  ≥0.75). A "go survey the literature so Alex can design the API" spike stays `research`.
- Rule of thumb baked into the deliverable-handoff template: *if the artifact is the product → `deliverable`;
  if the artifact informs a downstream build → `research`.*

## B. Gate 3 Non-Dev Branch Contract

### B.1 The branch mechanism — additive sibling section (P0-1 fix)

v1 proposed wrapping the existing Gate 3 keys in an `ELSE:`. That is **structurally impossible without
a byte change**: Gate 3 is ONE fenced ```yaml block (gate/SKILL.md:95-338); indenting every top-level
key under `ELSE:` re-indents every line. So the mechanism is an **additive sibling section**, not a wrap:

```
Existing prose header (gate/SKILL.md ~line 94, OUTSIDE the fence) — Phase 2 adds ONE guard line:
  "IF the active handoff is task_type: deliverable → SKIP this block; use
   '## Gate 3 — Deliverable Branch' below."

Existing fenced block (lines 95-338) — BYTE-IDENTICAL (no edit inside the fence).

NEW sibling section, inserted immediately AFTER line 338, its own ```yaml fence:
  ## Gate 3 — Deliverable Branch
  ```yaml
  When: After deliverable production (BLOCKING), only if task_type == deliverable
  ...  (B.2–B.6 spec)
  ```
```

Routing at gate-run time:
```
read task_type from active handoff frontmatter
IF task_type == deliverable → execute '## Gate 3 — Deliverable Branch'  (skip the original block)
ELSE                        → execute the original Gate 3 block          (byte-unchanged)
```

**Phase-2 byte-check AC (v2.1 corrected — offset-aware; the re-review caught that a fixed `95,338p` on both sides is WRONG because the guard line added above line 95 shifts the block down):** the guard line is inserted in the prose header ABOVE the fence, so the original block moves down by exactly the number of guard lines added (1). The correct byte-identity check is **offset-aware**:
`diff <(git show HEAD:.claude/skills/gate/SKILL.md | sed -n '95,338p') <(sed -n '96,339p' .claude/skills/gate/SKILL.md)` MUST be empty after the edit (HEAD 95-338 vs current 96-339 — the +1 offset = the single guard line). If Phase 2 adds N guard lines, offset the current-side range by N. **Robust alternative (shift-independent)** — extract the original Gate 3 fenced YAML body by content anchor and diff: `diff <(git show HEAD:.claude/skills/gate/SKILL.md | awk '/^## Gate 3:/{g=1} g&&/^[\`]{3}yaml$/&&!s{s=1;next} s&&/^[\`]{3}$/{exit} s') <(awk '/^## Gate 3:/{g=1} g&&/^[\`]{3}yaml$/&&!s{s=1;next} s&&/^[\`]{3}$/{exit} s' .claude/skills/gate/SKILL.md)` MUST be empty (extracts only the fence body, immune to any line-number shift from the guard line or the sibling section).

### B.3 Judge inputs (the contract for the spawned judge)

The gate/Conductor spawns the judge sub-agent with a prompt containing **only**:
1. `deliverable_paths` — the artifact file path(s) to evaluate (read by the judge).
2. `rubric_ref` — the rubric file path (e.g. `scholar-eval.md`). The judge reads and applies it.
3. `pass_threshold` — the numeric threshold (e.g. `0.75`).

The judge prompt MUST NOT include the producer's reasoning, chat transcript, persona, or any "this is
good because…" framing (see §C). Blue-team framing for the judge: "You are an independent reviewer.
Score the artifact at {path} against the rubric at {rubric_ref}. Report dimension scores + weighted
average + verdict."

### B.4 Judge output file + format

- **Path**: `.tad/evidence/reviews/{date}-rubric-eval-{task}.md` (mirrors the existing
  `{date}-testing-review-{task}.md` naming so Gate 4's prerequisite glob can be generalized later).
- **Format** (per dimension + roll-up):
  - A scores table: `| # | Dimension | Weight | Score (0-1) | Notes |` (one row per rubric dimension).
  - `weighted_score = Σ(score_i × weight_i)` shown explicitly with the arithmetic.
  - `verdict` line (see B.5).
  - Top-3 strengths / top-3 weaknesses (actionable, per scholar-eval Step 4-5).
- **Parser-safety constraint (BA P1-1)** — the rubric-eval file's weaknesses MUST NOT use the
  `^#+ *P[0-9]-` heading form (e.g. NOT `### P0-1`, `## P1-2`). The `post-write-sync.sh`
  `expert_review_finding` parser counts heading-form `P<n>-` labels; using them in a rubric-eval file
  would self-trigger false P0/P1 telemetry (the documented "Parser Self-Trigger" failure). Use plain
  prose weaknesses ("Weakness 1: …") or a `| # | Weakness | Severity |` table cell — never a P-label
  heading. This contract forbids the heading form in the rubric-eval evidence format.
- **Distinct evidence type (BA P1-1)** — `rubric-eval` is a NEW, DISTINCT evidence type. It MUST NOT be
  folded into the existing `*-review` glob or the `types:` enum at gate/SKILL.md:381. The Phase-4 Gate-4
  generalization adds `rubric-eval` to that enum as its own entry (see §F + §B.7), never aliasing it to
  `testing-review`/`code-review`. (This also keeps Gate 4's prerequisite glob unambiguous — §B.7.)

### B.5 Verdict mapping (generalized per pack)

The general rule, parameterized by the pack's `pass_threshold` (default ladder taken from scholar-eval):

```
IF weighted_score ≥ pass_threshold           → PASS
ELSE IF weighted_score ≥ partial_threshold    → PARTIAL   (default partial_threshold = 0.60)
ELSE                                          → FAIL
```

- For academic-research: `pass_threshold = 0.75`, `partial_threshold = 0.60` → exactly the
  scholar-eval ladder (≥0.75 Accept = PASS, 0.60-0.75 Minor Revision = PARTIAL, <0.60 = FAIL).
- `partial_threshold` is a per-pack field in `deliverable-rubrics.yaml` (defaults to 0.60 if omitted),
  so packs with a different rubric scale can override it.
- **PASS** → Gate 3 proceeds (KA + git checks). **PARTIAL/FAIL** → Gate 3 BLOCKS; the producer (§B.6)
  revises and re-runs (the judge re-scores the revised artifact — a fresh judge spawn each time).

### B.6 Deliverable production pipeline (P0-4 fix — who produces, and why not Blake)

v1 said "Blake produces the artifacts." That is **wrong for research deliverables**: the research tools
(NotebookLM is stateful/sequential and Conductor-side; WebSearch) cannot run inside a Blake sub-agent
(architecture.md: "Research must be Conductor-side — cannot be delegated to Blake sub-agents"). Corrected
pipeline:

1. **Producer** = a **Conductor-spawned PRODUCER sub-agent (or the Conductor itself)** that has the
   research tools. For Phase 3 academic-research: **WebSearch is the reliable path; NotebookLM is
   optional** (and Conductor-owned if used). The producer writes the artifact(s) to `deliverable_paths`.
2. **Blake's generic "implement" lane does NOT apply** to research/content deliverables — Blake builds
   code; the producer agent builds the deliverable. (For deliverables that happen to be code-shaped this
   could change, but no such pack is in scope; this contract scopes producer = Conductor-side agent.)
3. **Judge** (§C) is a SEPARATE fresh sub-agent spawned by the gate/Conductor. **judge ≠ producer is
   defined relative to THIS producer** (the Conductor-spawned producer sub-agent / Conductor), not
   relative to Blake. If the Conductor itself produced the artifact, the judge MUST be a distinct
   sub-agent (Conductor cannot judge its own output).
4. This is honest for Phase 3: producer sub-agent → artifact at `deliverable_paths` → completion report
   → Gate 3 spawns judge → rubric-eval → verdict → Gate 4 (§B.7).

## C. judge ≠ producer Mechanism (load-bearing — AC2)

### C.1 The hard rule

> The deliverable is produced by ONE agent (Blake / a producer sub-agent). The rubric score is
> computed by a **SEPARATE judge sub-agent**, spawned fresh by the gate/Conductor, whose prompt
> references **ONLY** `{deliverable_paths} + {rubric_ref} + {pass_threshold}` — never the producer's
> reasoning, persona, identity, or self-assessment.

### C.2 Why (self-enhancement bias)

The ai-evaluation pack's foundational rule is **Judge ≠ Optimizer/Producer**: an agent scoring its own
output exhibits measurable **self-enhancement bias (~10-15% inflation)**. A producer that knows it
authored the artifact will rationalize its choices and score generously. Independent judging is the
only defense — and is the exact "validation-theater-killer" the Epic Notes (Phase 1, line 78) flag as
the load-bearing decision. This mirrors the project's "Mechanical Enforcement / count ≠ signal" lessons:
a self-scored rubric is validation theater.

### C.3 Forbidden pattern (VIOLATION)

```
VIOLATION: the producing agent (or the same session/persona that produced the artifact)
           computes the rubric score for its own deliverable.
VIOLATION: passing the producer's reasoning / "why this is good" notes into the judge prompt.
VIOLATION: reusing the producer sub-agent (or its conversation) as the judge.
VIOLATION (artifact-channel, CR P1-2): the judge crediting self-assessment / quality-claim prose
           embedded INSIDE the artifact ("this report is rigorous and comprehensive", a self-scored
           rubric table the producer wrote into the deliverable, etc.). The judge scores on RUBRIC
           EVIDENCE it independently derives from the artifact's substance — NOT on the artifact's own
           claims about its quality. Self-praise in the artifact is ignored, not credited.
```

### C.4 Enforcement spec (for Phase 2 gate text)

1. The judge is spawned by the **gate/Conductor**, not by the producer (producer cannot self-elect a judge).
2. The judge prompt is constructed from **file paths only** (artifact + rubric). It is auditable: the
   `{date}-rubric-eval-{task}.md` evidence file must state "Judge: independent sub-agent; producer
   identity not provided."
3. Phase 3 dogfood AC4 proves this in practice (the report shows producer ≠ judge were different agents).
4. Treat any "the producer already self-scored, so skip the judge" rationalization as a VIOLATION
   (the "Express → exempt" anti-pattern applied to scoring).

- `pack-registry.yaml` is **auto-regenerated** by `scan-packs.sh` (`cat > $OUTPUT`, full overwrite
  from frontmatter). Any `rubric_ref`/`pass_threshold` field added there would be **clobbered** on the
  next scan — the exact registry-desync failure documented in code-quality.md ("Auto-Generated Registry
  → Persisted Decision State Belongs in a Side-File") and proven by `behavioral-eval-status.yaml`.
- A name-keyed side-file is **conductor/user-owned decision state** that survives registry regeneration,
  and keeps the rubric↔pack binding in ONE auditable place rather than scattered across 5 pack
  frontmatters (single source of truth; easier Phase-4 generalization).
