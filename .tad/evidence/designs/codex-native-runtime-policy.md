## Sandbox / Approval / Profile Policy

### Recommended Defaults

| Setting | Value | Rationale |
|---------|-------|-----------|
| `sandbox_mode` | `workspace-write` | TAD writes to `.tad/evidence/`, `.tad/active/`, and project source. Read-only would block implementation. Full-access is unnecessarily risky. |
| `approval_policy` | `on-request` | Model decides when to ask. `never` is too risky for TAD quality gates. `untrusted` is too restrictive for TAD workflows. |
| Protected paths | `.git`, `.codex` | Codex auto-protects these in workspace-write mode (Codex manual L2111). TAD does not need to override. |

### Permission Profile (Optional, for Stricter Control)

If the user wants stricter control, a custom permission profile can be defined:

```toml
[permissions.tad-workspace]
[permissions.tad-workspace.filesystem]
":minimal" = "read"
[permissions.tad-workspace.filesystem.":workspace_roots"]
"." = "write"
"**/*.env" = "deny"
```

This is optional — `sandbox_mode = "workspace-write"` is sufficient for most TAD use cases.

### Gate Pre-Check Compatibility

TAD gate pre-checks (`pre-accept-check.sh`, `pre-gate-check.sh`) run bash scripts via Codex's shell tool. `workspace-write` + `on-request` allows these to execute. If a stricter sandbox is used, these scripts must be allowlisted via rules.

### Current State Assessment

| PostToolUse (ask_user_question) | `askuser-capture.sh` | Capture structured decisions | Working (unknown_current_behavior on Codex — Codex may not have `ask_user_question` as a tool name) |

### Policy Decisions

1. **Keep `.codex/hooks.json` as the hook source of truth** — it is auto-generated by `tad.sh` and covers the TAD-required events. No need for `config.toml` inline hooks.
2. **Trust review**: First-time Codex users will be prompted to trust project hooks. Document this in onboarding.
3. **Timeout**: All hooks have explicit timeouts (10-30s). No hook blocks Codex indefinitely.
4. **Failure mode**: If a hook fails, Codex skips it (not fail-closed by default). TAD accepts this — hooks are smoke alarms per principles.md "Mechanical Enforcement Rejected on Single-User CLI."
5. **`ask_user_question` matcher**: Codex may not recognize `ask_user_question` as a tool name (Phase 1 V20: `unknown_current_behavior`). If this hook silently never fires, TAD loses decision provenance for the entire Codex platform — this IS an evidence-completeness gap in the quality chain, not merely a convenience loss. **Medium-priority**: Phase 5 regression must verify whether the hook fires; if not, either adapt the matcher to Codex's equivalent tool name or implement an alternative evidence-capture path.

### Future Consideration

If `.codex/config.toml` is activated, hooks could also be defined inline in TOML. The policy recommends against this — keep one hook source (`hooks.json`) to avoid merge ambiguity (Codex warns when both exist in the same layer).

## MCP Policy

### Project-Scoped MCP (`.codex/config.toml`)

| Category | Project-Scoped? | Example | Rationale |
|----------|-----------------|---------|-----------|
| No-auth MCP servers | ✅ Yes | Context7 (`npx -y @upstash/context7-mcp`) | No credentials needed; useful for documentation lookup |
| Auth-required MCP servers | ❌ No | Figma, Sentry, GitHub MCP | Token/key belongs in user config |
| Plugin-bundled MCP | ❌ No | Installed per-user via `codex plugin install` | User-scoped lifecycle |

### Credential Separation

- Server **definitions** (command, args, URL) may be project-scoped if no credentials are embedded.
- **`bearer_token_env_var`**, **`env`** values containing secrets, and **OAuth tokens** must be in `~/.codex/config.toml` or environment variables.
- `enabled_tools` / `disabled_tools` / `approval_mode` are project-safe — they control behavior, not credentials.

### Recommendation

Phase 2 draft config includes no MCP servers (commented-out examples only). TAD does not currently depend on any MCP server on Codex. If a future TAD workflow needs project-scoped MCP, add it case-by-case with the no-auth rule.

## Custom Agents Evaluation

### Role Decisions

| Role | Decision | Rationale | Draft File |
|------|----------|-----------|------------|
| spec-compliance-reviewer | `migrate_draft` | Narrow scope; well-defined AC-checking task; benefits from dedicated reasoning settings | `agents/spec-compliance-reviewer.toml.draft` |
| code-reviewer | `migrate_draft` | Narrow scope; P0/P1 classification is well-defined; most frequent Layer 2 reviewer | `agents/code-reviewer.toml.draft` |
| test-runner | `migrate_draft` | Narrow scope; can use cheaper model (gpt-5.4-mini) for test execution + coverage | `agents/test-runner.toml.draft` |
| security-auditor | `defer` | Conditional trigger only; not frequent enough to justify a dedicated agent file. Can be subagent-prompted when needed. | — |
| performance-optimizer | `defer` | Conditional trigger only; same reasoning as security-auditor. | — |
| backend-architect | `defer` | Used by Alex (Terminal 1), not Blake. Blake doesn't spawn architecture agents. If Alex needs Codex review, it's via explicit prompt, not a custom agent. | — |
| Blake (execution agent) | `keep_skill_only` | Blake's persona, Ralph Loop, execution checklist, completion protocol are all SKILL-body content. Duplicating into TOML `developer_instructions` would fork the protocol. SKILL.md is authoritative. | — |
| Alex (solution lead) | `keep_skill_only` | Alex's Socratic inquiry, expert review, handoff creation, gate protocols are all SKILL-body content. Same fork risk. SKILL.md is authoritative. | — |

### Design Principles for Custom Agent Drafts

1. **Minimal instructions**: Each agent's `developer_instructions` references TAD criteria conceptually but does NOT duplicate SKILL.md content. Instructions are ≤10 lines.
2. **Model selection**: Reviewer agents use `gpt-5.5` + `high` reasoning (quality-critical). Test-runner uses `gpt-5.4-mini` + `medium` (speed-critical).
3. **No sandbox override**: Agents inherit parent session sandbox. No per-agent sandbox escalation.
4. **No skill config**: Agents don't disable/enable skills — they inherit the session's skill set.
5. **No MCP config**: Agents don't configure their own MCP servers.

### Quality Parity Risk

Custom agents with minimal `developer_instructions` may not produce Layer 2 reviews of equivalent quality to Claude Code's rich Agent tool with `subagent_type`. This is the key Phase 5 regression question. The draft agents are intentionally conservative (instructions reference TAD criteria without explaining them) — if quality is insufficient, instructions can be expanded while staying within the "narrow-scope, no-protocol-fork" boundary.

### What Happens If Activation Criteria Fail

- **Phase 5 regression fails**: Investigate root cause. If custom agents produce insufficient review quality → expand `developer_instructions` or revert to `keep_skill_only` for that role.
- **Secrets found**: Remove immediately. Git history cleanup if already pushed.
- **Human declines**: Drafts remain in evidence. No active config created. TAD continues SKILL-only on Codex.

## Risks and Unknowns

### Risks

1. **Custom agent quality parity**: Unknown whether minimal `developer_instructions` in TOML produce reviews comparable to Claude Code Agent tool with rich subagent_type. Mitigated by Phase 5 regression requirement.
2. **Hook trust friction**: First Codex activation in a project prompts trust review for `.codex/hooks.json`. No way to pre-approve programmatically. Documented in onboarding.
3. **Config schema volatility**: Codex config.toml schema may change across versions. `model`, `sandbox_mode`, `approval_policy` are stable (documented). Less-documented settings may break. Mitigated by sticking to well-documented settings.
4. **Sandbox strictness mismatch**: If user has stricter global sandbox, project config's `workspace-write` may be overridden. Codex uses highest-precedence config. Need to document that user config wins.

### Unknowns (`unknown_current_behavior`)

1. **`ask_user_question` hook matcher**: Codex may not have this exact tool name. The `askuser-capture.sh` hook may silently never fire on Codex. Impact: loss of decision provenance — an evidence-completeness gap in the quality chain (consistent with Hooks Policy §5). Phase 5 must verify and resolve.
2. **Custom agent skill inheritance**: Whether custom agents inherit the session's skill set is documented ("Optional fields... inherit from the parent session") but the exact skill visibility in a custom agent context is not empirically tested. Phase 5 regression should include a skill-aware review task.
3. **Project trust auto-detection**: Whether `tad.sh --platform codex` can pre-trust the project or if manual trust is always required. Likely always manual (security model). Document in onboarding.
