## Cross-cutting observations

### (a) Injection level: SESSION vs SKILL

**Injected at SESSION level (every conversation, before any TAD command):**
- CLAUDE.md routing rules (§1 handoff reading rule, §2 usage table, §4 terminal isolation, §4.5 post-compact recovery, EnterPlanMode prohibition, research-tool exclusion).
- @imported knowledge: principles.md (full L1, ~15 entries including 12 SAFETY entries), patterns/_index.md, legacy category files (testing/ux/performance/api-integration/mobile-platform/frontend-design; several are placeholder-sized), frontend-design.md full text.
- Hooks: SessionStart (startup-health, notebook-dormant-sync), PreToolUse (Write/Edit Haiku check; gate/accept pre-checks), PostToolUse (trace emission, askuser capture) — these run mechanically regardless of which skill is active.
- Auto-memory MEMORY.md (user-level, outside repo).

**Loaded only at SKILL level (on command):**
- Alex/Blake full protocols (SKILL.md bodies, each ~2K lines) + their references/ (progressive loading: 28 Alex protocols, 2 Blake references load on the triggering step — circular-trigger content deliberately kept in body per principles.md).
- gate, tad-maintain, surplus, save-skill/save-workflow, research-notebook/research-github, capability-upgrade, release-runbook, knowledge-audit and all capability packs — each loads on invocation/keyword match only.
- Config modules load per command_module_binding at skill activation, not at session start.

Notable: the quality-gate system is enforced at BOTH levels — prompt-level (SKILL text, blocking rules) and mechanical (PreToolUse hooks blocking Gate 3/*accept without COMPLETION). Principles.md records the deliberate decision to keep mechanical enforcement minimal (fail-closed hook incident, 2026-04-15).

### (b) Internal duplications within TAD (factual, no verdicts)

1. **Knowledge health checking ×3**: Alex STEP 3.5 startup scan, `/knowledge-audit` skill, and knowledge-maintain step5b dedup-health metric all count/inspect project-knowledge entries at different granularities.
2. **Zombie/stale handoff detection ×2**: Alex STEP 3.5 zombie scan and tad-maintain drift-check.sh both detect aged/orphaned handoffs (drift-check also covers superseded chains and ghost tasks).
3. **Pack detection ×2 (by design)**: Alex step4_5 pack-awareness scan and Blake 1_5a auto-detection both match packs to task; Blake's is an explicit safety net for handoffs where Alex missed packs.
4. **Expert-review fan-out pattern ×3**: handoff-review.workflow.js (Gate 2), Blake Layer 2 (Gate 3), gate-review.workflow.js (per-AC verification) all spawn parallel narrow-scope reviewer agents against different artifacts.
5. **Knowledge capture ×2 (by design)**: Blake journal (Gate 3, raw) vs Alex distillation (Gate 4, typed) — intentional Capture/Distill separation; plus KA step7.C as a blocking fallback if distillation loop is skipped.
6. **State persistence ×2**: ralph-loop state YAML (machine resume) and session-state.md (human/compact recovery) checkpoint overlapping execution state in different formats.
7. **Reflexion recorded ×2**: in-execution reflexion_step (before fix) and completion-report Reflexion History (for hook parsing) restate the same diagnoses at two moments.
8. **Research preflight ×3**: research-notebook, research-github, and capability-upgrade each implement their own NotebookLM/gh CLI preflight checks.
9. **Express path vs Light TAD**: `*express` (≤5 files, explicit command) and adaptive-complexity "Light" depth overlap in spirit but differ in gates and entry.
10. **Version consistency checks ×2**: release-verify.sh version gates and tad-maintain FULL-mode version check.
11. **Sync mechanisms ×2 generations**: *sync push model (sync-registry.yaml) coexists with the newer pull-based update direction (MEMORY feedback_no-sync-pull-based), both still present in the repo.
12. **Native-capability overrides**: TAD explicitly disables three native features for its agents (EnterPlanMode, /deep-research, /code-review family) and re-implements the underlying function inside its own protocols — a structural overlap class relevant to the upcoming comparison.
