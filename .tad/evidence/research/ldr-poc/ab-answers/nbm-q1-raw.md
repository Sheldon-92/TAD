**模型上下文协议 (MCP) 官方支持两种主要的传输机制：Stdio 传输 (Stdio transport) 
和 Streamable HTTP 传输 (Streamable HTTP transport) [1]。**

这两者在通信媒介、部署架构、服务能力及安全策略上存在着以下显著的不同：

### 1. 通信机制与协议实现
* **Stdio 传输**：通过**标准输入/输出流 (standard input/output streams)** 
在进程之间进行直接的管道通信 [1]。
* **Streamable HTTP 传输**：利用 **HTTP POST** 
来传送客户端向服务器发送的消息，并可选配**服务器发送事件 (Server-Sent Events, 
SSE)** 以提供流式数据传输能力 [1]。

### 2. 部署场景与性能表现
* **Stdio 
传输**：专门针对运行在**同一台机器上的本地进程**。由于不经过网络协议栈，它具有**
最佳的性能，且没有任何网络开销** [1]。例如，在本地运行的 filesystem 服务器就采用
Stdio 传输，这通常被称为“本地”MCP 服务器 [2]。
* **Streamable HTTP 传输**：专门用于**远程服务器通信** 
[1]。例如，部署在第三方云平台（如 Sentry 平台）上的服务就会使用 Streamable HTTP 
传输，这通常被称为“远程”MCP 服务器 [2]。

### 3. 连接模型与客户端数量
* **Stdio 传输**：采用一对一模型，一个本地 MCP 服务器进程通常只服务于**单个 MCP 
客户端** [3]。
* **Streamable HTTP 传输**：采用一对多模型，一个远程 MCP 
服务器由于暴露在网络中，通常会同时服务于**许多个不同的 MCP 客户端** [3]。

### 4. 身份验证与安全防护
* **Stdio 传输**：由于通信完全局限在本地系统内部，因此不涉及复杂的跨网络安全鉴权
[1]。
* **Streamable HTTP 传输**：为了应对跨网络通信的安全挑战，它支持标准的 **HTTP 
身份验证方法**，包括 **Bearer 令牌 (bearer tokens)**、**API 密钥 (API keys)** 
以及**自定义请求头** [1]。在需要获取授权令牌的场景中，MCP 官方推荐使用 **OAuth**
协议 [1]。

---

**核心共同点**：尽管底层的通信机制大相径庭，但**传输层对上层的数据协议层（Data 
Layer）进行了完美抽象** [4]。这意味着无论你选择 Stdio 还是 Streamable HTTP 
传输通道，上层的数据交互都统一采用完全一致的 **JSON-RPC 2.0 消息格式** [4]。
