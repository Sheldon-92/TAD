Answer:
**模型上下文协议 (MCP) 
定义的工具调用生命周期是一个紧密衔接的状态流，涵盖了从连接协商、工具发现、大语言
模型 (LLM) 选择、用户安全确认、实际执行到最后的错误处理与动态更新的完整过程 
[1-3]。**

该生命周期在具体流程中主要包含以下几个关键阶段：

### 1. 建立连接与能力协商 (Initialization)
在能够进行任何工具调用之前，客户端与服务器必须通过**生命周期管理**完成握手并确定
彼此支持的能力 [1, 2]：
*   **客户端发送 `initialize` 
请求**：该请求用于建立连接并进行协议版本及支持能力的协商 [2, 4]。
*   **服务器声明 `tools` 能力**：支持工具调用的服务器必须在响应中声明 `tools` 
能力 [5, 
6]。例如，如果服务器支持在工具列表发生变化时向客户端发送实时通知，它会在 
capability 中声明 `"listChanged": true` [5, 6]。
*   
**客户端发送就绪通知**：在完成能力协商并成功初始化后，客户端会向服务器发送一个 
`notifications/initialized` 准备通知，此时连接正式确立 [7]。

### 2. 工具发现 (Tool Discovery)
连接就绪后，客户端需要“发现”服务器上具体有哪些可用工具：
*   **客户端发送 `tools/list` 
请求**：客户端主动发起该请求（该操作支持分页）以查询服务器端可用的工具 [3, 8]。
*   **服务器返回工具元数据列表**：服务器在响应中提供一个 `tools` 
数组，包含每个工具的详尽定义 [8]：
    *   `name`：在服务器命名空间内的唯一标识符，作为调用的主键 [9, 10]。
    *   `description`：对该工具功能和适用场景的自然语言描述，供 LLM 
理解何时该使用它 [9, 10]。
    *   `inputSchema`：使用 JSON Schema 定义的输入参数规范，客户端及 LLM 
据此进行类型与参数验证 [9, 10]。
    *   `outputSchema`（可选）：定义预期输出结构的 JSON Schema [10, 11]。
*   **构建统一工具注册表**：AI 
宿主应用在获取所有连接服务器的工具后，会将它们整合进一个统一的工具注册表中，供大
语言模型访问 [12, 13]。

### 3. 工具选择与用户确认 (Tool Selection & Consent)
*   **LLM 做出决策并拦截**：在用户与 AI 交互的过程中，LLM 
根据用户的提示词和上下文，自动选择最合适的工具并生成符合 Schema 的入参 [14, 
15]。AI 宿主应用会截获大模型的工具调用指令 [14]。
*   **人机协同与确认 (Human in the 
Loop)**：出于安全和信任考量，对于敏感或具有外部副作用的操作，客户端**应当在调用
前向用户发起确认提示** [6, 
16]。向用户清晰展示工具的输入参数，能够有效防止恶意或意外的数据外泄 [16]。

### 4. 工具调用与执行 (Tool Invocation & Execution)
用户授权同意后，调用正式进入执行阶段 [3, 17]：
*   **客户端发送 `tools/call` 请求**：向目标 MCP 服务器发送请求，其中包含 [17]：
    *   `name`：必须与发现阶段获取的工具名称精确一致 [17]。
    *   `arguments`：完全符合工具输入 Schema 的具体参数 [17]。
    *   标准的 JSON-RPC 2.0 格式与用于关联请求响应的唯一 `id` [17]。
*   **服务器执行操作**：服务器执行具体的业务逻辑，如数据库查询、API 
调用或数学计算 [15, 18]。
*   **服务器返回响应**：服务器返回包含执行结果的响应 
[18]。其结果可能包含多媒体内容（在 `content` 
数组中返回，支持文本、图像、音视频、资源链接或嵌入式资源等）[19, 20]，也可能通过
`structuredContent` 字段直接返回 JSON 对象 [11]。如果定义了 
`outputSchema`，服务器返回的结构化结果必须严格符合此规范 [11]。
*   
**反馈大模型**：宿主应用收到结果后，将这些执行数据注入到当前对话中，由大模型整合
为最终的文本或回答反馈给用户 [14, 18]。

### 5. 错误处理 (Error Handling)
如果在工具生命周期中遇到异常，MCP 规定了清晰的错误分层机制 [21]：
*   **协议错误 (Protocol Errors)**：使用标准的 JSON-RPC 
错误，用于处理协议层面的问题，例如“未知工具”、“无效的参数”或“服务器内部错误” 
[21]。
*   **工具执行错误 (Tool Execution 
Errors)**：如果是工具内部的业务或接口调用失败，服务器仍会返回正常的 JSON-RPC 
响应，但会在工具调用结果中明确标记 `isError: true` [21]。

### 6. 工具列表动态更新 (Dynamic Updates)（可选）
在某些动态环境中，服务器的可用工具可能随时发生变更 [22]：
*   **服务器主动发送通知**：在初始化中声明了 `listChanged: true` 
的服务器在工具列表变化时，会向客户端发送 `tools/list_changed` 通知（由于是 
JSON-RPC 通知，无需客户端响应）[3, 23-25]。
*   **客户端刷新缓存**：客户端收到通知后，通常会立即重新发送 `tools/list` 
请求，拉取最新的工具定义，并动态刷新 LLM 可用的工具注册表 [3, 13, 22]。
