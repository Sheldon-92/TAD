Answer:
在 Model Context Protocol (MCP) 官方规范中，针对**服务端实现者（Server 
Implementers）**提出了以下关键的安全考量与防护要求：

### 1. 必须执行的核心安全要求（MUST）
根据 MCP 
的工具（Tools）规范，服务端在执行工具调用时**必须（MUST）**实施以下四项防御措施 
[1]：
*   **验证所有工具输入（Validate all tool inputs）** 
[1]：由于工具调用本质上代表了任意代码的执行路径 
[2]，服务端必须对客户端传入的所有参数进行严格的格式和内容校验。
*   **实施妥善的访问控制（Implement proper access controls）** 
[1]：必须确保只有经过授权的客户端或用户，才能调用对应的工具、读取特定的敏感资源
。
*   **限制工具调用频率（Rate limit tool invocations）** 
[1]：服务端必须具备限流机制，以防止接口被恶意高频调用、滥用，或导致拒绝服务（DoS
）攻击。
*   **净化工具输出（Sanitize tool outputs）** 
[1]：在将工具的执行结果返回给客户端之前，必须过滤和净化输出数据，防止敏感信息意
外外泄，或输出包含能恶意诱导 LLM 的内容。

### 2. 身份验证与传输层安全（Transports & Authentication）
对于运行在网络环境、使用**流式 HTTP 传输（Streamable HTTP transport）**的远程 
MCP 服务端，安全考量包括：
*   **支持标准的 HTTP 认证方式** [3]：服务端需要支持 Bearer 令牌、API 
密钥以及自定义标头等标准认证机制 [3]。
*   **推荐采用 OAuth 协议** [3]：MCP 官方建议通过 **OAuth** 
来获取身份验证令牌，以确保令牌生命周期的安全性 [3]。

### 3. 通用安全与隐私指南（SHOULD）
虽然 MCP 协议本身无法在技术底层强制约束所有安全行为，但官方实现 guidelines 
建议服务端实现者**应当（SHOULD）** [4]：
*   **提供清晰的安全 implications 文档** 
[4]：明确说明服务端所暴露的工具和资源可能带来的安全影响。
*   **在功能设计中融入隐私考量** 
[4]：遵循最小必要原则，避免不必要的敏感用户数据传输。
*   **遵循行业安全最佳实践** 
[4]：在具体代码实现中应用标准的应用程序安全防护手段。

### 4. 需重点防范的潜在威胁
除了协议本身定义的规则，安全研究也指出 MCP 
生态面临以下安全漏洞，服务端在编写具体工具逻辑时需要特别防范：
*   **提示词注入（Prompt Injection）** [5]：攻击者可能通过输入注入恶意指令，诱导
LLM 滥用服务端工具。
*   **毒化工具与数据外泄（Poisoned Tools & Data Exfiltration）** 
[5]：需确保服务端工具之间有明确的安全隔离，防止某个被毒化的工具利用其他已连接的
工具将敏感数据秘密向外传输。
