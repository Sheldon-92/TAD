TAD 学会了"看懂代码结构"

v2.13.1 — 从一个 GitHub 项目的评估，到发现 Claude Code 自己就有这个能力

起因

你问了一个问题：codegraph-rust 对我们有用吗？

codegraph-rust 是一个开源工具，能把代码变成一张"关系图"——哪个函数调用了哪个函数、哪个文件依赖哪个文件。AI 助手可以查询这张图来理解代码。

这个想法很好。但在研究它的过程中，我们发现了一件更重要的事。

问题

TAD 的 Alex 和 Blake 一直在"猜"代码结构

当 Alex 设计一个新功能时，他需要知道"改了这个函数，还有哪些地方会受影响"。但他一直用的方法是：

以前的做法
Alex: 我要改 getApiUrl 这个函数... Alex: grep "getApiUrl" → 找到 30 处文字匹配 Alex: 但其中有 import 语句、注释、类型定义... Alex: 真正调用它的到底有几个？不确定 Alex: 算了，我猜大概改 3 个文件就够了 ← 经常猜错

这就像装修房子时，不看图纸，凭感觉猜"拆这面墙会不会影响楼上"。

我们的 architecture.md 里记录了 4 次因为猜错文件范围导致的返工。这是一个真实的、反复出现的痛点。

发现

Claude Code 自己就有这个能力——我们从来没用过

在评估 codegraph 的过程中，我检查了 Claude Code 自带的工具清单，发现了一个叫 LSP（Language Server Protocol）的工具。

类比

LSP 就像代码世界的"GPS 导航"。你问它"谁在调用这个函数"，它能精确告诉你——不是文字搜索，而是真正理解代码结构后的回答。

但这个能力有一个前提：需要安装一个小插件。Claude Code 插件市场里已经有 12 种语言的 LSP 插件，TypeScript 的只要一条命令就能装好。

选择

为什么选了 LSP，没选 codegraph？

codegraph-rust（第三方）

  • 需要安装 SurrealDB 数据库
  • 需要配置 AI 向量模型
  • 需要编译 Rust 程序
  • 每个项目都要单独搭环境
  • 依赖一个个人开发者维护
  • 功能更强（语义搜索等）

LSP（Claude Code 自带）

  • 一条命令安装
  • 零外部依赖
  • Anthropic 官方维护
  • 12 种语言开箱即用
  • 覆盖 80% 的实际需求
  • 没有就自动回退到以前的方式
类比

这就像你的手机本来就有内置地图 App，但你从来不知道。有人推荐你买一个专业的车载导航仪（功能更强），你去研究的时候发现——等等，手机地图已经够用了，而且不用额外花钱装。

改了什么

TAD 的 Alex 和 Blake 现在会"看图纸"了

Alex 设计功能时（新增的步骤）
检测项目语言 自动识别是 TypeScript / Python / Swift 等
检查 LSP 是否可用 如果没装 → 自动安装插件（下次生效）→ 本次用老方法
LSP 查询"谁在调用这些函数" 对要修改的每个关键函数，找出所有调用方
自动发现遗漏的文件 如果有调用方不在改动列表里 → 警告 "这个文件也会受影响"
输出精确的改动范围 不再猜测，基于代码结构的真实分析
Blake 写代码前（新增的步骤）
查看"爆炸半径" 改这个函数，会波及多少个文件？影响链有多长？
心中有数再动手 知道影响范围后再写代码，避免改了一个地方、别的地方悄悄坏了
结果

对你来说意味着什么

完全无感的体验升级

你不需要做任何事。下次在任何项目里启动 Alex 或 Blake 时：

如果项目有 TypeScript/Python 代码 → Alex 会自动帮你装 LSP 插件（第一次需要重启对话才生效）→ 之后每次设计功能时，Alex 都会用精确的代码分析来判断改动范围，而不是猜。

如果项目没有代码（比如纯文档项目）→ 一切照旧，完全不受影响。

已经同步到你的 14 个项目

menu-snap、my-openclaw-agents、合规ai、Next Guest、care... 所有注册项目都已经更新到 v2.13.1。每个项目里的 Alex 和 Blake 都有了这个新能力。

一句话总结

codegraph 启发了我们，但最终答案在 Claude Code 自己身上

你的提问让我们发现了一个藏在 Claude Code 里的、从未被 TAD 使用的原生能力。现在 Alex 和 Blake 不再是"找字符串的文本工具"，而是"能看懂代码结构的智能助手"。

这是 TAD 第一次有了代码结构理解层。
