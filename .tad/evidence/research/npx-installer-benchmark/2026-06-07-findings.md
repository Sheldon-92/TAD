**Method:** GitHub-First / WebFetch 源码与文档（NotebookLM auth 失效，preflight-fail fallback to in-session WebFetch；结论持久化于此）
**重点对标:** BMAD-METHOD (bmad-code-org/BMAD-METHOD, v4.4x, 100% OSS)

## BMAD 怎么做的（回答用户的核心问题）

### 1. 平台选择 = config-driven，不硬编码
- 入口: `npx bmad-method install` → 交互问 5 个问题，其一是"choose which AI tools/IDEs to integrate with"。
- 核心类 `IdeManager` [tools/installer/ide/manager.js] 读 **`platform-codes.yaml`**（声明每个 IDE 的 target 目录 + 特殊规则），支持 40+ 工具，统称 "unified configuration model"。
- 分流靠 `ConfigDrivenIdeSetup` —— **配置驱动，不为每个平台写代码**。
- 支持: claude-code, cursor, windsurf, roo, cline, trae, gemini-cli, vs-code-copilot…

### 2. 按平台装不同内容（关键 pattern）
| 平台 | 装到 | 特殊处理 |
|------|------|----------|
| Claude Code | `.claude/skills/` | `ancestor_conflict_check: true`（防父目录命令继承重复）|
| Cursor / Windsurf | `.agents/skills/` / `.windsurf/skills/` | "cross-tool standard" |
| GitHub Copilot | `.github/agents/` | 生成 **`.agent.md` pointer 文件**（指向完整 SKILL.md），`commands_filter: agents-only` 只装带 `[agent]` 段的 skill |
- **Verbatim Skill Installation**: 整目录 `cp`（非生成单文件 launcher）—— 与 TAD `tad.sh` 的 `cp -R` 同理念。
- **`commands_filter`** 决定每个平台装哪些 skill 子集。

### 3. 模块/expansion 选择
- 交互式 **checkboxes**: `core, bmm, bmb, cis, gds, tea`（模块代号）→ "Ready to install (all stable)?" 确认。
- CLI 非交互: `--modules bmm,bmb,cis`（逗号分隔）。
- `--list-tools` 列可选平台。

---

## BMAD 的两个缺口 = TAD 的差异化机会 🎯

1. **模块只显示代号，无描述** —— 用户在 checkbox 看到 `bmb / cis / gds / tea`，**没有一句话介绍**。
   → **TAD 应超越**：选 pack 时带一句话描述（我们 24 个 pack 的 SKILL.md frontmatter `description` 是现成的！）。这正是用户明确要的"选功能包时给简单介绍"。

2. **完全没有 context size / 最小化加载概念** —— BMAD 只管"装哪些模块"，不管装进去后**启动消耗多少 context**。
   → **TAD 可差异化**：这正是用户对 Codex 额度的核心担忧。BMAD 给不了答案；TAD 的"平台选择→Codex 装 13K 瘦版（非 86K Claude 版）"本身就是减压。

---

## 一个可借鉴的渐进加载 pattern（Copilot pointer）
BMAD 给 GitHub Copilot 装的是 `.agent.md` **pointer 文件**（小），指向完整 SKILL.md（按需加载），而非把全部内容塞进去。
→ 直接呼应用户选的"**渐进加载不删内容**"瘦身方式：装"核心激活层 + 按需 reference"，而非一次性全量。
