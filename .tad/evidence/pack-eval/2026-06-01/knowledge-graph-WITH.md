# Knowledge Graph Review: Full Microsoft GraphRAG over 40,000 internal docs (extract every entity → Neo4j → Global search)

The plan triggers all three reflex anti-patterns this pack exists to stop: **reflexive full GraphRAG**, **extract-everything with no entity resolution**, **Neo4j-by-reflex**, plus **Global-search-for-everything**. Four named anti-patterns in one four-line plan. Reviewed against `graphrag-architecture.md`, `kg-construction.md`, `entity-resolution.md`, `graph-database.md`, and the cross-cutting cost rule.

---

### P0 — Blocking (must fix before building)

- **[P0] Cross-Cutting Cost Rule + Rule ARC1 (graphrag-architecture): Full Microsoft GraphRAG chosen by reflex over 40,000 docs without ruling out LazyGraphRAG/LightRAG on evidence.** GraphRAG indexing cost scales with **DOCUMENT volume, not query volume** — full Leiden pre-summarization runs an LLM extraction + community-report pass over every chunk in all 40k docs (at the default 1,200-token chunk size that is on the order of hundreds of thousands of chunks). For the stated goal — broad *"what are the themes across all of this"* global-synthesis queries — **LazyGraphRAG matches full GraphRAG's global-search quality at 700x lower query cost and up to 99.9% lower upfront indexing cost**, by replacing LLM pre-summarization with lightweight NLP (noun-phrase + co-occurrence) and deferring LLM work to query time.
  → Default to **LazyGraphRAG** for this workload and prove full GraphRAG is needed on evidence, not reflex. If the corpus is updated frequently (internal docs usually are), **LightRAG** does incremental updates touching only affected nodes/relationships (<100 tokens/query, no full rebuild) — full GraphRAG would force a costly full re-index on every doc batch. (determinismLevel: deterministic — this is an upfront architecture decision.)

- **[P0] Cross-Cutting Cost Rule: No cost guardrails before a tens-of-thousands-of-chunks extraction.** The plan commits to "extract every entity" over 40k docs with no spend ceiling. Per the cross-cutting rule, before ANY full GraphRAG build you must install **(1) a hard spend limit, (2) a pipeline circuit breaker, and (3) per-pipeline cost attribution.** Without these the cost surfaces only at production scale.
  → Add all three guardrails as a precondition to any indexing run, and apply **Rule KGC6 selective extraction**: run full LLM extraction only on **structural regions** (headers, introductions, party lists, executive summaries) and **metadata-tag the body** chunks pointing back to core entities, bypassing full extraction on the bulk body text.

- **[P0] Rule ER1 + ER2 (entity-resolution): "Extract every entity, store it in Neo4j" skips entity resolution entirely — resolution is mandatory, not optional.** Storing raw extractions splits one concept across variant nodes ("Olympic Winter Games" vs "winter Olympic games"), **fragments relationship paths, and distorts node-degree / combined-degree topology** — the exact metrics Global search and community detection depend on. Garbage topology in → garbage themes out.
  → Insert the mandatory **two-stage resolution pipeline** between extraction and storage: lowercase normalize → **S-BERT** embeddings → **k-means clustering (cluster size = 128)** → within-cluster **fused BM25 + vector top-K = 16** candidates → LLM-guided canonical consolidation → canonical + alias map. Do NOT do naive O(n²) pairwise LLM dedup; the k-means(128)/top-K(16) pre-filter exists precisely to bound that cost. (semi-deterministic — numbers fixed, LLM merge calls vary.)

### P1 — Required (fix before production)

- **[P1] Rule ARC5 (graphrag-architecture): "Use Global search for everything" is the named anti-pattern.** Global search runs a **parallel Map-Reduce over entire community-report levels** (Map step `MATCH (c:__Community__) WHERE c.level = $level`, each batch scored 0-100 for helpfulness, score-0 filtered, survivors packed for Reduce). It is prohibitively expensive and **blind to granular edges**. It is correct ONLY for the broad-theme synthesis questions — which is genuinely this user's stated primary goal, so Global is right for *those* — but "for everything" means every entity-specific follow-up ("what did doc X say about Y") will also pay full Map-Reduce.
  → Route by intent: keep **Global** for the broad cross-document theme questions; route entity-specific / localized factual queries to **Local search** (entity-embedding nearest-neighbor seed → 1-2 hop traversal) at a fraction of the cost; use **Drift** for hybrid broad→precise questions.

- **[P1] Rule ARC4 (graphrag-architecture): No Leiden level chosen for query breadth.** Leiden builds a multi-layered hierarchy — **Level 0 = broad parent communities**, higher levels = specific micro-domains. The "themes across all of this" goal maps to **low Leiden levels (Level 0)**; querying a high level would be too narrow.
  → Explicitly target Level 0 community reports for the global-theme queries; reserve higher levels for sub-topic drill-down.

- **[P1] Rule GDB1 (graph-database): Neo4j chosen by reflex without matching it to the workload.** Neo4j is the *defensible* default here only if the workload is a **large, static, historical batch graph that must scale past physical RAM** (on-disk + page cache, ACID, Causal Clustering) — which a 40k-doc batch corpus plausibly is. But the choice was made by reflex, not against the profile. Neo4j **pre-allocates 4-5 GB JVM heap** (benchmark: 2,668 MB for 16k nodes vs **Memgraph 415 MB**, **FalkorDB 496 MB**) and has slow cold starts. QPS @ 8 threads: Neo4j 738 / Memgraph 467 / **FalkorDB 837**.
  → Confirm the workload is batch/historical and exceeds RAM before committing to Neo4j (then it IS the right pick). If the internal docs stream in continuously, **Memgraph** (native C++ in-memory, sub-ms latency, **native Kafka/Redpanda/Pulsar connectors**) avoids an ETL hop. If you want high throughput at low memory footprint, **FalkorDB** (compressed sparse matrices on Redis) — but **do not plan past 8 threads** (Redis single-threaded core plateaus). (determinismLevel: deterministic — selection from fixed profiles.)

- **[P1] Rule KGC2 + KGC3 (kg-construction): "Extract every entity" implies a bare zero-shot extract-then-store pipeline — no schema classification.** Zero-shot is the MVP baseline, **not the production default** for hard/jargon-heavy internal text (internal docs are full of org-specific jargon, acronyms, project codenames). Extract-then-store with no schema check lets every hallucinated entity into the graph, where it later fragments traversals.
  → Apply **OntologyRAG / TrustGraph classify-before-store**: classify each entity against the domain schema at extraction time. Upgrade the prompt from zero-shot to **Chain-of-Thought** (for relationship directionality) or **Stepwise Decomposition** (for a debuggable noisy stage). (semi-deterministic.)

### P2 — Advisory (improves quality / cost / governance)

- **[P2] Rule KGC1 (kg-construction): No ontology mentioned — bottom-up-only extraction drifts.** Combine **top-down OWL guidance** (translate the org's competency questions / business specs into a formal ontology) with **bottom-up LLM refinement**. Consider Ontogenia-style Metacognitive Prompting for self-correction during ontology synthesis.

- **[P2] Rule ER3 + ER4 (entity-resolution): Plan implies auto-merging all duplicates — destroys auditability for internal/possibly-legal docs.** Use **governed, threshold-driven merges**: **Auto-Merge** only above the high-confidence threshold; route **moderate-confidence pairs to a temporary `SAME_AS` edge + human reviewer**; keep below-threshold as separate **New Nodes**. Record each candidate as a reified **`ontodup:MatchAssertion`** carrying confidence + source algorithm + timestamp + approval status so merges are auditable and revertible. (ER5: the LLM consolidation step must also account for plurals / abbreviations / tenses — lowercasing alone won't catch them.)

- **[P2] Rule ARC3 (graphrag-architecture): If you do proceed with extraction passes, deploy a cheaper model for gleanings.** Gleanings (multi-pass extraction for recall) are token-intensive over 40k docs — use `gpt-4o-mini` for the gleaning passes to mitigate API spend, and keep the default 1,200-token chunk size unless you have a measured reason to change it.

- **[P2] Rule KGC5 (kg-construction): If internal facts change over time, use a bi-temporal model.** Tag facts with **Event Time** (when true in the real world) + **Ingestion Time** (when the agent learned it) so a deprecated/changed fact can be invalidated in real time **without a full index recomputation** — directly relevant since full GraphRAG otherwise forces full rebuilds.

- **[P2] Rule ARC6 (graphrag-architecture): When community context overflows the LLM window, never raw-truncate.** Reduce via **Hierarchical Substitution** (replace a parent community's raw data with its sub-community reports, largest sub-communities first) → then **Trimming** (drop lowest node-degree / combined-degree elements). Raw truncation discards the highest-degree topological bridges.

### Architecture Recommendation

**Switch the default from full Microsoft GraphRAG to LazyGraphRAG.** The user's sole stated goal is broad global "what are the themes" synthesis — exactly where LazyGraphRAG **matches full GraphRAG global-search quality at 700x lower query cost and up to 99.9% lower indexing cost**. Full Microsoft GraphRAG is only justified if (a) the corpus is truly static/batch, (b) you need the deepest semantic pre-summarization, AND (c) you've accepted the upfront LLM pre-summarization bill over all 40k docs with hard spend limits + circuit breaker + cost attribution in place. If the internal docs update frequently, prefer **LightRAG** for incremental updates (<100 tokens/query) instead of re-running full Leiden. Whatever the choice: entity resolution (ER2 two-stage pipeline) and schema-classified extraction (KGC2) are non-negotiable, and selective extraction (KGC6) controls the document-volume cost.

### Graph Database Recommendation

**Neo4j is acceptable ONLY if confirmed as a large, static, historical batch graph that must scale past physical RAM** — then its on-disk + page-cache + ACID + Causal Clustering (HA quorum **N = 2F + 1**) profile fits, and index-free adjacency plus indexing (latency drops up to **1712x** when indexed) makes it performant. Validate that assumption rather than defaulting. If ingestion is **streaming/real-time**, choose **Memgraph** (415 MB vs Neo4j's 2,668 MB at 16k nodes; native Kafka/Redpanda/Pulsar; sub-ms latency) to skip the ETL hop. If the priority is **high QPS at low memory**, choose **FalkorDB** (837 QPS @ 8 threads, 496 MB) — but cap planning at 8 threads (Redis single-threaded plateau). Always profile indexed-vs-unindexed before blaming architecture for latency.
