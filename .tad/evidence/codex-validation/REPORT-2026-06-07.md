## 诚实记录的差异 / 限制

1. **非交互单轮 `codex exec` 无法完成需真人多轮回答的完整苏格拉底** → 需 resume 或交互模式(已文档化，非缺陷)。
2. **R3(resume 多轮上下文) 未实测** → 下一步应补。
3. **交互入口未测** → 本实验走脚本式(`codex exec` 喂 SKILL)；`codex` + `AGENTS.md` + "当 Alex" 的交互路径未验证。
4. **Codex 输出冗长**(reasoning + final message 重复打印) → 不影响功能，但脚本化解析需注意。
5. **n=1** → 单次运行，非统计性；Codex 输出有不确定性，个别 run 可能表现不同。

## 高风险项验证（第三轮，2026-06-07 · "一等公民"两个最危险假设）

### 实验 C — 专家审查（并行→顺序降级）→ ✅ 强 PASS
拿一个**埋了 3 个缺陷**的 handoff,跑 2 个独立 `codex exec` reviewer:
- **code-reviewer**(静态审查视角): 抓到 `grep -P` 不可移植 + AC2/AC5 矛盾(P0) + AC1 命令不验证它声称的东西 + §7 漏 test 文件。
- **backend-architect**(架构视角,**实际读源码 + grep call sites**): 独有发现 — `slugify` 现用 `input="$*"`,新增位置参数会**破坏现有调用契约**(P0);grep 全库精确定位 `test_slugify.sh:5/:14` 是 downstream consumer。还自发加载了 web-backend capability pack。
- **关键结论: 顺序 reviewer 没有退化成同质化。** 两个视角真正互补,architect 做了 code-reviewer 不会做的源码勘查。**"并行→顺序会拖垮质量"这个最危险假设被推翻。** 缺陷 100% 被抓 + 额外发现更深 bug。

### 实验 D — 交互入口（AGENTS.md 角色路由）→ ✅ PASS（真 TUI 保留）
- 只发 `当 Alex`(**不喂 SKILL**) → codex 靠 `AGENTS.md` 自己读取 `codex-alex-skill.md`(1350 行) → 跑完整激活协议 → health check 精确报出真实 handoff/session 状态 → **适配了中文** → 显示菜单 → 停下。**AGENTS.md 路由机制走通。**
- ⚠️ 保留: 真正的 TUI 交互(`codex` 而非 `codex exec`)需真人 TTY,本环境(Bash 工具)无法测;仅验证了 AGENTS.md 路由逻辑。

### 第三轮新发现（对 adapter 有用）
1. **`codex exec "<prompt>"` 不带 stdin 会阻塞**在 `Reading additional input from stdin...`(非交互/后台环境)。脚本化调用须 `</dev/null` 或 `cat file | codex exec`。真交互 `codex` TUI 无此问题。→ 建议补进 README troubleshooting。
2. **`codex exec resume` 不接受 `-s` flag**(sandbox 继承自原 session)。
