# Gate Design Patterns (Layer 2)

> Reusable patterns for quality gate architecture, expert review, and verification integrity.

---

### Ralph Loop Two-Layer Architecture - 2026-01-26
- **Discovery**: Two-layer quality (Layer 1: fast self-check, Layer 2: expert review) with circuit breaker (3 same errors → escalate), state persistence for crash recovery, and priority groups (blocking gate before parallel verification).
- **Action**: Apply this pattern to any iterative workflow needing external quality validation.

### Gate Responsibility Matrix - 2026-01-26
- **Discovery**: Technical experts (code-reviewer, test-runner, security, performance) → Blake's Gate 3 v2. Business acceptance (requirement verification, user approval) → Alex's Gate 4 v2.
- **Action**: Separate technical automation from business judgment in quality gates.

### Epic Architecture: Spike-Driven Pivots - 2026-04-07
- **Discovery**: Plan for 2-3 architectural pivots as default. Split "design" and "contract validation" into separate spikes. Favor simpler architectures after mechanism surprises. Budget expert review by risk, not code volume. Light TAD + spike de-risks "is mechanism X possible?" unknowns. Build multi-axis verdicts (integration/accuracy/latency/cost).
- **Action**: Insert Light TAD spike as Phase 1 for mechanism unknowns. Pre-allocate forward-compatibility fields.

### Expert Review Blind Spots - 2026-04-04
- **Discovery**: (1) Pre-handoff vs post-impl reviewers catch different things — same agent type, different artifact = different findings. Post-impl catches blast radius; pre-handoff catches spec bugs. (2) Cross-file internal references (peer references) are missed by expert review — broader grep is the safety net. (3) Pre-handoff reviewers grep the SPEC; post-impl reviewers grep the WHOLE codebase.
- **Action**: Always run post-impl Layer 2 review separately from pre-handoff review. Save both with distinguishing suffixes.

### Gate 4 Verification Integrity - 2026-04-14
- **Discovery**: Re-derive key pass/fail numbers from primary evidence, don't read Blake's summary. Run the same command Blake ran. Dogfood check: apply any validator to the top-level artifact. Check `git status` for uncommitted changes.
- **Action**: Gate 4 requires: raw-data recompute, validator dogfood, git status before declaring acceptance.

### honest_partial_protocol: Real-Use Validation - 2026-04-25
- **Discovery**: Self-installed hard rules without env-aware exception clauses WILL eventually deadlock self-dogfood (e.g., sub-agent quota exhaustion). honest_partial is the right exit — PARTIAL not PASS, explicit conflict statement, named options for Alex. Audit script correctly refused PASS. Future hard-rule handoffs need "env edge case" sub-section. Alex MUST NOT pick options that violate the rule under acceptance-time pressure.
- **Action**: Include env edge cases in hard-rule handoffs. Use honest_partial for environmental deadlocks, not silent compliance.

### Epic Auto-Conductor: Sub-Agent Constraints - 2026-05-14
- **Discovery**: (1) Sub-agents have NO Agent tool — cannot spawn their own sub-agents. All review at Conductor level. (2) Sub-agents fabricate reviewer labels — don't trust self-generated "CR-P0-1" labels. (3) File is source of truth, prompt is not — write REVIEW.md, don't pass via prompt text. (4) worktree isolation doesn't protect downstream projects. (5) Conductor must ground before dispatching Alex.
- **Action**: Conductor = only real reviewer layer. Every step produces a persistent file. Sub-agent instructions reference file paths not inline content.

### YOLO Mode Strengths and Constraints - 2026-05-15
- **Context**: First full YOLO Epic execution covering research→build→validate→freeze→cross-agent→template.
- **Discovery**: (1) **Research must be Conductor-side**: NotebookLM is stateful and sequential — cannot be delegated to Blake sub-agents. Conductor runs research in Y2, saves to evidence file, Blake reads file in Y5. (2) **Pipeline parallelization works**: while Blake builds pack N, Conductor can research pack N+1 (notebook creation + source addition + deep research launch). This overlapping saved ~40% wall-clock time. (3) **Expert review diminishing returns on repetitive packs**: first pack (ai-evaluation) got full 2×2 expert review (design + implementation) catching real P0s. Subsequent packs following same pattern got lighter Conductor-level verification — appropriate for identical architecture. (4) **Cross-agent validation is cheap and high-value**: two `codex exec` calls (<2 min total) proved the AGENTS.md routing pattern works. Worth doing for every major pack change.
- **Action**: For future YOLO Epics: (A) Always pipeline research ahead of build. (B) Full expert review on first instance of a pattern, lighter verification on repetitions. (C) Include at least 1 cross-agent smoke test per Epic. (D) Conductor owns NotebookLM, Blake owns file creation — never mix.

### Non-Dev Execution Track: A Rubric Gate Is Only Credible If It Can FAIL; Branch a Fenced-YAML Protocol via Additive Sibling, Not ELSE-Wrap - 2026-05-31
- **Context**: EPIC-nondev-execution-track (4-phase YOLO) gave TAD its first NON-CODE delivery lane: `task_type: deliverable` routes Gate 3/4 to additive sibling sections that score a content artifact (research/voice/video/PRD) against a pack rubric via an INDEPENDENT judge sub-agent, instead of `tsc/test/lint`. Turns the orphaned non-dev packs (academic-research/ai-voice-production/video-creation/product-thinking) into a runnable pipeline. Phase-3 dogfood: a real LLM-as-judge-bias literature brief → judge-A 0.737 PARTIAL → revise → fresh judge-B 0.7725 PASS (3 distinct agents).
- **Discovery**: (1) **A non-dev quality gate is only credibly NOT-theater if it can FAIL** — the dogfood's strongest evidence was the round-1 PARTIAL (the gate discriminated + BLOCKED), not a clean PASS. Future non-dev dogfoods should target a deliverable that plausibly lands PARTIAL first, then prove the PARTIAL→revise→PASS recovery loop. A gate that only ever passes its own first artifact is unproven. (2) **judge ≠ producer is enforced by SEPARATE fresh Agent spawns with paths-only prompts** (artifact path + rubric path; never the producer's reasoning or a prior judge's score) — self-enhancement bias is ~10-15%; a reused judge anchors on its prior verdict. The round-2 judge correctly did NOT inflate (novelty stayed 0.55; the +0.035 came only from the methodology/ethics fixes actually made). (3) **Producer for a deliverable lane is Conductor-side, NOT Blake** — research tools (WebSearch/NotebookLM) can't run inside a Blake sub-agent (architecture.md "Research must be Conductor-side"); judge≠producer is defined relative to the Conductor-spawned producer. (4) **To branch a single-fenced-YAML protocol block (Gate 3 = one ```yaml fence, lines 95-338), use an ADDITIVE SIBLING section + a guard line in the prose header OUTSIDE the fence — NEVER an ELSE-wrap** (wrapping top-level YAML keys under `ELSE:` re-indents every line = byte change, contradicting "augment-not-replace"). Byte-checks on a block that gets a line inserted ABOVE it MUST be offset-aware (`HEAD A,B` vs `current A+N,B+N`) or content-anchored (awk fence-body extraction) — a fixed `sed -n 'A,Bp'` on both sides silently mis-fires once the block shifts (the verification re-review caught this, not mental simulation). (5) **Wiring an observability marker to a role the same change FORBIDS = dead-on-arrival telemetry** — the deliverable `gate3_verdict` marker was templated as "filled by Blake" while the same commit barred Blake from the lane → zero gate_result emission (the "paper machine" failure). When reassigning who-does-X, grep every consumer that names the old role.
- **Action**: For any rubric/quality gate on non-code deliverables: (a) prove it can FAIL via a real PARTIAL→revise→PASS loop, not a clean pass; (b) enforce judge≠producer with separate fresh Agent spawns + paths-only prompts + fresh-judge-per-round; (c) producer = Conductor-side for research/content. For branching a fenced-YAML protocol: additive sibling section + guard line outside the fence, offset-aware/content-anchored byte-checks, and re-grep every role-named consumer when reassigning a writer. Pin a machine-readable `verdict:` line for any downstream gate that greps the score.

### Decouple Detect-from-Heal at Release Gates — 2026-06-01
- **Context**: P3 codex-parity-phase3-releasegate. Backend-architect P0: auto-heal (regen+ship in *publish) silently ships an unreviewed LLM-generated 46KB artifact into a permanent tag → 14 downstream projects. User re-decided: detect-only + separate reviewed regen.
- **Discovery**: A release-time quality gate on derived artifacts (Codex editions, generated docs, compiled bundles) should DECOUPLE detection from remediation. Detection = fast read-only check in the release flow (shell grep/awk, no external tools needed, fail-CLOSED). Remediation = separate human-invoked command that writes, with the result reviewed via `git diff` before commit. This separation means: (1) the gate never fails due to tool unavailability (codex auth expired doesn't block the DETECTION), (2) unreviewed AI-generated content cannot enter a tagged release (the human reviews the diff), (3) the common case (no drift) is silent-pass with zero latency added. The anti-pattern is "self-heal" — auto-regen+auto-commit in the release flow, which makes "hard block on drift" a lie (it silently heals instead of blocking, and ships an unreviewed artifact).
- **Action**: For any release gate on derived/generated artifacts: detect-only in the release flow (read + block, never write); separate human-invoked regen command (write + review + commit). The gate and the regen share the same check script but run at different times with different permissions. Never auto-commit regenerated content in a release flow.

### Awk Range Pattern Start/End Overlap on macOS — 2026-06-03
- **Context**: AC7 verification for YOLO workflow P3. The awk command `awk '/^yolo_execution_protocol:/,/^[a-z_]+:/' SKILL.md | wc -l` returned 1 instead of the expected ~50. The start pattern `^yolo_execution_protocol:` also matches the end pattern `^[a-z_]+:` because the key starts with lowercase letters.
- **Discovery**: On macOS BWK awk, when a range pattern's start line also matches the end pattern, the range opens and closes on the same line (returns 1 line). This makes the verification command useless. Fix: exclude the start key from the end pattern: `awk '/^yolo_execution_protocol:/{found=1} found && /^[a-z_]+:/ && !/^yolo_execution_protocol:/{print;found=0;next} found{print}'`.
- **Action**: When writing AC verification commands with awk range patterns, ensure the end pattern cannot match the start line. Use explicit exclusion (`&& !/start_pattern/`) or use a state-variable approach instead of comma-range syntax.
