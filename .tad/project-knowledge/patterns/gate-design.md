# Gate Design Patterns (Layer 2)

> Reusable patterns for quality gate architecture, expert review, and verification integrity.

---

### Ralph Loop Two-Layer Architecture - 2026-01-26
- **Discovery**: Two-layer quality (Layer 1: fast self-check, Layer 2: expert review) with circuit breaker (3 same errors → escalate), state persistence for crash recovery, and priority groups (blocking gate before parallel verification).
- **Action**: Apply this pattern to any iterative workflow needing external quality validation.
- **failure_mode**: Naive default: single-pass review with no escalation or crash recovery. Why wrong: repeated errors go undetected, and a crash loses all review progress — no circuit breaker means the same mistake cycles forever.

### Gate Responsibility Matrix - 2026-01-26
- **Discovery**: Technical experts (code-reviewer, test-runner, security, performance) → Blake's Gate 3 v2. Business acceptance (requirement verification, user approval) → Alex's Gate 4 v2.
- **Action**: Separate technical automation from business judgment in quality gates.
- **failure_mode**: Naive default: mix technical checks and business acceptance in one gate owned by one role. Why wrong: technical experts rubber-stamp business requirements they can't evaluate, and business reviewers skip automated checks they don't understand — both pass without real verification.

### Epic Architecture: Spike-Driven Pivots - 2026-04-07
- **Discovery**: Plan for 2-3 architectural pivots as default. Split "design" and "contract validation" into separate spikes. Favor simpler architectures after mechanism surprises. Budget expert review by risk, not code volume. Light TAD + spike de-risks "is mechanism X possible?" unknowns. Build multi-axis verdicts (integration/accuracy/latency/cost).
- **Action**: Insert Light TAD spike as Phase 1 for mechanism unknowns. Pre-allocate forward-compatibility fields.
- **failure_mode**: Naive default: design the full architecture upfront, then implement in one phase. Why wrong: mechanism unknowns (e.g., "can library X do Y?") discovered mid-implementation force expensive pivots that a cheap spike would have caught in hours.

### Expert Review Blind Spots - 2026-04-04
- **Discovery**: (1) Pre-handoff vs post-impl reviewers catch different things — same agent type, different artifact = different findings. Post-impl catches blast radius; pre-handoff catches spec bugs. (2) Cross-file internal references (peer references) are missed by expert review — broader grep is the safety net. (3) Pre-handoff reviewers grep the SPEC; post-impl reviewers grep the WHOLE codebase.
- **Action**: Always run post-impl Layer 2 review separately from pre-handoff review. Save both with distinguishing suffixes.
- **failure_mode**: Naive default: run expert review only once (either pre-handoff OR post-impl) and assume it covers both. Why wrong: pre-handoff reviewers miss blast-radius issues only visible in code; post-impl reviewers miss spec bugs only visible in the design — and cross-file peer references slip through both unless you grep the whole codebase.

### Gate 4 Verification Integrity - 2026-04-14
- **Discovery**: Re-derive key pass/fail numbers from primary evidence, don't read Blake's summary. Run the same command Blake ran. Dogfood check: apply any validator to the top-level artifact. Check `git status` for uncommitted changes.
- **Action**: Gate 4 requires: raw-data recompute, validator dogfood, git status before declaring acceptance.
- **failure_mode**: Naive default: read Blake's summary numbers and accept at face value. Why wrong: summary numbers can be stale, cherry-picked, or computed from a dirty worktree — without re-deriving from primary evidence and checking git status, the gate accepts a fabricated or outdated result.

### honest_partial_protocol: Real-Use Validation - 2026-04-25
- **Discovery**: Self-installed hard rules without env-aware exception clauses WILL eventually deadlock self-dogfood (e.g., sub-agent quota exhaustion). honest_partial is the right exit — PARTIAL not PASS, explicit conflict statement, named options for Alex. Audit script correctly refused PASS. Future hard-rule handoffs need "env edge case" sub-section. Alex MUST NOT pick options that violate the rule under acceptance-time pressure.
- **Action**: Include env edge cases in hard-rule handoffs. Use honest_partial for environmental deadlocks, not silent compliance.
- **failure_mode**: Naive default: write hard rules without environment-aware exception clauses, then silently comply (report PASS) when the rule deadlocks in practice. Why wrong: silent compliance hides a real environmental conflict — the gate reports PASS while the rule was never actually satisfied, which is validation theater.

### Epic Auto-Conductor: Sub-Agent Constraints - 2026-05-14
- **Discovery**: (1) Sub-agents have NO Agent tool — cannot spawn their own sub-agents. All review at Conductor level. (2) Sub-agents fabricate reviewer labels — don't trust self-generated "CR-P0-1" labels. (3) File is source of truth, prompt is not — write REVIEW.md, don't pass via prompt text. (4) worktree isolation doesn't protect downstream projects. (5) Conductor must ground before dispatching Alex.
- **Action**: Conductor = only real reviewer layer. Every step produces a persistent file. Sub-agent instructions reference file paths not inline content.
- **failure_mode**: Naive default: trust sub-agent self-reported review labels and pass instructions via prompt text. Why wrong: sub-agents fabricate reviewer labels (no real review happened), prompt-passed content is ephemeral (lost on crash), and sub-agents cannot spawn their own sub-agents so any "review" they claim to run is fiction.

### YOLO Mode Strengths and Constraints - 2026-05-15
- **Context**: First full YOLO Epic execution covering research→build→validate→freeze→cross-agent→template.
- **Discovery**: (1) **Research must be Conductor-side**: NotebookLM is stateful and sequential — cannot be delegated to Blake sub-agents. Conductor runs research in Y2, saves to evidence file, Blake reads file in Y5. (2) **Pipeline parallelization works**: while Blake builds pack N, Conductor can research pack N+1 (notebook creation + source addition + deep research launch). This overlapping saved ~40% wall-clock time. (3) **Expert review diminishing returns on repetitive packs**: first pack (ai-evaluation) got full 2×2 expert review (design + implementation) catching real P0s. Subsequent packs following same pattern got lighter Conductor-level verification — appropriate for identical architecture. (4) **Cross-agent validation is cheap and high-value**: two `codex exec` calls (<2 min total) proved the AGENTS.md routing pattern works. Worth doing for every major pack change.
- **Action**: For future YOLO Epics: (A) Always pipeline research ahead of build. (B) Full expert review on first instance of a pattern, lighter verification on repetitions. (C) Include at least 1 cross-agent smoke test per Epic. (D) Conductor owns NotebookLM, Blake owns file creation — never mix.
- **failure_mode**: Naive default: run research and build sequentially (research all packs, then build all packs), and apply full expert review to every repetitive instance. Why wrong: sequential execution wastes ~40% wall-clock time that pipelining would save, and full review on identical-pattern repetitions yields diminishing returns while consuming budget that cross-agent smoke tests (cheap and high-value) never get.

### Non-Dev Execution Track: A Rubric Gate Is Only Credible If It Can FAIL; Branch a Fenced-YAML Protocol via Additive Sibling, Not ELSE-Wrap - 2026-05-31
- **Context**: EPIC-nondev-execution-track (4-phase YOLO) gave TAD its first NON-CODE delivery lane: `task_type: deliverable` routes Gate 3/4 to additive sibling sections that score a content artifact (research/voice/video/PRD) against a pack rubric via an INDEPENDENT judge sub-agent, instead of `tsc/test/lint`. Turns the orphaned non-dev packs (academic-research/ai-voice-production/video-creation/product-thinking) into a runnable pipeline. Phase-3 dogfood: a real LLM-as-judge-bias literature brief → judge-A 0.737 PARTIAL → revise → fresh judge-B 0.7725 PASS (3 distinct agents).
- **Discovery**: (1) **A non-dev quality gate is only credibly NOT-theater if it can FAIL** — the dogfood's strongest evidence was the round-1 PARTIAL (the gate discriminated + BLOCKED), not a clean PASS. Future non-dev dogfoods should target a deliverable that plausibly lands PARTIAL first, then prove the PARTIAL→revise→PASS recovery loop. A gate that only ever passes its own first artifact is unproven. (2) **judge ≠ producer is enforced by SEPARATE fresh Agent spawns with paths-only prompts** (artifact path + rubric path; never the producer's reasoning or a prior judge's score) — self-enhancement bias is ~10-15%; a reused judge anchors on its prior verdict. The round-2 judge correctly did NOT inflate (novelty stayed 0.55; the +0.035 came only from the methodology/ethics fixes actually made). (3) **Producer for a deliverable lane is Conductor-side, NOT Blake** — research tools (WebSearch/NotebookLM) can't run inside a Blake sub-agent (architecture.md "Research must be Conductor-side"); judge≠producer is defined relative to the Conductor-spawned producer. (4) **To branch a single-fenced-YAML protocol block (Gate 3 = one ```yaml fence, lines 95-338), use an ADDITIVE SIBLING section + a guard line in the prose header OUTSIDE the fence — NEVER an ELSE-wrap** (wrapping top-level YAML keys under `ELSE:` re-indents every line = byte change, contradicting "augment-not-replace"). Byte-checks on a block that gets a line inserted ABOVE it MUST be offset-aware (`HEAD A,B` vs `current A+N,B+N`) or content-anchored (awk fence-body extraction) — a fixed `sed -n 'A,Bp'` on both sides silently mis-fires once the block shifts (the verification re-review caught this, not mental simulation). (5) **Wiring an observability marker to a role the same change FORBIDS = dead-on-arrival telemetry** — the deliverable `gate3_verdict` marker was templated as "filled by Blake" while the same commit barred Blake from the lane → zero gate_result emission (the "paper machine" failure). When reassigning who-does-X, grep every consumer that names the old role.
- **Action**: For any rubric/quality gate on non-code deliverables: (a) prove it can FAIL via a real PARTIAL→revise→PASS loop, not a clean pass; (b) enforce judge≠producer with separate fresh Agent spawns + paths-only prompts + fresh-judge-per-round; (c) producer = Conductor-side for research/content. For branching a fenced-YAML protocol: additive sibling section + guard line outside the fence, offset-aware/content-anchored byte-checks, and re-grep every role-named consumer when reassigning a writer. Pin a machine-readable `verdict:` line for any downstream gate that greps the score.
- **Grounded in**: EPIC-20260531-nondev-execution-track.md, .tad/evidence/yolo/nondev-execution-track/ (phase1-architecture-contract.md v2.1, phase3-dogfood-report.md, gate reports), .tad/guides/nondev-execution-track.md, .claude/skills/gate/SKILL.md (Gate 3/4 Deliverable Branch sibling sections), .tad/capability-packs/deliverable-rubrics.yaml
- **failure_mode**: Naive default: gate non-code deliverables by their conclusion (BUILD=PASS, KILL=FAIL) and reuse the same judge agent across rounds. Why wrong: a rigorously-argued KILL is the highest-value deliverable — conclusion-based scoring rewards sycophancy over rigor; reusing judges anchors on prior verdicts (~10-15% self-enhancement bias), destroying independence.

### Decouple Detect-from-Heal at Release Gates — 2026-06-01
- **Context**: P3 codex-parity-phase3-releasegate. Backend-architect P0: auto-heal (regen+ship in *publish) silently ships an unreviewed LLM-generated 46KB artifact into a permanent tag → 14 downstream projects. User re-decided: detect-only + separate reviewed regen.
- **Discovery**: A release-time quality gate on derived artifacts (Codex editions, generated docs, compiled bundles) should DECOUPLE detection from remediation. Detection = fast read-only check in the release flow (shell grep/awk, no external tools needed, fail-CLOSED). Remediation = separate human-invoked command that writes, with the result reviewed via `git diff` before commit. This separation means: (1) the gate never fails due to tool unavailability (codex auth expired doesn't block the DETECTION), (2) unreviewed AI-generated content cannot enter a tagged release (the human reviews the diff), (3) the common case (no drift) is silent-pass with zero latency added. The anti-pattern is "self-heal" — auto-regen+auto-commit in the release flow, which makes "hard block on drift" a lie (it silently heals instead of blocking, and ships an unreviewed artifact).
- **Action**: For any release gate on derived/generated artifacts: detect-only in the release flow (read + block, never write); separate human-invoked regen command (write + review + commit). The gate and the regen share the same check script but run at different times with different permissions. Never auto-commit regenerated content in a release flow.
- **Grounded in**: HANDOFF-20260601-codex-parity-phase3-releasegate.md §11 Decision #1, .tad/codex/regen-codex-editions.sh, .claude/skills/alex/SKILL.md publish_protocol step3b
- **failure_mode**: Naive default: auto-heal drift at release time (detect + regenerate + commit in one step). Why wrong: unreviewed LLM-generated content silently enters a tagged release shipped to downstream projects — the "hard block on drift" is a lie because the gate heals instead of blocking.

### Awk Range Pattern Start/End Overlap on macOS — 2026-06-03
- **Context**: AC7 verification for YOLO workflow P3. The awk command `awk '/^yolo_execution_protocol:/,/^[a-z_]+:/' SKILL.md | wc -l` returned 1 instead of the expected ~50. The start pattern `^yolo_execution_protocol:` also matches the end pattern `^[a-z_]+:` because the key starts with lowercase letters.
- **Discovery**: On macOS BWK awk, when a range pattern's start line also matches the end pattern, the range opens and closes on the same line (returns 1 line). This makes the verification command useless. Fix: exclude the start key from the end pattern: `awk '/^yolo_execution_protocol:/{found=1} found && /^[a-z_]+:/ && !/^yolo_execution_protocol:/{print;found=0;next} found{print}'`.
- **Action**: When writing AC verification commands with awk range patterns, ensure the end pattern cannot match the start line. Use explicit exclusion (`&& !/start_pattern/`) or use a state-variable approach instead of comma-range syntax.
- **Grounded in**: HANDOFF-20260603-yolo-workflow-p3.md AC7, COMPLETION-20260603-yolo-workflow-p3.md KA section
- **failure_mode**: Naive default: use awk comma-range syntax (`/start/,/end/`) without checking whether the start line also matches the end pattern. Why wrong: on macOS BWK awk, the range opens and closes on the same line (returns 1 line instead of ~50), making the verification command silently useless.

### Workflow-Internal Stop Gates Between Sub-Agent Steps — 2026-06-03
- **Context**: Safety validation of yolo-epic.workflow.js. Experiment 1 confirmed Codex's finding: no deterministic gate between Y4 (design review) and Y5 (implement). If reviewers found P0s, Blake implementation proceeded anyway.
- **Discovery**: When a workflow orchestrates multiple sub-agent steps sequentially, judgment must be possible BETWEEN steps, not only at workflow-end. The Conductor receiving `design_review_p0_count` in the final result is too late — by then the expensive implement step has already consumed tokens and produced potentially-broken code. This is structurally identical to the SKILL.md constraint "Review must be Conductor-spawned" — the point is that judgment needs to happen at the step boundary, and a workflow without an internal gate delegates that judgment to no one.
- **Action**: For any workflow where step N produces pass/fail findings that should block step N+1: add an explicit `if (failCount > 0) { return result }` between the steps. Don't rely on the Conductor reading the final result to retroactively stop — the expensive step already ran.
- **Grounded in**: .tad/evidence/research/2026-06-03-workflow-safety-validation.md Experiment 1, yolo-epic.workflow.js lines 261-265 (the fix)
- **failure_mode**: Naive default: run all workflow steps sequentially and check pass/fail only at the end. Why wrong: if step N (design review) finds P0s, step N+1 (implement) runs anyway — consuming expensive tokens and producing broken code that must be discarded. The judgment opportunity between steps is lost.

### A Gate on an OPINIONATED Artifact Must Score Rigor Independent of the Opinion — Prove It with a Thin Control + Order-of-Emission Firewall - 2026-06-06
- **Context**: EPIC-nondev-verdict-shapes (3-phase YOLO) added `categorical` (product-thinking BUILD/PIVOT/KILL) + `checklist` (voice/video export-spec) verdict_shapes to the non-dev deliverable Gate 3/4 branches (the prior entry built only the `weighted` shape). The categorical shape had a trap: the artifact being graded ENDS IN AN OPINION (BUILD/PIVOT/KILL), and the naive gate `BUILD→PASS / KILL→FAIL` is semantically WRONG — a rigorously-argued KILL (correctly killing a bad idea) is the HIGHEST-value deliverable and must PASS. Dogfood (PalateBox hot-sauce subscription, 4 distinct agents): rigorous analysis → content_verdict KILL, judge band `rigorous` → **PASS**; thin sycophantic control → content_verdict BUILD, judge band `superficial` → **FAIL**.
- **Discovery**: (1) **When the graded artifact contains a conclusion/opinion, the gate must score the RIGOR of the reasoning, NOT the conclusion** — and the proof is the INVERSE result: a rigorous KILL PASSing while a superficial BUILD FAILs is the exact inverse of the naive conclusion→verdict mapping, which is the only result that demonstrates decoupling actually holds. A clean same-direction PASS proves nothing. (2) **Prose ("score rigor not conclusion") does NOT enforce decoupling — a structural firewall does**: (a) ORDER OF EMISSION — the judge must commit `band:` WITH per-dimension justification BEFORE it writes `content_verdict:`, so the conclusion can't anchor the band; (b) a SWAP TEST in the judge prompt ("if you flipped the final verdict word, would the band change? if yes you're scoring the conclusion"); (c) band criteria phrased ONLY about rigor (evidence depth, fatal-flaw coverage), NEVER "concludes BUILD / is optimistic". Both judges independently reported the swap test changed nothing. (3) **A deliberately-THIN control is a cleaner discrimination instrument than waiting for an honest first-pass PARTIAL** (the prior nondev-execution-track entry used v1-PARTIAL→v2-PASS). A thin control GUARANTEES a non-PASS data point on the same input in one shot, and its contrast with the rigorous artifact isolates exactly what the rubric rewards — without needing a lucky borderline first draft. (4) **A judge rubric claiming "scorable from rubric+artifact paths alone" must EMBED any differentiator it would otherwise fetch from sibling files** — the D5 product-type-adapter dimension was unscorable until a compact per-type differentiator table was embedded IN the rubric (a judge can't open `adapters/*.md`). Self-containment is a testable property: trace each dimension's scoring and ask "does this require a file the judge wasn't given?". (5) **A checklist (pass/fail) verdict_shape needs a ≥1-required malformed_guard** — an all-optional checklist rubric becomes a gate that can never FAIL; and item pass/fail must be judge-derived from artifact substance, never from the artifact's own "passed" claim (artifact-channel rule). Verified PASS+FAIL via a synthetic fixture with zero hardware (P1-4/DR-20260606) when the real consumers (voice/video) are hardware-blocked for dogfood.
- **Action**: For any gate scoring an artifact that ends in a recommendation/opinion: (a) score rigor, decouple from the conclusion via order-of-emission firewall + swap test + conclusion-neutral band criteria; (b) prove decoupling with the INVERSE result (rigorous-negative PASSes, superficial-positive FAILs), not a same-direction pass; (c) use a deliberately-thin control to guarantee a discrimination data point; (d) make judge rubrics self-contained — embed cross-file differentiators; (e) checklist shapes need a ≥1-required guard + substance-derived (not self-claimed) item scoring; (f) when real dogfood is hardware-blocked, a synthetic PASS+FAIL fixture verifies gate logic honestly (label it "gate verified, real-content dogfood pending").
- **Grounded in**: EPIC-20260606-nondev-verdict-shapes.md, .tad/evidence/yolo/nondev-verdict-shapes/EPIC-COMPLETION.md, .tad/evidence/reviews/2026-06-06-rubric-eval-palatebox-{rigorous,thin}.md, .claude/skills/gate/SKILL.md (Verdict_Mapping categorical/checklist), .claude/skills/product-thinking/references/pressure-test-rubric.md, DR-20260606-checklist-shape-dogfood-deferral.md
- **failure_mode**: Naive default: score the artifact's conclusion (BUILD=PASS, KILL=FAIL) and reuse the same judge for revision rounds. Why wrong: this rewards sycophantic conclusions over rigorous reasoning, and a reused judge anchors on its prior score (~10-15% bias). The gate never proves it can FAIL a superficial artifact, so it is unproven theater.

### AC-Driven Universal Gate: §9.1 as Primary Verification Source, with a Dev-Floor Smoke Alarm - 2026-06-07
- **Context**: TAD v3.1 (HANDOFF-20260607-universal-gate-ac-driven). Converted Gate 3 from hardcoded dev checks (tsc/test/lint + a separate `task_type: deliverable` rubric routing branch) to executing the handoff's §9.1 Spec Compliance Checklist row-by-row as the PRIMARY VERIFICATION SOURCE. The 2 deliverable branches (Gate 3 + Gate 4) were removed as routing branches; their judge≠producer SAFETY machinery was migrated byte-exact into a new universal `## Rubric Evaluation Protocol` section that ACTIVATES when a §9.1 AC references a rubric/judge. Goal: one Gate that serves dev (menu-snap) AND non-dev (podcast/content/电商) projects without forcing the latter through a tsc/lint path.
- **Discovery**: (1) **A SAFETY-keyword count-floor AC (grep -cE 'BLOCKING|MANDATORY|VIOLATION' >= baseline) is structurally in tension with a refactor that legitimately DELETES duplicate gate infrastructure** — exactly the principles.md "global count floor cannot detect must-cover loss when legit stripping also lowers the count" failure. Faithful implementation (delete 2 branches = ~21 markers, migrate only the rubric machinery = 8) lands BELOW the 44 floor on count alone. Resolution that is NOT padding: the new AC-driven architecture has GENUINE new blocking surface — the defining new failure mode is paper-accepting a §9.1 row without running its Verification Method — so adding VIOLATION anti-rationalizations to that real surface (plus elevating verdict_shape_guard's already-real BLOCK to an explicit (BLOCKING)/VIOLATION) recovers the count truthfully. The reviewer must explicitly judge each added marker GENUINE vs PADDING; the line-SET (must-cover preserved byte-exact + legit-dedup removed + genuine-new-surface added) is ground truth, the count is the smoke alarm. (2) **Switching a hardcoded check to AC-derived creates TWO holes, not one**: the obvious one (empty §9.1 → must BLOCK, FR7) AND the subtle one (present-but-thin §9.1 that omits the tsc/test rows the producer was supposed to auto-generate → would PASS with tsc never run). The empty guard catches only the first. The second needs a separate `task_type ∈ {code,mixed} ∧ touches buildable files ∧ no compile/test row → WARN` smoke alarm (WARN not BLOCK — a legit pure-config code handoff has no compile row, so hard-block would false-positive). This is the old hardcoded test-runner floor's intent restored as a soft reminder, per the single-user-CLI "smoke alarm not fire suppressor" principle. (3) **Prose-phrase activation of a SAFETY sub-protocol is bypassable; back it with the strong structured signal.** The Rubric Evaluation Protocol activating only when §9.1 text contains "spawn independent judge per Rubric Evaluation Protocol" lets a rubric handoff slip past Judge_Not_Producer if the author omits the phrase. Fix: OR-in a frontmatter backstop (task_type: deliverable OR non-empty rubric_ref forces activation) so the strong signal makes activation fail-safe. (4) **Removing a routing branch means hunting EVERY sibling that routes to it** — the same deny-list-at-every-granularity lesson: `.tad/tasks/handoff-creation.md` (a LIVE file loaded by tad-handoff/SKILL.md) still routed deliverable→deprecated-template after the SKILL itself was updated; only an out-of-§7 architect sweep caught it. (5) **A self-referential grep AC can contain its own false-negative**: AC16 (`grep -c 'deliverable-handoff' == 0`) failed because the implementer's own DEPRECATION note mentioned the filename — the de-routing was correct, the prose tripped the count. And AC14's grep was authored without `-E` so its `|` was literal (matched nothing) — the implementation was right, the verifier was broken.
- **Action**: For a hardcoded→AC-driven gate conversion: (a) expect the SAFETY count-floor to fight the branch deletion — recover the count ONLY via genuine new blocking surface (the new failure mode the conversion introduces), have the reviewer judge GENUINE-vs-PADDING, document the line-set; (b) add BOTH guards — empty (BLOCK) and present-but-thin dev-floor (WARN); (c) back any prose-triggered SAFETY sub-protocol with the strong frontmatter signal as a fail-safe backstop; (d) when deleting a routing branch, grep the whole repo for siblings that route to it (tasks/, templates/, SKILLs), not just the file in §7; (e) when writing a grep AC, keep the artifact's own prose from matching the count, and dry-run the grep exactly as written (`-E` for `|` alternation).
- **Grounded in**: HANDOFF-20260607-universal-gate-ac-driven.md, .claude/skills/gate/SKILL.md (Spec_Compliance_Verification + Spec_Compliance_Empty_Guard + Spec_Compliance_Dev_Floor + Rubric Evaluation Protocol + rubric_ac_trigger frontmatter backstop), .claude/skills/alex/SKILL.md (step1_ac_generation), .tad/evidence/reviews/blake/universal-gate-ac-driven/{cr-review,arch-review,acceptance-verification-report}.md
- **failure_mode**: Naive default: use a global SAFETY-keyword count floor to verify constraint preservation across a refactor that legitimately deletes sections. Why wrong: legit removal and illegit loss both reduce the same number — the floor passes a regen that dropped a must-cover constraint as long as enough other constraints were legitimately removed elsewhere. Also: prose-only activation of a SAFETY sub-protocol lets handoff authors silently skip it by omitting the trigger phrase.

### A Human-in-the-Loop Gate Step Must Verify the Designated Human CAN Perform the Judgment - 2026-07-02
- **Context**: trajectory-eval-p1 Gate 4. The handoff (on data-analyst P0 advice, textbook anti-anchoring methodology) mandated human blind-scoring of 3 trajectories as calibration ground truth. At Gate 4 the user declined: "这些东西是我判断不了的" — scoring TAD process-discipline requires trajectory forensics that the maintainer delegates to agents by design.
- **Discovery**: The expert-review fix imported an unstated premise: a domain-capable human labeler exists. In a single-user TAD project the human is the DECISION-maker, not a domain ORACLE — protocol steps that cast the human as a measurement instrument (labeler, rater, scorer) fail at execution even when methodologically correct. The role-preserving substitution: independent blind agent raters (fresh context, paths-only prompts, forbidden from reading drafts) produce the independent opinion; the human adjudicates only surfaced disagreements — a decision task, which humans always can do. Empirical validation: 2 blind raters overturned 2 of 15 draft scores (GS-07 D1/D4) where the drafter had anchored on outcome — the anchoring the protocol feared was real, and was caught without human rating.
- **Action**: When designing any gate step requiring human input, classify the input: DECISION (approve/choose/adjudicate — always feasible) vs JUDGMENT-AS-DATA (label/score/measure — verify labeler feasibility FIRST, at design time not gate time). For judgment-as-data on single-user projects, default to ≥2 independent blind agent raters + human adjudication of divergences ≥ threshold; record as DEGRADED_WITH_APPROVAL with the same-model-family independence caveat.
- **failure_mode**: Naive default: assign the human the "ground truth labeler" role because calibration methodology requires human labels. Why wrong: the single-user maintainer lacks (by design — they delegate execution) the forensic context to label; the step either blocks the epic at acceptance or gets rubber-stamped — and a rubber-stamped label set is worse than the anchoring it was meant to prevent, because it looks independent while being nothing.
- **Grounded in**: HANDOFF-20260702-trajectory-eval-p1.md §6 Phase C step4 + gate4_delta, .tad/eval/golden-set/INDEX.md Human Confirmation Status, .tad/evidence/acceptance-tests/trajectory-eval-p1/gate4-acceptance-report.md

### Claims Need Carriers: A Completion Claim Without an On-Disk Artifact Is Not a Claim - 2026-06-10
- **Context**: Three same-day, same-shape failures across unrelated protocols: (1) skillify SCANDs marked `status: accepted` with zero materialized skills (Colin, 3 candidates); (2) sep-phase1 Gate 3 "complete" existed only as a chat message — no COMPLETION file, so KA/Friction had no carrier; (3) sep-phase2 claimed "spec-compliance 19/19 (inline)" and "code review P0=0" with no artifact files in evidence/reviews/ (DISTINCT_COUNT=0). In case 3 the layer2-audit smoke alarm itself printed PASS on zero distinct reviewers (fail-open), masking the gap.
- **Discovery**: The recurring rationalization is "the work happened, writing the file is ceremony." But TAD's verification chain (Gate 4 recompute, layer2-audit, *harvest, *evolve-class queries) reads DISK, not chat. A claim whose only carrier is conversation text is invisible to every downstream verifier — structurally identical to validation theater, even when the underlying work was genuinely done. "Inline" execution is fine for DOING; it is never a substitute for PERSISTING.
- **Action**: Every protocol that lets an agent claim completion of X must name X's carrier file and pair it with an existence AC (`test -f` / count), enforced at the consuming gate: acceptance → COMPLETION file; expert review → evidence/reviews/{agent}/{slug}/{reviewer}.md per reviewer; skill acceptance → materialized_at/reference_at path. Smoke-alarm scripts must fail CLOSED on zero carriers (layer2-audit fixed 2026-06-10: DISTINCT_COUNT=0 → FAIL exit 1). When designing a new claim type, ask first: "which file carries this claim, and which gate tests its existence?"
- **failure_mode**: Naive default: accept completion claims from chat conversation text ("the review is done, P0=0") without requiring an on-disk artifact. Why wrong: every downstream verifier (Gate 4 recompute, layer2-audit, *harvest) reads DISK not chat — a claim with no carrier file is invisible to the entire verification chain, structurally identical to validation theater.
