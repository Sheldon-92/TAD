# Pack Evaluation Patterns (Layer 2)

> Reusable patterns for capability pack quality assessment, behavioral evaluation, and cross-model review.

---

### Anti-AI-Slop as Cross-Pack Quality Bar - 2026-04-25
- **Discovery**: Anti-slop targets the default behavior of the agent itself, not expertise. Needs positive framing alongside negative (bold aesthetic direction, not just "don't use Inter/Roboto"). Quality bar moves with training corpus (~6 month review cycle).
- **Action**: Pair anti-slop negative patterns with positive "commit to a direction" criteria. Review every ~6 months.
- **failure_mode**: Naive default: define anti-slop purely as a negative checklist ("don't use Inter/Roboto") without pairing positive direction criteria. Why wrong: negative-only rules leave the agent with no alternative to converge on, producing hesitant/bland output that avoids the banned defaults but commits to nothing — a different flavor of slop.

### Capability Pack Quality Bar: Anti-Slop Metrics - 2026-05-15
- **Context**: Codex and Gemini independently scored 5 new packs on Anti-Slop dimension (does the pack add value beyond frontier LLM training data?).
- **Discovery**: (1) **Highest Anti-Slop scores** (5/5 from both): packs with specific numbers (n=550, exit code 183, 10-32x token cost ratio, ICC>0.92) that an LLM wouldn't produce from training data alone. (2) **Lowest Anti-Slop score** (Codex 3/5 on web-testing): unit testing rules partially restated common knowledge (AAA pattern, co-location, behavior-over-implementation). (3) **The differentiator is economic/statistical thresholds**: Gemini specifically called out "these packs define 'better' using concrete numbers that are easy for an agent to validate" as the top strength. (4) **Anti-slop formula**: specific threshold from research > generic principle from training data. Example: "n≥550 for production A/B decisions" > "use sufficient sample size."
- **Action**: When building future packs, prioritize extracting specific numbers/thresholds/exit codes from research. If a rule could be generated by a frontier LLM without the research notebook, it's low-value — either sharpen with specifics or remove.
- **failure_mode**: Naive default: populate pack rules with generic principles from training data (e.g., "use sufficient sample size") instead of specific research-derived thresholds. Why wrong: generic rules add zero value over what a frontier LLM already knows — the pack's differentiation comes from concrete numbers (n>=550, exit code 183, ICC>0.92) that the agent cannot produce from training data alone.

### Cross-Model Adversarial Review Catches a Defect Class Same-Model Review Misses — But Verify the Reviewer Too - 2026-06-01
- **Context**: 8 agent-adjacent capability packs were built + reviewed by an all-Claude loop (build agent + 2 adversarial Claude reviewers + fixer) → 0 P0, passed discriminative behavioral gate (7/8). Then Codex (codex-cli 0.130.0) adversarially reviewed the same 8 packs and returned 8/8 FIX-FIRST with ~44 concrete findings.
- **Discovery**: (1) **Same-model review has a systematic blind spot for factual/API correctness.** The Claude loop verified PROVENANCE (every number traces to findings.md) and ANTI-SLOP (markers discriminate) but rubber-stamped a whole class of concrete errors: wrong class names (`DeanonymizerEngine`→`DeanonymizeEngine`), deprecated/renamed APIs (Pydantic AI `result_type`→`output_type`, `from rebuff import Rebuff`→`RebuffSdk`), wrong metric types (OTel `gen_ai.client.token.usage` is a Histogram not Counter), wrong constants (STS 600rps not 500, F2 β=2 weights recall 4× not 2×), and an API-reject pattern (`{"role":"system"}` in Anthropic `messages`). Codex's different training surfaced all of these — the documented "independent second view" value, now quantified at ~44 catches the same-model loop missed. (2) **The cross-model reviewer is ALSO fallible — verify its claims against primary sources before applying.** Of Codex's headline P0s, 2 were Codex errors: it claimed GraphRAG Leiden "Level 0 is reversed" (WebSearch confirmed Microsoft GraphRAG Level 0 = root = broadest — the pack was RIGHT) and claimed LangChain Python HITL middleware has only 3 decisions (docs confirm 4: approve/edit/reject/respond — the pack was RIGHT). A fix agent correctly SKIPPED both (catA_skipped); a separate verify agent then FALSE-FAILed those packs by trusting the Codex finding — i.e. **a verifier inherits the reviewer's errors unless it independently checks**. (3) "Verified" on the behavioral gate (pack changes agent behavior) is ORTHOGONAL to factual correctness — a pack can pass the discriminative eval and still carry ~6 API errors.
- **Action**: (a) For any research-grounded artifact (capability pack, doc, rules file), run at least one CROSS-MODEL adversarial review (Codex/Gemini) in addition to same-model review — same-model review will not catch its own factual blind spots. (b) Triage cross-model findings into code/factual (fix) vs over-absolute (judgment) vs unsourced-number (anti-slop) before acting. (c) NEVER blind-apply an adversarial reviewer's P0 — for version-sensitive claims (API names, version numbers, deprecations, metric types) verify against CURRENT primary docs (WebSearch) first; budget for the reviewer being wrong (~2 of N here). (d) A verify stage must independently check, not trust the prior reviewer's verdict, or it propagates the reviewer's errors.
- **Grounded in**: .tad/evidence/pack-eval/2026-06-01/codex-review/ (SYNTHESIS.md + 8 codex reviews), commit 6c79a3d, fix-workflow.js (catA_skipped fields)
- **failure_mode**: Naive default: rely solely on same-model (Claude-only) review for factual/API correctness of research-grounded packs. Why wrong: same-model review has a systematic blind spot for factual errors (wrong class names, deprecated APIs, wrong constants) — it validates provenance and anti-slop but rubber-stamps concrete errors its own training shares, missing ~44 findings that a cross-model reviewer catches.

### Structural-Gold ≠ Depth-Gold; a Single Count Mis-Ranks Both — Use a Two-Layer Bar with Per-Layer Negative Controls - 2026-06-13
- **Context**: Phase 1 of the 24-pack quality-leveling Epic built a two-layer quality bar (Layer A meta-design/structure /10, Layer B domain-depth 0/2/5) + baselined all 24 packs, grounded in Anthropic's official skill-authoring docs + 3 internal gold packs (web-ui-design/web-frontend/web-backend).
- **Discovery**: (1) **A pack can be a depth gold and a structural anti-pattern at the same time.** `web-ui-design` scores Layer B 5/5 (deepest domain content) yet its SKILL.md body is 1202 lines — violating Anthropic's `<500 line` progressive-disclosure rule — so its Layer A drops to 6/10. "Most mature pack" conflated two orthogonal axes; separating structure from depth surfaced a real refactor target that a single composite score hid. (2) **A single counted metric (specN = specific-threshold density) mis-ranks the gold packs**: web-backend specN=27 sits *below* academic-research's 137, because backend depth lives in *operationalized criteria* (tiered checklists, P0/P1/P2 severity, CONSUMES/PRODUCES contracts, production-incident rules) — not raw number density. Fix: anchor the 3 golds at Layer-B 5 *by definition*, use specN only as a counted sub-dimension/tiebreaker for non-gold packs, and disclose that single-metric ranking is advisory + re-scored at each batch entry. (3) **A reference-only structural scan under-credits deep-skill/router architectures**: `product-thinking` looked thin (1 `references/` file) but actually carries 2492 lines across `skills/`+`adapters/`+`checklists/` — the scan must count all content subdirs, or it punishes packs for *not* using the reference-based layout. (4) **Two-sided negative controls are what make the bar non-theater**: a deliberately-bad *structure* sample must FAIL Layer A (scored 0/10) AND a deliberately-shallow *domain* sample must score ≤2 on Layer B (scored 1/5) — proving each layer discriminates, symmetric to the 2026-05-31 discriminative-gate lesson. The bad-structure sample initially self-leaked (prose containing "CONSUMES/PRODUCES" → false grep match, +2 phantom points) — same self-leak class as ac-verification; clean it to reach a true 0/10.
- **Action**: For any cross-pack quality bar: (a) split STRUCTURE from DEPTH into separate scored layers and never let one composite hide a gap in the other; (b) anchor depth golds by definition, treat any single count (specN) as advisory sub-dimension not the driver; (c) scan ALL content subdirs (skills/checklists/adapters/references), not just references/, so deep-skill/router packs aren't under-credited; (d) give EACH layer its own negative control with a real recomputed FAIL verdict; (e) scrub negative-control samples of the marker tokens themselves (self-leak).
- **Grounded in**: .tad/evidence/pack-quality/QUALITY-BAR.md (§1/§2/§4), .tad/evidence/pack-quality/BASELINE-AUDIT.md (§1 dist table + §3 gold structural gap), EPIC-20260613-capability-pack-quality-leveling.md, notebook capability-pack-meta-design (b29b362d)
- **failure_mode**: Naive default: use a single composite score to rank pack quality (e.g., one number combining structure and depth). Why wrong: a pack can be a depth gold and a structural anti-pattern simultaneously — a composite score hides the gap, so a 1202-line SKILL.md violating the <500-line rule gets rated "most mature" because its domain content is deep, masking a real refactor target.

### Behavioral-Eval Gate Must Run on SEPARATE Discriminative Field - 2026-05-31
- **Context**: Gate 3 Layer 2 review of `pack-eval-runner.sh` (P5 follow-up). The runner gated PASS on `min_marker_count` over the fixture's `## Verification Command` grep pattern. That pattern was authored to MIX pack-specific markers WITH generic ones (severity tags `[P0]/[P1]/[P2]`, generic stats `n=20`/`confidence interval`, and nouns any senior practitioner emits). Proof of the flaw: the committed ai-evaluation CONTROL (a review written with NO pack loaded) scored **3/3 PASS** on the combined gate — pure validation theater (it matched only `confidence interval`, `n ≥ 100`, `n=20`).
- **Discovery**: The previous entry got the MARKER-AUTHORING right (named-rule/threshold, not vocabulary) but the GATE was still wired to the contaminated combined count. Having a correct exclusion list in the Anti-Slop Check is necessary but NOT sufficient — those generic markers were still PRESENT in the live grep pattern, so the gate still counted them. The fix is structural: the gate needs its OWN field. Add `discriminative_pattern` (a `grep -oE` alternation of ONLY the pack-specific markers) + `min_discriminative` to frontmatter; the runner PASSes iff `disc = grep -oE "$discriminative_pattern" out | sort -u | wc -l ≥ min_discriminative`, and reports the combined count as a SECONDARY context number only. After the fix the same CONTROL scores **disc 0/3 → FAIL** while WITH scores **disc 5/3 → PASS** — the gate now discriminates. A false PASS is worse than a false FAIL here, so when unsure make the discriminative pattern TIGHTER (fewer, more pack-unique terms). Thin packs justify `min_discriminative: 2`; the all-patterns photo fixture uses 4. Two macOS gotchas surfaced: (1) YAML double-quoted patterns escape `\` as `\\` (e.g. `DESIGN\\.md`) — the parser must collapse `\\`→`\` after stripping quotes, else the regex over-matches; (2) keep the combined-count fallback + WARN for any fixture lacking `discriminative_pattern` so the runner stays backward-compatible and never fail-closed.
- **Action**: For any marker-based behavioral/eval gate: separate the PASS-driving field (discriminative, pack-only markers) from any human-facing combined/context count. Never let the gate count generic markers even if they live in the same fixture. Prove the discrimination with a NEGATIVE control (no-pack output) that must FAIL — a gate that only PASSes the positive case is unproven. Collapse YAML `\\`→`\` when reading regex patterns from double-quoted frontmatter.
- **Grounded in**: .tad/scripts/pack-eval-runner.sh (parse_disc_pattern / assert_one PRIMARY path), .claude/skills/*/examples/*.md (16 fixtures, `discriminative_pattern`), .tad/evidence/pack-eval/2026-05-31/ai-evaluation-{WITH,CONTROL}.md, COMPLETION-20260531-tad-lean-trustworthy-phase5.md (Discriminative-Gate Fix follow-up)
- **failure_mode**: Naive default: gate PASS/FAIL on a single grep pattern that mixes pack-specific markers with generic terms (severity tags, common statistics). Why wrong: a no-pack control output matches the generic terms and passes the gate — pure validation theater. The gate must use a SEPARATE discriminative field containing only pack-unique markers, proven by a negative control that FAILs.

### Named Workflow Resolution Caches Script; Use scriptPath for Iteration — 2026-06-17
- **Context**: Running pack-dogfood functional test for the SkillOpt methodology handoff. Modified `.claude/workflows/pack-dogfood.workflow.js` (added regression stage), then invoked `Workflow({name: "pack-dogfood"})`.
- **Discovery**: The named workflow invocation (`name: "pack-dogfood"`) resolved and COPIED the script at invocation time, but used the PRE-EDIT version (the persisted copy lacked all regression changes). The functional test ran on the old 3-stage pipeline instead of the new 5-stage pipeline. Fix: re-invoked with `scriptPath` pointing to the actual file — this read the current disk version and all 5 stages (including Regression) fired correctly. This is distinct from the documented "args not reliably injected in scriptPath mode" known issue — it's a NAME-resolution caching behavior.
- **Action**: When iterating on a workflow file (edit → test → edit cycle), always use `Workflow({scriptPath: "path/to/file.js"})` rather than `Workflow({name: "workflow-name"})`. The named resolution may cache a stale version. Use named resolution only for stable, finalized workflows.
- **Grounded in**: COMPLETION-20260617-skillopt-tad-methodology.md, pack-dogfood.workflow.js functional test (workflow wf_303801d9 = stale, wf_d9a5be0e = correct)
- **failure_mode**: Naive default: use `Workflow({name:'X'})` to re-test a workflow after editing it. Why wrong: named resolution can cache a stale pre-edit snapshot, so the test runs old code while the developer believes the fix is being exercised — phantom bugs persist and debugging chases the wrong version.
