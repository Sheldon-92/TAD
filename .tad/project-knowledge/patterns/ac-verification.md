### Alex Handoff AC Design Rules - 2026-04-14
- **Discovery**: (1) AC lists become the operational contract — anything not in ACs is effectively optional. Alex MUST explicitly list ALL required evidence files. (2) "≥N triggers" with a concrete list is ambiguous — use "All of A, B, C must PASS" form. (3) When 3 co-mandated ACs are mutually exclusive by construction (byte-preservation + optimization + behavioral invariant), add AC Conflict Matrix sub-step. (4) AC verification commands need pre-ship smoke test — Alex MUST dry-run each on representative existing artifacts before shipping handoff.
- **Action**: Use imperative AC form. Dry-run verification commands. Add AC Conflict Matrix for structural ACs.
- **failure_mode**: Naive default: write ACs as descriptive prose ("system should handle X") without dry-running verification commands on real artifacts. Why wrong: ACs become the operational contract — anything ambiguous or untested becomes an unenforceable requirement that surfaces as a false PASS or false FAIL only during implementation.

### AC Verification Drift Pattern - 2026-04-25 (recurring through 4 phases)
- **Discovery**: Alex specifies AC verification commands without testing them on real artifacts. Failures surface only when Blake runs the literal command. Three sub-patterns: (1) sentinel/marker substring leak, (2) output-shape assumption mismatch (single vs multi-file grep), (3) expert reviewer scope mismatch. Root cause: mental simulation of regex is insufficient. §9.2 "Verified Output" column should be MANDATORY-FILLED by Alex before handoff ships.
- **Action**: Every non-trivial AC verification command MUST be dry-run on a representative existing artifact during handoff drafting. Paste actual command output in §9.2.
- **failure_mode**: Naive default: mentally simulate regex/grep commands instead of running them on real files. Why wrong: mental simulation misses sentinel substring leaks, output-shape mismatches (single vs multi-file grep), and scope errors that only surface when Blake runs the literal command — wasting an entire implementation cycle.

### AC Self-Leak from Removal Rationale - 2026-04-27
- **Discovery**: When a grep-substring AC verifies "no occurrence of X" and the impl adds a rationale comment containing X (e.g., handoff slug containing the forbidden word), the comment self-leaks. Fix: reference META artifacts (deprecation.yaml entry, ADR id) not the handoff slug.
- **Action**: Removal rationale pointers must reference META artifacts, never the removed-feature name verbatim.
- **failure_mode**: Naive default: include the forbidden substring in the rationale comment explaining why it was removed (e.g., "removed feature-X" contains "feature-X"). Why wrong: a grep-based "no occurrence of X" AC catches the rationale comment itself, causing a false FAIL that blocks implementation even though the actual feature code is correctly removed.

### Recurring failure: tsc missing type - 2026-05-19
- **Context**: Detected by dream-scanner from v2 trace analysis. Pattern 'tsc: missing type' appeared in ≥2 reflexion_diagnosis events across TAD handoff cycles.

### Behavioral-Fixture Discrimination: Anti-Slop = Threshold/Named-Rule, Not Vocabulary - 2026-05-31
- **Context**: P5 authored 14 behavioral fixtures (1 per installed capability pack) for `pack-eval-runner.sh`. Each fixture greps a captured agent output for "markers" proving the agent APPLIED a pack rule. The anti-gaming risk: a marker a no-pack frontier LLM would also emit makes the fixture validation theater (architecture.md YOLO audit).
- **Discovery**: The reliable discriminator is the SAME as the anti-slop formula (architecture.md "Capability Pack Quality Bar"): a marker is pack-specific iff it is a NAMED rule or a SPECIFIC threshold/number the pack introduces — NOT a domain noun. Concrete contrast that survived spot-check: markers like "self-enhancement bias 10-15%", "TruffleHog exit 183", "ACX RMS -23 to -18 dB", "n=550 a11y / 30-50%", "Four-Gate fastest-fail-first", "Judge ≠ Optimizer", "60-30-10 + 3-level token". NOT "make it secure", "add more tests", "use a TTS tool", or any word from the input scenario. Two fast sanity gates per fixture: (1) every Verification Command MUST be `grep -oE 'a|b|c' {out} | sort -u | wc -l` — NEVER `grep -c` (P4 lint Rule A: grep -c + sort -u | wc -l always returns 1). (2) every fixture needs ≥1 `[structural]` marker (proves the agent produced the rule's OUTPUT SHAPE, e.g. per-photo first_frame/last_frame decomposition or D1-D10 decision IDs, not just mentioned the vocabulary). The list in the Anti-Slop Check is load-bearing: it documents the generic terms deliberately EXCLUDED from the grep pattern.
- **Action**: When authoring a behavioral fixture, pull markers from the pack's specific numbers/named-rules/exit-codes/thresholds, never domain nouns or input words. Use `grep -oE … | sort -u | wc -l`. Require ≥1 [structural] marker that verifies output SHAPE. Self-check each marker: "would an agent WITHOUT this pack emit this?" — if yes, replace it.
- **Grounded in**: .tad/scripts/pack-eval-runner.sh, .claude/skills/*/examples/*.md (16 fixtures), .tad/templates/pack-example-fixture.md, COMPLETION-20260531-tad-lean-trustworthy-phase5.md AC5.3
- **failure_mode**: Naive default: use domain nouns or generic vocabulary (e.g., "make it secure", "use a TTS tool") as behavioral fixture markers. Why wrong: a frontier LLM without the pack loaded will also emit those terms, making the fixture pass without the pack — pure validation theater that cannot distinguish pack-influenced output from baseline.

### `node --check` on a `.workflow.js` File is a Structural False-FAIL — Use the Wrapped-Body Check - 2026-06-07
- **Context**: HANDOFF-20260607-surplus-scan-phase1 AC1 specified `node --check .claude/workflows/surplus-scan.workflow.js` must exit 0. It exited 1 with `SyntaxError: Illegal return statement`.
- **Discovery**: TAD workflow files use **top-level `return`** statements (the workflow runtime wraps the file body in a function before executing, and injects `agent`/`parallel`/`phase`/`log`/`budget` as locals). Bare `node --check` parses the file as a plain ESM module where top-level `return` is illegal — so it ALWAYS exits 1 for EVERY workflow in the repo. Verified empirically: `loop-discover`, `epic-audit`, `tournament-design`, AND `surplus-scan` all exit 1 under bare `node --check`. An AC of the form "`node --check <workflow>` exit 0" is therefore a structural false-FAIL by construction — it can never pass for a correctly-written workflow. The runtime-faithful syntax check is: strip the `export const meta` ESM line (`sed 's/^export const meta/const meta/'`), wrap the remainder in `async function __wf(){ ... }` with the injected globals declared as `let`, then `node --check` that — exit 0 confirms the body is valid JS including the legal top-level `return`.
- **Action**: Never write an AC that runs bare `node --check` on a `.workflow.js` file. For workflow syntax validation use the wrapped-body check: `{ echo 'async function __wf(){'; echo 'let args,agent,parallel,phase,log,budget;'; sed 's/^export const meta/const meta/' <wf>; echo '}'; } | node --check /dev/stdin`. This is the workflow-file extension of the existing "dry-run verification commands on real artifacts" rule — the AC author must dry-run the check against an existing known-good workflow (loop-discover) and see it behave as expected before shipping the AC.
- **Grounded in**: .claude/workflows/surplus-scan.workflow.js, .claude/workflows/loop-discover.workflow.js (top-level return convention), COMPLETION-20260607-surplus-scan-phase1.md Note A
- **failure_mode**: Naive default: use bare `node --check` to syntax-validate a `.workflow.js` file. Why wrong: workflow files use top-level `return` (legal inside the runtime wrapper but illegal as standalone JS), so `node --check` ALWAYS exits 1 on correct workflow files — a false gate that trains the operator to ignore it, hiding real syntax errors.

### Workflow `agent({schema})` Requires Top-Level `type: 'object'` — a Top-Level Array Schema is Rejected by the StructuredOutput API - 2026-06-08
- **Context**: Gate 3 live run of surplus-scan.workflow.js FAILED instantly — every reader + generator returned `API Error: 400 tools.6.custom.input_schema.type: Input should be 'object'` with 0 tokens. Two call sites passed `schema: { type: 'array', items: CANDIDATE_SCHEMA }`.
- **Discovery**: The `agent(prompt, { schema })` StructuredOutput tool turns `schema` into an Anthropic tool `input_schema`, which the API requires to have TOP-LEVEL `type: 'object'`. A top-level array schema is rejected at request validation — BEFORE the agent runs (0 tokens consumed), so it fails fast and total, not partially. Fix: wrap the array under an object key — `{ type: 'object', properties: { candidates: { type: 'array', items: CANDIDATE_SCHEMA } }, required: ['candidates'] }` — and read results via `result.candidates`. The prompt must also instruct the agent to return `{"candidates": [...]}` not a bare array. Note: verifier `grep -c "type: 'array', items: CANDIDATE_SCHEMA"` is NOT 0 after the fix because the wrapper definition legitimately nests that substring; the load-bearing check is `grep -nE "schema: \{ type: 'array'"` (no top-level array at any CALL site) being empty.
- **Action**: Any workflow `agent(…, {schema})` that wants a list output MUST wrap it as `{ items_key: { type: 'array', items: ITEM } }` with top-level `type: 'object'`; never pass a top-level array schema. Read back via `result.items_key`. **NOTE: epic-audit.workflow.js:80 has the same latent top-level-array bug** (not yet triggered) — fix when next touched.
- **Grounded in**: .claude/workflows/surplus-scan.workflow.js (CANDIDATES_SCHEMA wrapper, scan + generate call sites), .claude/workflows/epic-audit.workflow.js:80 (latent)
- **failure_mode**: Naive default: pass a top-level array schema (`{ type: 'array', items: ... }`) to `agent({schema})` for list outputs. Why wrong: the Anthropic StructuredOutput API requires top-level `type: 'object'` — a top-level array is rejected at request validation (0 tokens consumed, instant API 400), causing total workflow failure before any agent runs.

### Workflow `{name:}` Invocation Can Load a STALE Cached Copy — Use `scriptPath` to Test Fresh Edits - 2026-06-08
- **Context**: Surplus Epic Phase 1 Gate 3. After fixing a schema bug in `.claude/workflows/surplus-scan.workflow.js` and committing, two consecutive `Workflow({name:'surplus-scan'})` runs failed with the IDENTICAL pre-fix error (0 tokens, instant API 400). The committed file was correct; the per-run runtime copy (`surplus-scan-wf_*.js`) still contained the OLD code.
- **Discovery**: `Workflow({name:'X'})` resolved `X` to a cached/stale snapshot, not the freshly-edited source file — so the fix never actually ran. Invoking `Workflow({scriptPath:'<abs path to .claude/workflows/X.workflow.js>'})` force-loads the real file (the tool result's "Script file:" line then points at the actual path, not a `-wf_*` copy). Confirmed by grepping the runtime copy vs the committed file: call sites differed.
- **Action**: When iterating on a workflow and re-testing, invoke via `scriptPath` (absolute path to the committed `.workflow.js`), NOT `{name:}`. Verify the tool result's "Script file:" line points at your real path. Cross-check `grep "schema:" <runtime-copy>` vs the committed file if a fix "didn't take."
- **Compounding lesson (validation theater)**: 4 expert reviews (2 design + 2 impl) all PASSED, yet the first live run still caught 2 real bugs (top-level-array schema rejected by StructuredOutput API; stale-copy). Static review + `grep`/`node --check` cannot see API-level schema rejection or runtime arg propagation. Only a live workflow run is ground truth for a `.workflow.js`. (Reinforces YOLO-audit "validation theater" + honest_partial.)
- **Also surfaced**: workflow `args` (`date`/`output_path`) did not propagate in direct invocation → `dateStamp` defaulted to `'undated'`; robust fix is for the invoking SKILL to OWN the output filename rather than trust the workflow's returned `plan_path`.
- **Grounded in**: .claude/workflows/surplus-scan.workflow.js, EPIC-20260607-surplus-burn-mode.md, this session's Gate 3 (runs wf_fa778ee9 / wf_e1f59ad4 stale, wf_82ab1266 scriptPath PASS)
- **failure_mode**: Naive default: invoke a workflow by name (`Workflow({name:'X'})`) after editing the file, assuming it picks up the latest version. Why wrong: named resolution can cache a stale snapshot — edits never take effect, and debugging chases phantom bugs in the old code while the fix sits unused on disk.

### Shell Case-Glob Backslash and AC Shell-Expansion Consistency - 2026-06-09
- **Context**: Migration manifest schema v1 (HANDOFF-20260609-migration-schema-phase1). Validator snippet used `*'\\'*` to match a single backslash in a bash case statement. AC2a used `for t in '\\'; do grep -cF -- "\`$t\`" file; done` to verify the backslash token appeared in the document.
- **Discovery**: Two independent shell quoting pitfalls compounded: (1) In bash `case`, `*'\\'*` matches a literal TWO-character `\\` sequence, NOT a single backslash. The correct single-backslash case pattern is `*\\*` (unquoted). (2) In `for t in '\\'`, the variable `$t` is literally `\\` (two chars), so the AC grep searches for `` `\\` `` (backtick + double-backslash + backtick). The document must therefore use `\\` between backticks to pass the AC, even though the forbidden CHARACTER is a single `\`. These are complementary: the case pattern must use `*\\*` to match single backslashes in paths, while the AC grep must find `\\` in backticks in the documentation. Neither is wrong individually — the issue is that their quoting behaviors must be independently correct and consistent with each other.
- **Action**: (1) For bash `case` patterns matching a single backslash: use `*\\*` (no quotes around the backslash). (2) For AC grep targets involving backslash: the document must use `\\` between backticks to match the `'\\'` shell-expansion. (3) When writing both a validator snippet AND an AC that verifies the snippet's documentation, dry-run BOTH independently — a correct case pattern and a correct AC grep can still disagree on the document's literal content.
- **Grounded in**: .tad/evidence/designs/migration-manifest-schema-v1.md (validator snippet + Step 2), HANDOFF-20260609-migration-schema-phase1.md AC2a
- **failure_mode**: Naive default: assume shell quoting of backslashes in a `case` pattern and in an AC `grep` command will behave identically. Why wrong: `*'\\'*` matches a literal two-character `\\` (not a single `\`), and `for t in '\\'` expands `$t` to `\\` — so the case pattern and the AC grep silently disagree on what they're matching, causing either false PASSes or false FAILs depending on document content.

### Fail-Safe-Default Requirements Need an Undecidable-Input AC - 2026-06-10
- **Context**: Gate 4 acceptance of the codex-parity step3b gate. Handoff FR2 mandated "when the heuristic cannot decide → agents-newer (STOP), false-positive preferred" for a destructive rsync --delete direction guard. Implementation inverted it: default claude-newer, STOP only on positive agents-newer evidence. Blake's 11 ACs all passed and his Layer 2 review missed it; three independent Gate 4 verifiers (AC adversarial probe, code-reviewer, security-auditor) converged on the same root cause, security reproducing latent data loss (spaced filename → parse fail → fail-open to the destructive direction).
- **Discovery**: ACs that only exercise DECIDABLE inputs (clean drift on one side, orphan, clean tree) cannot distinguish "defaults to STOP, promotes on proof" from "defaults to proceed, demotes on evidence" — both pass identical decidable-case tests. The fail-safe default is only observable under undecidable/degraded input (parse failure, non-git tree, mixed commit, deleted-side file). A spec line "when unsure → X" is untested unless some AC constructs the "unsure" state.
- **Action**: For any requirement of the form "when undecidable/unknown → safe default", add an AC that constructs an undecidable input and asserts the safe outcome. Cheap deterministic instance for git-heuristic tools: copy the trees to a NON-GIT temp dir, introduce drift, assert the destructive path REFUSES (heuristic blind → must STOP). Also: a "documented-limitation comment" is not a fix for a fail-open — reviewers should flag comment-only fixes on load-bearing safety paths.
- **Amendment (2026-06-10, Gate 4 rounds 2-3 of the same task)**: One representative undecidable-input AC is NOT enough — Blake's AC12 (non-git tmpdir) passed while the spaced-filename undecidable case still destroyed data, because the fix inverted the DIRECTION seed but left the promotion gate (`all_claude_newer=true` + skip-on-parse-failure) optimistic. Two refinements: (a) enumerate ALL input classes where the heuristic goes blind (non-git, parse failure, mixed commit, deleted-side) and write one AC per class; (b) when reviewing a fail-safe fix, check EVERY variable in the decision chain — a fix that flips the seed but not the exit condition migrates the fail-open instead of closing it.
- **failure_mode**: Naive default: test fail-safe defaults only with decidable/clean inputs (e.g., clear drift, clean orphan). Why wrong: both "default to STOP" and "default to proceed" produce identical outputs on decidable inputs — the fail-safe is only observable under undecidable/degraded input (parse failure, non-git tree), so decidable-only ACs cannot distinguish a safe implementation from a dangerous one.

### A "Real-Browser E2E Passed" Claim Is Confounded If the Test Path Doesn't Enforce the Property Under Test - 2026-06-14
- **Context**: Gate 4 acceptance of AI-Native Reading Companion Phase 3 (localhost co-read bridge). Blake's completion report claimed a REAL Chrome end-to-end (send→reply renders; XSS inert) and 15/15 ACs PASS. The independent Gate 3 review (security-auditor + code-reviewer) then found **P0**: the strict CSP (`default-src 'self'`, no `script-src`/nonce) would make an *enforcing* browser refuse the reader's OWN inline `<script>`/`<style>` → the chat panel/SSE never run → feature dead. The earlier "worked in Chrome" was real but ran on a **non-enforcing-CSP path** (the `file://` / served-without-strict-CSP route), so it never exercised the very property (CSP enforcement) that was broken.
- **Discovery**: An e2e/behavioral claim is only as strong as whether the test ENVIRONMENT actually enforces the property being asserted. "Rendered fine in a browser" silently passes when the browser path doesn't apply the production CSP — a false green that looks like the strongest possible evidence (a live browser). This is the verification-integrity sibling of "禁止纸面验收": the test wasn't paper, but the environment confounded it. Independent adversarial review on the *enforcing* path is what caught it; a single self-reported happy-path browser run did not. The fix (per-response CSP nonce injected into the in-memory HTML, on-disk file nonce-free) also has to be verified on the enforcing path — and here even the fix's final browser-nav step was denied, so it fell back to a nonce-match assertion, leaving the real-browser *visual* confirmation genuinely UNVERIFIED and explicitly carried forward to first real use.
- **Action**: (1) When an AC asserts a behavioral/e2e property, name the ENVIRONMENT that enforces it and verify on that path — for CSP/security headers, a browser that actually enforces the header, not a `file://` or stripped-header route. (2) Treat "worked in a browser" as INSUFFICIENT for a property the browser may not be enforcing on that path; require the enforcing-path run or mark the AC UNVERIFIED honestly. (3) When a self-reported real-browser e2e exists, still run independent adversarial review — it caught a feature-dead P0 that the live-browser claim missed. (4) If the final enforcing-path step can't be run (nav denied), say so and carry it forward; a nonce-match/structural fallback is not the same evidence as a rendered page.
- **Grounded in**: .tad/evidence/yolo/ai-native-reading-companion/phase3-completion.md (Gate 3 Fix Round P0#1), .claude/skills/reading-companion/tools/bridge-server.py (_inject_nonce + per-response CSP), EPIC-20260613-ai-native-reading-companion.md Phase 3 detail block
- **failure_mode**: Naive default: trust a self-reported "worked in a browser" e2e claim as proof that a security property (CSP, CORS, etc.) is enforced. Why wrong: the browser path may not enforce the property under test (e.g., `file://` ignores CSP), producing a false green that looks like the strongest possible evidence while the feature is actually dead under production enforcement.

### A Pre-Declared Exclusion Must Be Restated in Every Gate Metric Definition That Inherits It - 2026-07-02
- **Context**: trajectory-eval-p2 calibration. Phase 1 pre-declared "D4 is data-poor — excluded from 止损判定". The §4.4 metrics contract encoded the exclusion explicitly in the PRIMARY gate (within-1 over D1,D2,D3,D5) but the contrast-pair gate's mean definition said only "与配对规则一致" (consistent with pairing rules). At Gate 3 this produced a verdict-flipping ambiguity: contrast = 1.75 PASS (D4-excluded) vs 1.40 FAIL (D4-included). Blake resolved via the pre-declaration and self-reported the ambiguity; Gate 4 verified no metric-shopping (round 1 FAILed under both readings) and accepted.
- **Discovery**: A pre-declared exclusion propagates reliably only where it is RESTATED INLINE. A downstream metric that references another metric's pairing rules inherits the mechanics (numeric-vs-numeric pairing) but NOT the scoped exclusions, because the exclusion was syntactically attached to the primary metric's dim list, not to the pairing rule itself. When the ambiguous metric is a gate, the ambiguity is verdict-flipping by construction. Two saving graces made this recoverable rather than corrupting: (a) the exclusion was PRE-declared before any data existed (so applying it is not post-hoc fitting), (b) the implementer surfaced the ambiguity instead of silently picking the passing reading.
- **Action**: When writing a metrics contract with N gate metrics: restate every exclusion (advisory dims, data-poor dims, UNRECOVERABLE handling) inline in EACH gate metric's definition — never rely on "consistent with the other metric's rules". At review time, test each gate metric definition by asking "compute this with and without each pre-declared exclusion — do any two readings flip the verdict?" If yes and the text doesn't force one reading, the contract is incomplete.
- **failure_mode**: Naive default: define exclusions once (in the primary metric or a preamble) and let sibling metric definitions inherit by reference. Why wrong: the reference inherits pairing mechanics but not scoped exclusions; the implementer must choose a reading at runtime, and under gate pressure the passing reading is the tempting one — a verdict-flipping ambiguity that looks like a spec detail until it decides PASS vs FAIL.
- **Grounded in**: HANDOFF-20260702-trajectory-eval-p2.md §4.4 + gate4_delta, COMPLETION-20260702-trajectory-eval-p2.md KA #2 (independent convergence), .tad/evidence/acceptance-tests/trajectory-eval-p2/gate4-acceptance-report.md (双口径重算)

### A `node --check` Syntax-Gate Is a FALSE GATE on Workflow Files (Top-Level return+await); Wrap the Body in an async Function First - 2026-06-14
- **Context**: Fixing 3 top-level-array StructuredOutput schemas across `epic-audit`/`loop-discover` workflows (`.claude/workflows/*.workflow.js`). The handoff's AC4 specified `node --input-type=module --check < file` (and naive `node --check`) to syntax-verify the edits. A Gate-2 reviewer ran it empirically: it **fails on the UNMODIFIED files** with `SyntaxError: Illegal return statement` — exit 1 for clean and broken alike. A gate that red-lights correct code is worse than no gate (the operator learns to ignore it; a real syntax error hides in the noise).
- **Discovery**: `.workflow.js` files are NOT standalone ES modules. The Workflow runtime evaluates each file body **inside an async function**, so they legally use module-level top-level `return` AND top-level `await` AND start with `export const meta`. `node --check` / `--input-type=module` treat the file as a top-level script/module where top-level `return` is illegal → false FAIL. (The handoff author guessed the cause was "ESM + top-level await" but the load-bearing illegality is top-level `return`.) A syntax gate must reproduce the REAL evaluation context. Verified working command: prepend `async function __wf__(agent,log,args,parallel,phase,pipeline,budget,workflow){`, strip the line-1 `export ` (`sed '1s/^export const /const /'`), append `}`, pipe to `node --check --input-type=commonjs`. This PASSES all clean workflow files and CATCHES an injected mid-file syntax error (line number reported). A `.js`-extension temp-file variant works too (a temp without `.js` throws `ERR_UNKNOWN_FILE_EXTENSION`).
- **Action**: (1) Before trusting ANY verifier in an AC, run it against (a) a known-GOOD input — it must pass, and (b) a known-BAD input — it must fail. A verifier that fails clean inputs is a false gate; fix it before the AC ships. (2) To syntax-check TAD `.workflow.js` files, use the async-wrap command above — never bare `node --check`/`--input-type=module`. (3) When you fix a bug CLASS (here: top-level-array schema), grep the whole `.claude/workflows/` tree for siblings — this sweep turned 1 reported site (epic-audit:80) into 3 (loop-discover:80 + :111, the latter the core discovery loop = whole workflow broken).
- **Grounded in**: HANDOFF-20260614-fix-workflow-toplevel-array-schemas.md Gate-2 Audit Trail (AC4 false-gate) + §9.1 AC4 verified command; surplus-scan.workflow.js:73-76 (the documented top-level-array contract)
- **failure_mode**: Naive default: trust a verifier command in an AC without first running it against both a known-GOOD input (must pass) and a known-BAD input (must fail). Why wrong: a verifier that fails on correct input (like `node --check` on workflow files) is a false gate — the operator learns to ignore it, so when a real syntax error occurs, the gate has already been dismissed as noise.
