> Reusable patterns for knowledge management, trace emission, staleness detection, and compact recovery.

### Drift-Check and Staleness Detection - 2026-04-24
- **Discovery**: (1) Project-level paths (.tad/project-knowledge/*, NEXT.md, configs) are intentionally cross-handoff — allowlist them in drift detectors. (2) Staleness detection needs a `Revalidated: YYYY-MM-DD` quieting path to defeat alarm fatigue. (3) BSD `date -j` needs explicit time component.
- **Action**: Maintain allowlists for shared files. Design quieting paths for smoke-alarm tools from day one.
- **failure_mode**: Naive default: treat all file modifications as drift violations without allowlisting intentionally-shared cross-handoff paths, and provide no quieting mechanism for staleness alerts. Why wrong: false positives on shared files (project-knowledge, configs) cause alarm fatigue, training the operator to ignore the drift detector — so real drift goes unnoticed.

### Two-Layer Compact Recovery Pattern - 2026-04-28
- **Discovery**: Layer 1 (trigger): self-check rule in CLAUDE.md fires every reply. Layer 2 (persistent state): `.tad/active/session-state.md` on disk. Stale detection via Status field + file existence. Hook writes metadata; SKILL writes semantics. sed delimiter `#` not `|` for path safety.
- **Action**: Anchor trigger in system-prompt content. Store state on-disk. Add stale detection. Separate hook-managed from agent-managed fields.
- **failure_mode**: Naive default: rely on a single recovery mechanism (either a self-check rule OR an on-disk state file, but not both). Why wrong: a single layer fails silently — a self-check rule alone has no persistent state to recover from, and an on-disk file alone has no trigger to read it. Without stale detection, the agent resumes from an outdated checkpoint and produces inconsistent work.

### Observational > Imperative Trace Emission; Stable Marker Contract for Consumed Artifacts - 2026-05-30
- **Context**: trace-instrumentation-fix. The Auto-Evolve Epic built a trace analysis engine + 6 emission helpers, but only 1 imperative call site was ever wired (Blake reflexion) — and it fired exactly once across 328 events. Real-environment gate/expert/decision events: 0. The self-evolution data layer was a "paper machine".
- **Discovery**: (1) **Imperative emission (agent remembers to call a helper) is structurally unreliable** — proven by the 1/328 fire rate. The fix is OBSERVATIONAL: a PostToolUse hook (post-write-sync.sh) parses agent-written artifacts (HANDOFF §11 Decision tables, COMPLETION frontmatter markers, review files) into trace events. The agent's only job is to write the artifact it would write anyway. (2) **For signals the hook cannot reliably infer from prose, the agent writes a STABLE MACHINE-READABLE MARKER, not prose** — gate verdict = `gate3_verdict: pass` frontmatter (allowlisted pass|fail|partial), NOT parsed from the COMPLETION narrative (free-form AND collides with the template's `PASS / PARTIAL / FAIL` menu line). The `.router.log` 5-tuple lesson applied: a consumed artifact needs a contract, not a guess. (3) **Timing contract for verdict markers**: `*complete` writes COMPLETION BEFORE `/gate 3` runs, so the verdict cannot exist at creation. Blake writes it as a Gate 3 POST-STEP Edit; that Edit re-triggers the hook, which now finds the marker. Dedup per (slug, type, day); gate_result re-emits on verdict CHANGE. (4) **JSON-object contexts MUST use detail_level=full** — `record_trace` truncates summary context to 200 chars; a JSON-object context (decision_point) gets sliced mid-string → `fromjson` fails downstream. Set `TRACE_DETAIL=full` (2048 budget) for any helper whose context is a JSON object and whose outcome doesn't auto-escalate.
- **Action**: For decision-level telemetry: prefer observational parsing of artifacts the agent already writes; reserve agent-written content to stable allowlisted markers; never parse free-form prose for load-bearing signals; place verdict-marker writes AFTER the verdict exists; use full detail for JSON-object contexts; hooks NEVER fail-closed (`|| true` on every parse path, BSD-safe regex, compact no-space `"type":"x"` grep).
- **Grounded in**: .tad/hooks/post-write-sync.sh, .tad/hooks/lib/trace-writer.sh, .tad/evidence/reviews/blake/trace-instrumentation-fix/backend-architect.md (P1)
- **failure_mode**: Naive default: instrument trace emission imperatively (require the agent to call a helper function at the right moment). Why wrong: imperative emission is structurally unreliable — proven by a 1/328 fire rate in production. The agent forgets or the call site is never wired, producing a "paper machine" with near-zero real data.

### A Parser Feeding a Human-Review Queue Must Propagate VALUE Fields, Not Just the Key - 2026-05-31
- **Context**: Auto-dream produced 6 `human_override` candidates (slug `trace-instrumentation-fix`) that were content-free shells: Discovery = "Human explicitly overrode agent suggestion for 'X'", Action = generic boilerplate. dream-scanner Pass C extracted only `.context.decision` (the label) and hardcoded the rest — yet the trace events carried fully populated `.chosen` and `.rationale`. All 6 rejected on review; fixed in bugfix-dream-scanner-override-content (Pass C now weaves chosen+rationale into Discovery/Action with a fallback to boilerplate when absent).
- **Discovery**: A parser that consumes a structured artifact to feed a human-review queue can capture the KEY/label while silently discarding the VALUE fields, emitting items that LOOK like knowledge but carry zero signal. This is the inverse of "Parser Self-Trigger" (2026-05-30, false-POSITIVE inflation) — here it is false-EMPTY (content-loss). Critically it is NOT an emission-side gap: the producer (post-write-sync.sh) captured the rationale correctly; the consumer threw it away. The empty candidates then duplicated (worse) an existing architecture.md entry, wasting reviewer time — a scanner-side instance of the validation-theater pattern this very Epic was built to fight.
- **Action**: When a parser/scanner emits candidates for human review, verify it propagates the VALUE/rationale fields, not just the label. Acceptance for such tools MUST include "regenerate one item and confirm the body contains real source content," never just "N items produced" (count ≠ signal). When triaging low-value auto-generated candidates, first check whether the source EVENT already holds richer fields the parser dropped — fix the parser, don't just reject the output.
- **Grounded in**: .tad/hooks/lib/dream-scanner.sh Pass C, HANDOFF/COMPLETION-20260531-bugfix-dream-scanner-override-content.md
- **failure_mode**: Naive default: extract only the key/label from a structured trace event and hardcode boilerplate for the body when generating review candidates. Why wrong: the source event carries fully populated value fields (chosen, rationale) that the parser silently discards — producing candidates that LOOK like knowledge but carry zero signal, wasting reviewer time on content-free shells.

### Batch Migration failure_mode Inferability: ~70% Text-Inferable, ~30% Need Doer Context — 2026-06-22
- **Discovery**: When batch-migrating 110 legacy knowledge entries to add a required `failure_mode` field, ~70% of entries contained enough information in their existing Discovery/Action text to infer the naive default. ~30% only said "do X" without explaining what someone would do WITHOUT this knowledge — the original author omitted it (curse of knowledge at write-time). The distillation loop's gap-handback mechanism exists precisely for this 30%.
- **Action**: When migrating legacy entries to a schema with required `failure_mode`: budget for ~30% needing gap-handback or author consultation. Do not force-fill vague placeholders to make lint pass — an honestly marked gap ([UNRESOLVABLE]) is safer than a fabricated failure_mode that future readers will trust as authoritative.
- **failure_mode**: Naive default: assume 100% of entries can have failure_mode inferred from existing text, and when stuck, fabricate a vague one ("not doing this causes problems") to satisfy the schema/lint. Why wrong: a fabricated failure_mode is worse than a marked gap — it looks complete (lint passes, schema satisfied) but contains invented content the original author never validated. Future readers trust it as authoritative and defend against the wrong naive default. UNRESOLVABLE honestly marks the gap for someone with execution context to fill later.
- **Grounded in**: EPIC-20260622-knowledge-recording-redesign P4 migration (110 entries, 0 UNRESOLVABLE achieved but 30% required significant inference effort); distillation loop e2e validation.
