# Shell Portability Patterns (Layer 2)

> Reusable patterns for shell scripting, macOS/BSD compatibility, and CLI tool usage.

---

### Hook Shell Portability Rules - 2026-04-03
- **Discovery**: (1) No `grep -P` on macOS — use `grep -o` + `sed`. (2) Glob prefix `*.tad/` not `*/.tad/` for relative paths. (3) Single awk process beats N×grep loop (7-9x faster due to fork/exec overhead). (4) Use `ENVIRON["VAR"]` not `awk -v` for user content. (5) `perl -MTime::HiRes=time` for per-step timing, NOT python3 (~130ms startup). (6) bash `$()` strips `\x00` — use `\x1E` for separators. (7) `jq @tsv` escapes content tabs — use `join("")` instead. Supersedes: 4 separate shell portability entries.
- **Action**: Follow these rules for all TAD hook scripts. Test both relative and absolute paths.

### Shell Pattern: Word-Boundary Matching for Slugs - 2026-04-24
- **Discovery**: `\b` is wrong for identifier-style slugs containing `-`. BSD grep treats `-` as word boundary. Use `(^|[^A-Za-z0-9_-])PATTERN([^A-Za-z0-9_-]|$)` bracket class instead.
- **Action**: For shell-level slug matching, use bracket class not `\b`.

### Shell Dispatcher Patterns - 2026-05-09
- **Discovery**: (1) `set -e` propagates non-zero exits through `case` arms. (2) Portable timeout: `gtimeout` → `timeout` → no-op fallback. (3) Set-difference for newly-added item ID (`comm -13`), not positional index. (4) UTM normalization: per-param split by `&`, not bulk regex. (5) Fast-fail phases before slow-fail in fallback chains (1-2s API check before 5-10s yt-dlp). (6) Phase-specific `method:` field in frontmatter is zero-cost audit trail.
- **Action**: Single awk process for classification. Per-param split for URL query manipulation. Fast-fail first.

### mikefarah yq -i Normalizes Whole File on First Write - 2026-05-31
- **Discovery**: mikefarah yq v4 reformats the WHOLE file on its first `-i` write (strips blank lines, normalizes comment spacing, re-folds multiline strings), then is byte-stable on subsequent edits. No reliable preservation flag exists. A "byte-identical untouched entries" AC is only satisfiable relative to an already-yq-normalized file.
- **Action**: When mandating `yq -i` for recurring edits AND requiring byte-identity of untouched entries: (1) expect a one-time normalization, (2) trigger it via an early mandated edit, (3) prove idempotency with normalize→snapshot→edit→diff, (4) run byte-identity checks against the NORMALIZED file. Never reach for line-based sed to dodge the reformat.
- Graduated from: incidents/2026-05/yq-normalizes-once-idempotent.md
- **Grounded in**: .tad/hooks/lib/notebook-lifecycle.sh

### Shell Env-Var Convention for Backward-Compatible Function Extension - 2026-05-19
- **Context**: Extending `record_trace()` from 3 positional params to 10+ fields for v2 trace schema. Expert review (code-reviewer + backend-architect) independently flagged positional params as P0: "10-positional-param record_trace() is maintenance hazard — silent data corruption when callers swap arg order."
- **Discovery**: Env-var convention (`TRACE_*` variables set before call, function reads with `${TRACE_VAR:-default}`, `unset` after call) is the correct pattern for extending shell functions beyond ~3 positional args. Advantages: (1) Backward compat — existing callers unchanged, (2) Self-documenting at call site (`TRACE_OUTCOME="fail"` vs positional arg 7), (3) No silent corruption from arg order mistakes, (4) Inline assignment (`VAR=val command`) scopes to single call without export. The `unset` after call prevents bleed between sequential calls in the same script.
- **Action**: For shell functions with >3 params, use env-var convention. Reserve positional args for the 2-3 most common required params only.

### Double-Parse Pattern for String-Encoded JSON Fields - 2026-05-20
- **Context**: dream-scanner.sh needs to extract sub-fields from v2 trace events where `context` is a JSON-encoded string inside JSONL (e.g., `"context": "{\"what_failed\":\"tsc: missing type\"}""`). Three bugs discovered during implementation.
- **Discovery**: (1) **Single-pass jq is mandatory**: `jq '.context | fromjson | .what_failed'` works in one invocation. Two-step extraction (`jq -r '.context'` → shell variable → `echo "$var" | jq 'fromjson | .field'`) fails because `jq -r` outputs raw text that a second `jq` receives as a JSON object (not a string), so `fromjson` errors. (2) **jq string interpolation**: `\(expr)` is required inside `"\(...)"` — bare `((expr))` without the backslash-paren wrapper silently produces no output. (3) **Bash `read` with `IFS=$'\t'` collapses consecutive empty fields**: input `a\t\tb` splits as `field1=a field2=b` (not `field1=a field2="" field3=b`). Fix: use file-based intermediary (jq -c → temp file → while read per JSON object) instead of tab-delimited output for multi-field extraction.
- **Action**: For string-encoded JSON: always use single-pass jq with `fromjson`. For multi-field extraction from jq: use compact JSON objects per line, not tab-delimited fields.

### Bash heredoc Python injection via unvalidated CLI args - 2026-05-28
- **Context**: Phase 4 academic-search.sh script. Code-reviewer P0: `--limit` parameter assigned from user input (`LIMIT="$2"`) without integer validation, then interpolated into Python heredocs via `${LIMIT}` inside `python3 -c "... [:${LIMIT}]"`.
- **Discovery**: When bash `${VAR}` is expanded inside a heredoc containing embedded Python/awk/perl code, the variable content becomes part of the target language's source code. A malicious `--limit` value like `5]; import os; os.system("id"); x=[0` would execute arbitrary Python. This applies to any bash script that interpolates user input into embedded language blocks.
- **Action**: Always validate CLI numeric arguments with `[[ "$var" =~ ^[0-9]+$ ]]` before interpolating into any embedded language (Python/awk/perl). For string arguments, prefer passing via environment variable (`LIMIT=$var python3 -c "import os; limit=int(os.environ['LIMIT'])"`) instead of heredoc interpolation.
- **Grounded in**: .tad/evidence/reviews/blake/academic-research-pack-phase4/code-review.md (P0-1)

### Heredoc injection depends on the SINK: file-write ≠ interpreter-exec - 2026-05-31
- **Context**: bugfix-dream-scanner-override-content. code-reviewer raised a P0: routing human-authored `.chosen`/`.rationale` through the unquoted heredoc `cat > "$file" <<CAND_EOF ... ${discovery} ... CAND_EOF` enables command injection (a rationale containing `` `id` `` or `$(date)` would execute). Empirical dry-run **refuted** it: `$(date)`/`` `id` ``/`$HOME` all emitted verbatim, nothing ran. The reviewer withdrew the P0.
- **Discovery**: Two heredoc cases look identical but have opposite risk. (1) **Embedded-interpreter heredoc** (`python3 -c <<EOF ... ${VAR} ... EOF`, see "Bash heredoc Python injection" entry 2026-05-28) — the expanded value becomes SOURCE CODE the interpreter then EXECUTES → genuine injection. (2) **File-write heredoc** (`cat > f <<EOF ... ${VAR} ... EOF`) — the value is inserted into the output STREAM as data and **never re-scanned**. Shell expansion is NOT recursive: the unquoted heredoc expands its own literal `${VAR}` token exactly once; `$(...)`/backticks/`$OTHER` arriving INSIDE that value are not re-evaluated, and a literal delimiter line inside the value does not terminate the heredoc (terminator matches input lines only). The real (minor) residual for a file-write heredoc is NEWLINES — `jq -r` renders JSON `\n` as a real newline → multi-line body; flatten with `gsub("\n";" ")` inside jq.
- **Action**: When reviewing/triaging a heredoc "injection" finding, FIRST ask "does the sink EXECUTE the output (interpreter -c, eval, source) or just WRITE it (cat > file, tee)?" Only the executing sink is injection. Demand an empirical reproduction string before accepting a P0 whose fix expands scope — mental simulation of shell expansion is insufficient (the reviewer itself flagged this as the documented failure mode). For file-write heredocs the correct hardening is newline-flatten, not a quoted-delimiter rewrite.
- **Grounded in**: .tad/evidence/reviews/blake/bugfix-dream-scanner-override-content/code-reviewer.md (P0 raised → empirically refuted → withdrawn), .tad/hooks/lib/dream-scanner.sh Pass C

### comm -12 Set-Intersection CJK Needs LC_ALL=C on BOTH sorts AND comm - 2026-05-31
- **Context**: pack-collision detector (scan-collisions.sh) keyword PRE-FILTER intersected two packs' keyword sets via `comm -12 <(... | sort -u) <(... | sort -u)` to decide which pack PAIRS to scan. Code-reviewer P1-1: under the ambient `en_US.UTF-8` locale this produced a PHANTOM shared keyword (`设计`) for `web-ui-design × video-creation`, two packs that share ZERO actual keywords — admitting an un-pre-filtered pair that became the only false-positive candidate.
- **Discovery**: `comm` requires its two inputs to be sorted in the SAME collation it uses internally — and `comm` collates by BYTE (C locale). But `sort` without `LC_ALL=C` uses the ambient locale's collation, which orders CJK/multibyte strings differently than byte order. The mismatch makes `comm` mis-pair lines and report a spurious intersection (or miss a real one). This is the same locale-collation class as the project's BSD-portability lessons. The fix is to force byte collation on EVERY participant: `LC_ALL=C comm -12 <(... | LC_ALL=C sort -u) <(... | LC_ALL=C sort -u)` — the comm AND both sorts, not just one. With `LC_ALL=C` the `web-ui-design × video-creation` intersection is correctly empty and the pair is no longer scanned. (Verified: candidate set dropped the video-creation false positive; the 3 real collisions remained.) A downstream consequence worth noting: a `grep -c '设计' <other-set>` = 0 sanity check is the cheap way to PROVE a phantom intersection — if comm claims a shared token that grep can't find in one input, the collation is the culprit.
- **Action**: Any `comm` over text that MAY contain non-ASCII (CJK keywords, accented strings, emoji) MUST set `LC_ALL=C` on the comm and on both feeding `sort`s. When a set-intersection pre-filter admits a pair that "shouldn't share anything", first suspect locale collation, not the matching logic — verify with `grep -c '<claimed-token>'` against the other set (0 hits ⇒ phantom intersection ⇒ add `LC_ALL=C`).
- **Grounded in**: .tad/scripts/scan-collisions.sh (shared-keyword pre-filter), .tad/evidence/yolo/pack-collision-detection/phase1-impl-review-cr.md (P1-1), COMPLETION-20260531-pack-collision-detection-phase1.md

### Copy-After-Deprecation Ordering for Platform Root Files - 2026-06-07
- **Context**: npx cross-platform installer adds AGENTS.md to codex's `extra_root_files`. tad.sh `copy_framework_files` copies root files from source. But `apply_deprecations` (v2.3.0 entry) deletes AGENTS.md as part of "multi-platform cleanup." If root-file copy runs BEFORE deprecation, the freshly-copied AGENTS.md is immediately deleted.
- **Discovery**: When a file copy and a deprecation cleanup both touch the same path, ordering is load-bearing. Deprecation = "clean up old state"; platform-specific root-file copy = "install new state." New state must be written AFTER old-state cleanup, or it's silently destroyed. This generalizes: any install step that (re-)introduces a file previously marked deprecated must run AFTER `apply_deprecations`.
- **Action**: Place platform-specific root-file copies AFTER apply_deprecations in copy_framework_files. When adding a file to a platform's `extra_root_files` that was previously deprecated (check deprecation.yaml), verify the copy-after-deprecation ordering is maintained.
- **Grounded in**: tad.sh copy_framework_files, .tad/deprecation.yaml v2.3.0 (AGENTS.md removal), COMPLETION-20260607-npx-platform-installer.md

### YAML Frontmatter description Fields Must Quote Colons - 2026-06-08
- **Context**: Codex CLI `$` skill mechanism parses YAML frontmatter of `.agents/skills/*/SKILL.md` on activation. Two packs (ai-agent-architecture, web-ui-design) had unquoted `description` fields containing colons (e.g., `Two modes: /design`). Claude Code's skill loader is lenient and ignores parse errors; Codex's loader fails on them.
- **Discovery**: YAML `description: value with colons: inside` is ambiguous — the parser may interpret the second colon as a nested key. Claude Code never surfaced this because its skill loader either doesn't parse `description` or tolerates errors. Codex's `$` mechanism parses frontmatter strictly and reports errors on activation. The fix is trivial (wrap in double quotes) but the root cause is that SKILL.md frontmatter was never validated against a strict YAML parser.
- **Action**: All SKILL.md `description` fields containing colons, parentheses, or other YAML-special characters must be double-quoted. When creating new capability packs, validate frontmatter with `python3 -c "import yaml; yaml.safe_load(open('SKILL.md').read().split('---')[1])"` or equivalent.
- **Grounded in**: .tad/evidence/dogfood/codex-unification-dogfood.md Finding 2, .claude/skills/ai-agent-architecture/SKILL.md, .claude/skills/web-ui-design/SKILL.md

### Command Substitution Swallows Gate Markers - 2026-06-09
- **Context**: Phase 4 runtime-freshness gate used `age=$(days_between ...)`. The helper printed `GATE: runtime-freshness exit=2` and exited inside the command substitution when semantic date parsing failed (`2026-13-45`).
- **Discovery**: `exit` inside `$(...)` terminates the subshell, not the parent function. With `set -e`, the parent may still exit with the right code, but stdout from the helper is captured into the assignment instead of reaching the operator or machine parser. For release gates, this can preserve the exit code while silently losing the required `GATE:` marker.
- **Action**: Helpers called via command substitution should return non-zero and print only diagnostics that are safe to capture. Emit machine-readable gate markers and perform `exit` at the call site in the main script context: `value=$(helper) || { echo "GATE: ..."; exit 2; }`.
- **Grounded in**: .tad/hooks/lib/runtime-freshness-verify.sh, .tad/evidence/reviews/blake/runtime-freshness-loop/code-review-r3.md

### macOS APFS pwd -P Preserves Input Case on Case-Insensitive Filesystems - 2026-06-10
- **Context**: Migration engine uses `cd "$dir" && pwd -P` to physically resolve paths for ZERO_TOUCH comparison. On macOS APFS (case-insensitive by default), `cd .tad/Project-Knowledge && pwd -P` was expected to return the canonical lowercase `project-knowledge` but instead returned `Project-Knowledge` (preserving the input case).
- **Discovery**: `pwd -P` resolves symlinks (e.g., `/var` → `/private/var`) but does NOT canonicalize case on APFS case-insensitive volumes. Two `cd` calls to the same directory with different case produce different `pwd -P` output strings, even though they refer to the same inode. This breaks string-based directory identity comparison for security-sensitive checks like ZERO_TOUCH protection — a case-variant path (`.tad/Project-Knowledge/x`) would bypass a check comparing against `pwd -P` of `.tad/project-knowledge`.
- **Action**: When comparing directory identity on potentially case-insensitive filesystems, normalize both `pwd -P` outputs to lowercase with `tr '[:upper:]' '[:lower:]'` before comparison. This is safe on case-sensitive Linux (paths already match exactly, lowercasing is a no-op for matching paths). For inode-level identity (strongest but not prefix-checkable), use `stat -f '%d:%i'` (macOS) or `stat -c '%d:%i'` (Linux).
- **Grounded in**: .tad/hooks/lib/migration-engine.sh check_zero_touch, HANDOFF-20260609-migration-engine-phase2.md code-review P1-2

### rm Chokepoint Constraint vs Temp File Cleanup Are Separate Concerns - 2026-06-10
- **Context**: Migration engine's single-rm-chokepoint (guarded_remove at L231) ensures all user file deletions flow through a single revalidated path. Phase 4 merge execution needs temp file cleanup (mktemp-created files) on pipeline failure (P1-1 from code review).
- **Discovery**: A `grep -c rm` constraint meant to enforce "single deletion chokepoint for user files" conflicts with `rm -f` for engine-created temp files. These are categorically different: guarded_remove deletes USER content (requires backup, revalidation, containment check); cleanup_merge_tmp deletes a TEMP FILE the engine itself just created via mktemp (never user content, no backup needed). The resolution: consolidate temp cleanup into a named helper (`cleanup_merge_tmp`) that clearly documents its scope, keeping the user-file chokepoint at exactly one location. The grep count changes from 1 to 2, but the INTENT (single chokepoint for user file deletion) is preserved.
- **Action**: When a "single chokepoint" constraint is enforced by grep-count, new code that needs the same primitive for a categorically different purpose should use a named helper to make the scope distinction explicit. Document the count delta in the completion report. Do not obfuscate (e.g., `/bin/rm` instead of `rm`) to game the count.
- **Grounded in**: .tad/hooks/lib/migration-engine.sh (guarded_remove L231, cleanup_merge_tmp L791), HANDOFF-20260610-merge-capability-phase4.md P1-1
