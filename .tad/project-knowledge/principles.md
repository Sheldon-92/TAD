Only truly mechanical logic (file I/O, config duplication) is safe to extract. Resolution (v2.8.1): commands consolidated into skills, single source of truth restored.

### YOLO Epic Execution: Cross-Model Audit Findings - 2026-05-15
- **Context**: YOLO mode executed a full Epic (5 capability pack builds + validation + freeze + cross-agent + template) in one session. Post-completion, Codex and Gemini independently audited the workflow (23/35 and 24/35 respectively).
- **Discovery**: (1) **Validation Theater** (Codex 3/5, Gemini 2/5): structural checks (grep, word count, install exit code, frontmatter) prove files exist correctly but do NOT prove the pack improves agent behavior. "13/13 installed" confirms file operations, not functional quality. (2) **Rule Soup / Context Saturation** (both 2/5 on scalability): 13 packs × 178 rules will choke reasoning at 50+ packs. Progressive disclosure (step4_5 max 2 packs) is the right direction but needs enforcement. (3) **Zero Collision Detection**: packs don't know about each other. A code-security rule could contradict a web-frontend performance rule with no mechanism to detect or resolve. (4) **Research evidence lacks auditability**: findings cite tool names and numbers but not source URLs or retrieval dates, making freshness/accuracy unverifiable.
- **Action**: (A) Add mandatory behavioral eval per pack: 3-5 before/after task comparisons with fixed rubric before marking accepted (Codex recommendation). (B) Implement pack collision detection: when ≥2 packs load, scan for contradicting rules across loaded references (Gemini recommendation). (C) Enforce max 2 packs per session via step4_5 guardrail — already exists, verify it's not bypassed. (D) Add source URLs + retrieval dates to research findings files.
- ⚠️ SAFETY ENTRY — requires human review for any modification

### Never Hand-Write What an Existing Tool Already Does — 2026-05-28
- **Context**: Installing TAD v2.18.0 to a new project (Colin声音项目). `tad.sh` failed due to interactive `/dev/tty` prompt in non-TTY context. Alex wrote a manual install script from memory instead of fixing `tad.sh` or reusing `*sync` logic.
- **Discovery**: (1) The manual script listed 12 of 32 `.tad/` subdirectories from memory, missing 14 directories (`hooks`, `codex`, `cross-model`, `github-registry`, `pair-testing`, etc.) and causing SessionStart hook errors + `/alex` unrecognized. (2) Skills copy also failed partially (9 of 34 installed) due to glob pattern issues with spaces in paths. (3) Root cause: bypassed two existing mechanisms (`tad.sh` installer + `*sync` process) that already have complete, tested file lists. Reimplemented from scratch → incomplete. (4) Post-install verification only checked version.txt + skill count, not a structural diff against the source — insufficient to catch missing directories.
- **Action**: (a) When an existing tool fails, fix the tool (e.g., add `--yes` flag to `tad.sh`) — don't rewrite it ad-hoc. (b) For new project installation, prefer `*sync` logic (which was just validated on 13 projects) over a one-off script. (c) Post-install verification must include structural comparison: `diff <(ls -d source/.tad/*/) <(ls -d target/.tad/*/)` to catch missing directories. (d) Future improvement: add `tad.sh --non-interactive` or `--yes` flag so it works in Claude Code terminal.
- ⚠️ SAFETY ENTRY — requires human review for any modification

### Rewiring a Gate's Prose Can Trip a `grep -c` SAFETY Count — Use Line-Set Diff + Re-cite the Constraint - 2026-05-31
- **Context**: research-engine-wire-phase4 rewired the Phase 0c/4c/5b adversarial-challenge gates in alex/SKILL.md from opt-in AskUserQuestion keystrokes to a complexity-derived `run_adversarial_challenge` boolean (DR-20260531 carve-out). The 3 gate-step headers each contained the substring "(NOT_via_alex_auto constraint)". The pre-impl AC4.5 baseline used `grep -c 'NOT_via_alex_auto\|forbidden_implementations'` = 17. Naively rewording the gate headers dropped the count to 14 — even though zero `forbidden_implementations` entries were illegitimately removed.
- **Discovery**: (1) A `grep -c` SAFETY baseline conflates two artifact classes that share a token: the load-bearing `forbidden_implementations` entries / `NOT_via_alex_auto: true` anchor (MUST be preserved) vs ordinary protocol gate-step PROSE labels that merely cite the constraint (which the handoff explicitly mandated rewiring). The count can't tell them apart. (2) The correct verifier is the AC4.5 bidirectional line-SET diff (`comm` on text-only lines), which makes the delta explainable: FORWARD-missing = exactly {the 3 sanctioned amendments} plus {the 3 mandated gate-prose rewordings}; REVERSE-added = the 1:1 replacements. (3) The cheap fix that keeps BOTH a clean line-set AND a stable count: when rewording a gate label that cites a constraint, RETAIN the constraint citation verbatim ("…NOT_via_alex_auto constraint, now satisfied via DR-20260531 carve-out") — the gate still enforces it, so the citation stays true and the substring survives the count.
- **Action**: For SAFETY/forbidden edits guarded by a substring count: (a) treat the count as a smoke alarm, the line-SET diff as ground truth; (b) classify each "missing" baseline line as forbidden-entry/anchor (must NOT change) vs prose-citation (may be reworded per handoff); (c) when rewording prose that cites a constraint, keep the constraint name in the new text so the count stays stable and the citation accurate; (d) always confirm the anchor with `grep -Fxq` (exact full-line) for byte-identity.
- ⚠️ SAFETY ENTRY — requires human review for any modification
- **Grounded in**: alex/SKILL.md cross_model_awareness L487/L488 + Phase 0c/4c/5b gate headers, DR-20260531-ar001-research-challenge-carveout.md, COMPLETION-20260531-research-engine-wire-phase4.md AC4.5 line-set results

### A Coverage Gate's Global-Count Floor Cannot Detect Must-Cover SAFETY Loss When Legit Stripping Also Lowers the Count - 2026-06-01
- **Context**: Gate 4 acceptance of the Codex-parity Phase-1 spike (Alex raw-recompute). The parity criterion's constraint layer checks `MUST|MANDATORY|VIOLATION` ≥ a source-derived floor. The regen PASSED (59 ≥ floor) but Gate-4 recompute showed 59 vs source 150 (~60% drop). Whole-protocol stripping (9 expected-absent Conductor protocols) legitimately removes many constraint lines, so a large global drop is EXPECTED. honest_partial 4→0 was fully legit (all 4 in stripped yolo); the cross_model AR-001 anchor survived. But forbidden_implementations 12→6 / anti_rationalization_registry 6→3 mix legit-stripped losses with POSSIBLE must-cover losses (express/experiment/cancel/step1c are NOT on the allowlist) — and the floor cannot tell them apart.
- **Discovery**: When a coverage/parity gate validates a "preserve these" inventory via a GLOBAL COUNT FLOOR, and the SAME artifact legitimately removes items from that same inventory elsewhere (stripped sections), the floor is structurally blind to must-cover loss: legit removal and illegit loss both reduce the one number, so any floor low enough to allow the legit removal also passes a regen that dropped a must-cover SAFETY constraint. The fix is per-CATEGORY / per-SECTION presence checks scoped to the must-cover set, NOT a global tally. Generalizes to any "strip-then-verify-preservation" pipeline: verification must be scoped to the kept region, because a global metric conflates the two populations the strip deliberately separates.
- **Action**: For any gate verifying a preserve-list across an artifact that ALSO legitimately drops items from that list: verify presence per-category within the must-cover scope, never via a global count floor. When designing such a gate, first ask "does the same artifact legitimately remove items I'm counting?" — if yes, a floor is invalid by construction. (Applied: EPIC codex-edition-parity P3 gate AC requires per-SAFETY-category preservation; P2 must trace each must-cover constraint's survival.)
- ⚠️ SAFETY ENTRY — requires human review for any modification
- **Grounded in**: HANDOFF-20260601-codex-parity-phase1-spike.md gate4_delta, EPIC-20260601-codex-edition-parity.md Context-for-Next-Phase

### Deny-List Beats Allow-List for Sync Sets; Version Grep Must Scope to git-ls-files; diff-r is the Universal Omission Catcher - 2026-06-01
- **Context**: TAD publish/sync had recurring silent-omission failures — `.tad/codex/` (+~11 other framework dirs) never in the hardcoded 14-dir sync allow-list → downstream codex frozen a month; tad.sh stuck at 2.19.1 (not in the 18-item version-string list). One root cause: a hardcoded list goes stale when structure evolves. Fixed by EPIC-20260601-self-deriving-release-sync P1 (deny-list derivation + structure-agnostic verification gates), built YOLO.
- **Discovery**: (1) **An ALLOW-list of framework paths fails CLOSED-WRONG (silent omission) when a new dir appears; a DENY-list fails OPEN-SAFE** (new dir → defaults to SYNC + REPORTED → auditable). For a sync set that grows over time, deny-list is the correct direction. BUT its safety rests entirely on the EXCLUSION path: a broken `grep -vxE` leaking a zero-touch dir (active/evidence/project-knowledge) into the sync set CLOBBERS downstream project data — WORSE than the omission. So the load-bearing AC is the EXCLUSION assertion (`--dirs | grep -cxE '<deny>' == 0`), not the inclusion test. (2) **A version-consistency grep MUST scope to `git ls-files`, not a raw FS walk** — raw walk picks up gitignored ephemera (worktrees, backups, bundles): real run OLD=2.19.0 went 62→5 after scoping to tracked files (88% noise). 88%-noise gate trains the operator to ignore it = the failure it exists to prevent. (3) **Prefer FALSE-POSITIVE over false-negative in a version gate** — over-report a historical ref rather than silently ship a stale one; narrow historical-exclusion table-row regex to a LEADING version cell so a live `| Install | x==9.9 |` is still CAUGHT. (4) **A pinned dir count (`wc -l == 20`) is ITSELF a hardcoded list** — assert set-equality (`diff <(--dirs) <(live ls-minus-deny)`) computed live instead. (5) **`diff -rq source==target` is the universal structure-agnostic omission catcher** — doesn't care how structure evolved; it caught that an inline while-loop silently synced only 1 of 14 projects in the v2.21.0 sync.
- **Action**: For any growing sync/distribution set: deny-list not allow-list; make the EXCLUSION assertion the load-bearing AC. For version checks: scope to `git ls-files`; prefer false-positive; narrow historical-exclusion to leading-cell. Never pin an absolute count — assert set-equality live. Use `diff -r source==target` as the final "did the copy land completely" verification.

### Deny-List Must Be Applied at EVERY Copy Granularity, and Verifiers Must Match Each Granularity — 2026-06-01
- **Context**: P2 of the self-deriving-release-sync Epic killed the hardcoded 14-DIR allow-list in tad.sh (deny-list derivation). But a SECOND hardcoded allow-list survived one layer down: the top-level FILE copy used a fixed extension glob `*.yaml *.md *.txt`, which silently dropped `.tad/portable-extract.sh` (git-tracked, runbook-referenced) and any future top-level `.sh`/`.json`. Both the installer self-check AND P1's `release-verify.sh structural` verified DIRS only → the omission was invisible to every gate. Impl-review arch-P1-1 caught it; this fix converted the file copy to deny-list (`every regular top-level file MINUS TAD_TOP_DENY`) and extended the verifier to top-level files.
- **Discovery**: (1) **Fixing the deny-list at ONE copy granularity (dirs) does not fix it at OTHERS (top-level files, and potentially `.claude/skills` globs)** — the omission disease relocates to whatever copy path is still pattern/extension-based. An extension allow-list (`*.yaml *.md *.txt`) is the SAME failure class as a dir allow-list, just one layer down. Audit EVERY copy loop for "enumerate-by-pattern" vs "derive-by-deny". (2) **A verifier is only as good as the granularity it inspects** — a dir-only `diff`/presence check reports PASS while a top-level framework file is missing. When you add a new copy granularity, you MUST add a matching verification granularity, or the gate is silently blind. (3) **Presence + non-empty ≠ content-complete** — a dir that copied 1-of-50 files passes a presence check; only `diff -rq source target` (source is LOCAL at install time) catches PARTIAL copies. The empty-dir dotfile bug passed presence-only earlier ONLY because it went fully empty; a partial copy would have slipped through. (4) **Drive cross-file drift checks off a PUBLIC FLAG interface, not internal var-name scraping** — tad.sh's inline-DENY_LIST drift check awk-scraped the lib's `ZERO_TOUCH=`/`TRANSIENT=` assignment lines (fragile to benign rename/fold/re-quote). Adding a `--transient` flag (mirroring `--zero-touch`) lets the check call `bash lib --zero-touch; bash lib --transient` — coupled to the lib's stable contract, not its byte layout. (5) **Self-check `return 1` under `main`'s `set -e` + `trap ERR` IS fatal (rolls back)** — a header comment claiming "non-fatal, ERR trap NOT triggered" was wrong and would mislead maintainers; the fatal-on-broken-source behavior is correct, the comment just has to describe it accurately.
- **Action**: When converting an allow-list to a deny-list, sweep ALL sibling copy paths at every granularity (dirs, top-level files, nested globs) — fix them symmetrically or the disease relocates. For every copy granularity add a matching verifier granularity. Use `diff -rq source target` (not presence-only) for content-completeness when the source is local. Expose lib internals as FLAGS for any cross-file drift check rather than scraping variable names. Keep self-check fatality comments accurate to the actual `set -e`/trap behavior.
- ⚠️ SAFETY ENTRY — requires human review for any modification
