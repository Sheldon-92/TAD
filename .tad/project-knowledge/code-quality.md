### Recurring failure: tsc missing type ‚Äî 2026-05-19
- **Context**: Detected by dream-scanner from v2 trace analysis. Pattern 'tsc: missing type' appeared in ‚â•2 reflexion_diagnosis events across TAD handoff cycles.
- **Discovery**: TypeScript type-checking failures (missing type declarations) are a recurring root cause of Blake Layer 1 failures. This pattern triggers reflexion cycles that could be avoided with upfront type completeness checks in handoff ACs.
- **Action**: When writing handoff ACs for TypeScript projects, include an explicit AC: "npx tsc --noEmit passes with zero errors before Layer 1." Consider adding type coverage threshold to Gate 3 v2 checklist.

### AC grep-count for reference-based pack files ‚Äî 2026-05-27
- **Context**: Upgrading video-creation pack with vimax-patterns.md. Alex wrote AC8/AC13 as `grep -c 'vimax-patterns.md' SKILL.md` expecting `= 1`.
- **Discovery**: Reference filenames in reference-based capability packs naturally appear in 2 locations: (1) Context Detection table row, (2) Quick Rule Index section heading. All existing references follow this pattern (storytelling.md, audio-design.md, etc.). ACs using `grep -c 'filename'` should expect `= 2`, not `= 1`.
- **Action**: When writing ACs for pack reference additions, use `grep -c 'filename' SKILL.md` with expected `= 2` (or `‚â• 1` if only checking existence). Dry-run the grep against an existing reference filename to confirm the expected count before shipping handoff.

### Bash heredoc Python injection via unvalidated CLI args ‚Äî 2026-05-28
- **Context**: Phase 4 academic-search.sh script. Code-reviewer P0: `--limit` parameter assigned from user input (`LIMIT="$2"`) without integer validation, then interpolated into Python heredocs via `${LIMIT}` inside `python3 -c "... [:${LIMIT}]"`.
- **Discovery**: When bash `${VAR}` is expanded inside a heredoc containing embedded Python/awk/perl code, the variable content becomes part of the target language's source code. A malicious `--limit` value like `5]; import os; os.system("id"); x=[0` would execute arbitrary Python. This applies to any bash script that interpolates user input into embedded language blocks.
- **Action**: Always validate CLI numeric arguments with `[[ "$var" =~ ^[0-9]+$ ]]` before interpolating into any embedded language (Python/awk/perl). For string arguments, prefer passing via environment variable (`LIMIT=$var python3 -c "import os; limit=int(os.environ['LIMIT'])"`) instead of heredoc interpolation.
- **Grounded in**: .tad/evidence/reviews/blake/academic-research-pack-phase4/code-review.md (P0-1)

### Heredoc "injection" depends on the SINK: file-write ‚âÝ interpreter-exec ‚Äî 2026-05-31
- **Context**: bugfix-dream-scanner-override-content. code-reviewer raised a P0: routing human-authored `.chosen`/`.rationale` through the unquoted heredoc `cat > "$file" <<CAND_EOF ... ${discovery} ... CAND_EOF` enables command injection (a rationale containing `` `id` `` or `$(date)` would execute). Empirical dry-run **refuted** it: `$(date)`/`` `id` ``/`$HOME` all emitted verbatim, nothing ran. The reviewer withdrew the P0.
- **Discovery**: Two heredoc cases look identical but have opposite risk. (1) **Embedded-interpreter heredoc** (`python3 -c <<EOF ... ${VAR} ... EOF`, see "Bash heredoc Python injection" entry 2026-05-28) ‚Äî the expanded value becomes SOURCE CODE the interpreter then EXECUTES ‚Üí genuine injection. (2) **File-write heredoc** (`cat > f <<EOF ... ${VAR} ... EOF`) ‚Äî the value is inserted into the output STREAM as data and **never re-scanned**. Shell expansion is NOT recursive: the unquoted heredoc expands its own literal `${VAR}` token exactly once; `$(...)`/backticks/`$OTHER` arriving INSIDE that value are not re-evaluated, and a literal delimiter line inside the value does not terminate the heredoc (terminator matches input lines only). The real (minor) residual for a file-write heredoc is NEWLINES ‚Äî `jq -r` renders JSON `\n` as a real newline ‚Üí multi-line body; flatten with `gsub("\n";" ")` inside jq.
- **Action**: When reviewing/triaging a heredoc "injection" finding, FIRST ask "does the sink EXECUTE the output (interpreter -c, eval, source) or just WRITE it (cat > file, tee)?" Only the executing sink is injection. Demand an empirical reproduction string before accepting a P0 whose fix expands scope ‚Äî mental simulation of shell expansion is insufficient (the reviewer itself flagged this as the documented failure mode). For file-write heredocs the correct hardening is newline-flatten, not a quoted-delimiter rewrite.
- **Grounded in**: .tad/evidence/reviews/blake/bugfix-dream-scanner-override-content/code-reviewer.md (P0 raised ‚Üí empirically refuted ‚Üí withdrawn), .tad/hooks/lib/dream-scanner.sh Pass C
### mikefarah yq `-i` Normalizes Once Then Is Idempotent ‚Äî Plan the One-Time Reformat ‚Äî 2026-05-31
- **Context**: research-engine-wire-phase4 ¬ß4.2/¬ß4.3 mandated structure-aware REGISTRY.yaml edits via `yq -i` (per-entry, atomic temp+mv) with an AC4.6 requirement that all non-edited entries stay byte-identical. First `yq -i` touch produced a 43-45 line diff (stripped blank lines, normalized inline-comment spacing, re-folded long multiline strings) ‚Äî NOT the single targeted status line.
- **Discovery**: mikefarah yq v4 reformats the WHOLE file on its first write, then is byte-stable: every subsequent `yq -i` edit changes ONLY the targeted node. There is no reliable blank-line/comment-preservation flag in yq v4. So a byte-identical-others AC is satisfiable ONLY relative to an already-yq-normalized file. The clean resolution: perform the one-time normalization as part of a mandated edit (here, the ¬ß4.3 archive edit), after which any recurring automated editor (the SessionStart dormant hook) produces byte-surgical diffs forever. Verify idempotency explicitly: normalize a temp copy, snapshot, run a second edit, `diff` must show only the one line.
- **Action**: When a handoff mandates yq for recurring structure-aware edits AND requires byte-identity of untouched entries, (1) expect+accept a one-time whole-file normalization, (2) trigger it via a single mandated edit up front, (3) prove idempotency with a normalize‚Üísnapshot‚Üíedit‚Üídiff test, (4) run the AC4.6-style byte test against a copy of the NORMALIZED file, not the raw original. Never reach for line-based sed to dodge the reformat ‚Äî that reintroduces the multi-entry corruption risk the yq mandate exists to prevent.
- **Grounded in**: .tad/hooks/lib/notebook-lifecycle.sh, .tad/evidence/acceptance-tests/research-engine-wire-phase4/dormant-recompute-smoke.md

### Line-Anchored blockquote‚Üícol-0 sed Does NOT Fix Single-Line `CONSUMES ‚Ä¶ PRODUCES` Blockquotes ‚Äî 2026-05-31
- **Context**: tad-lean-trustworthy-phase2 Step 1b converted blockquote `> **CONSUMES**:`/`> **PRODUCES**:` to col-0 across academic-research, ml-training, video-creation so scan-packs' `^\*\*CONSUMES\*\*:` col-0 grep would index real values. academic-research & ml-training carry CONSUMES and PRODUCES on SEPARATE blockquote lines ‚Üí both converted cleanly. video-creation carries BOTH on ONE line (`> **CONSUMES**: ‚Ä¶ **PRODUCES**: ‚Ä¶`).
- **Discovery**: A line-anchored `sed 's/^> \*\*CONSUMES\*\*:/.../'` strips the leading `> ` so the line starts col-0 with `**CONSUMES**:` (now matched) ‚Äî but the `**PRODUCES**:` token sits MID-LINE on that same line, so scan-packs' `grep -m1 '^\*\*PRODUCES\*\*:'` still misses it ‚Üí produces stays "Not specified". This is NOT a regression when the committed registry already had produces "Not specified" for that pack (video-creation did), but it IS a silent gap: a single-line two-marker blockquote can only ever yield ONE col-0 marker via line-anchored sed. To get BOTH indexed, the markers must be on their own lines (split into two col-0 lines), not just de-blockquoted. The AC2.6 "no-degradation" check (committed vs post-scan `grep -c 'Not specified'`) correctly passed because the count went 2‚Üí1 (improved), masking that video-creation's produces is still unindexed.
- **Action**: When converting blockquote pack metadata to col-0 for scan-packs indexing, first check whether CONSUMES and PRODUCES share a line. If so, SPLIT them onto separate col-0 lines (not just strip `> `), or scan-packs will only index the first. For AC verification prefer a per-marker grep (`grep -c '^\*\*PRODUCES\*\*:' CAPABILITY.md` == 1 per pack) over a registry-wide `Not specified` count, which an improvement elsewhere can mask.
- **Grounded in**: .tad/capability-packs/video-creation/CAPABILITY.md:12, .tad/scripts/scan-packs.sh:79-85 (extract_produces), COMPLETION-20260531-tad-lean-trustworthy-phase2.md Step 1b note

### ¬ß9.1 Region Marker is `### 9.1` (3-hash subsection), NOT `## 9.1` ‚Äî 2026-05-31
- **Context**: P4 verify-ac-commands.sh linter. Spec text said "extract the ¬ß9.1 region from `## 9.1` to the next `## `". Built the awk that way; the self-dogfood (AC4.6) on the archived vimax handoff returned 0 findings ‚Äî missing the very `grep -ocE` AC15 bug it was designed to catch.
- **Discovery**: Real TAD handoffs mark ¬ß9.1 as `### 9.1 Spec Compliance Checklist` (THREE hashes ‚Äî a subsection nested under `## 9. Acceptance Criteria`), and ¬ß9.1 ends at the next `### 9.2` (also 3-hash) or the next `## 10`. A `^## *9\.1` matcher finds NOTHING because the top-level ¬ß9 heading is `## 9. Acceptance Criteria`, with `9.1` only ever appearing at `### ` depth. The spec's `## 9.1` was an abbreviation, not the literal on-disk format. A linter/parser keyed to the wrong heading depth silently scans an empty region and reports clean ‚Äî a false-negative that looks like success.
- **Action**: When extracting a numbered sub-section region (¬ßN.M) from a TAD handoff, match `^#{2,} *N\.M([^0-9]|$)` (any heading depth ‚â•2) and terminate at the next `^#{2,} ` heading ‚Äî do NOT hard-code `## N.M`. ALWAYS validate region-extraction by dogfooding on a real artifact known to contain the target pattern; "0 findings" on a fixture that should fire is the tell for a wrong-depth region matcher.

### Behavioral-Fixture Discrimination: Anti-Slop = Threshold/Named-Rule, Not Vocabulary ‚Äî 2026-05-31
- **Context**: P5 authored 14 behavioral fixtures (1 per installed capability pack) for `pack-eval-runner.sh`. Each fixture greps a captured agent output for "markers" proving the agent APPLIED a pack rule. The anti-gaming risk: a marker a no-pack frontier LLM would also emit makes the fixture validation theater (architecture.md YOLO audit).
- **Discovery**: The reliable discriminator is the SAME as the anti-slop formula (architecture.md "Capability Pack Quality Bar"): a marker is pack-specific iff it is a NAMED rule or a SPECIFIC threshold/number the pack introduces ‚Äî NOT a domain noun. Concrete contrast that survived spot-check: ‚úÖ "self-enhancement bias 10-15%", "TruffleHog exit 183", "ACX RMS -23 to -18 dB", "n=550 a11y / 30-50%", "Four-Gate fastest-fail-first", "Judge ‚âÝ Optimizer", "60-30-10 + 3-level token". ‚ùå "make it secure", "add more tests", "use a TTS tool", or any word from the input scenario. Two fast sanity gates per fixture: (1) every Verification Command MUST be `grep -oE 'a|b|c' {out} | sort -u | wc -l` ‚Äî NEVER `grep -c` (P4 lint Rule A: grep -c + sort -u | wc -l always returns 1). (2) every fixture needs ‚â•1 `[structural]` marker (proves the agent produced the rule's OUTPUT SHAPE, e.g. per-photo first_frame/last_frame decomposition or D1-D10 decision IDs, not just mentioned the vocabulary). The ‚ùå list in the Anti-Slop Check is load-bearing: it documents the generic terms deliberately EXCLUDED from the grep pattern.
- **Action**: When authoring a behavioral fixture, pull markers from the pack's specific numbers/named-rules/exit-codes/thresholds, never domain nouns or input words. Use `grep -oE ‚Ä¶ | sort -u | wc -l`. Require ‚â•1 [structural] marker that verifies output SHAPE. Self-check each marker: "would an agent WITHOUT this pack emit this?" ‚Äî if yes, replace it.
- **Grounded in**: .tad/scripts/pack-eval-runner.sh, .claude/skills/*/examples/*.md (16 fixtures), .tad/templates/pack-example-fixture.md, COMPLETION-20260531-tad-lean-trustworthy-phase5.md AC5.3

### Auto-Generated Registry ‚Üí Persisted Decision State Belongs in a Side-File ‚Äî 2026-05-31
- **Context**: P5 needed a `behaviorally_verified` flag per capability pack. The natural home (pack-registry.yaml) is regenerated by scan-packs.sh.

- **Action**: Before adding any persisted flag to an auto-generated file, check whether the generator does a full regen (`cat >`/full overwrite). If so, put the flag in a name-keyed side-file with explicit status semantics, not in the regenerated artifact.

### A Behavioral-Eval Gate Must Run on a SEPARATE Discriminative Field, Not the Combined Marker Pattern ‚Äî 2026-05-31
- **Context**: Gate 3 Layer 2 review of `pack-eval-runner.sh` (P5 follow-up). The runner gated PASS on `min_marker_count` over the fixture's `## Verification Command` grep pattern. That pattern was authored to MIX pack-specific markers WITH generic ones (severity tags `[P0]/[P1]/[P2]`, generic stats `n=20`/`confidence interval`, and nouns any senior practitioner emits). Proof of the flaw: the committed ai-evaluation CONTROL (a review written with NO pack loaded) scored **3/3 PASS** on the combined gate ‚Äî pure validation theater (it matched only `confidence interval`, `n ‚â• 100`, `n=20`).
- **Discovery**: The previous entry got the MARKER-AUTHORING right (named-rule/threshold, not vocabulary) but the GATE was still wired to the contaminated combined count. Having a correct ‚ùå-exclusion list in the Anti-Slop Check is necessary but NOT sufficient ‚Äî those generic markers were still PRESENT in the live grep pattern, so the gate still counted them. The fix is structural: the gate needs its OWN field. Add `discriminative_pattern` (a `grep -oE` alternation of ONLY the ‚úÖ pack-specific markers) + `min_discriminative` to frontmatter; the runner PASSes iff `disc = grep -oE "$discriminative_pattern" out | sort -u | wc -l ‚â• min_discriminative`, and reports the combined count as a SECONDARY context number only. After the fix the same CONTROL scores **disc 0/3 ‚Üí FAIL** while WITH scores **disc 5/3 ‚Üí PASS** ‚Äî the gate now discriminates. A false PASS is worse than a false FAIL here, so when unsure make the discriminative pattern TIGHTER (fewer, more pack-unique terms). Thin packs justify `min_discriminative: 2`; the all-patterns photo fixture uses 4. Two macOS gotchas surfaced: (1) YAML double-quoted patterns escape `\` as `\\` (e.g. `DESIGN\\.md`) ‚Äî the parser must collapse `\\`‚Üí`\` after stripping quotes, else the regex over-matches; (2) keep the combined-count fallback + WARN for any fixture lacking `discriminative_pattern` so the runner stays backward-compatible and never fail-closed.
- **Action**: For any marker-based behavioral/eval gate: separate the PASS-driving field (discriminative, pack-only markers) from any human-facing combined/context count. Never let the gate count generic markers even if they live in the same fixture. Prove the discrimination with a NEGATIVE control (no-pack output) that must FAIL ‚Äî a gate that only PASSes the positive case is unproven. Collapse YAML `\\`‚Üí`\` when reading regex patterns from double-quoted frontmatter.
- **Grounded in**: .tad/scripts/pack-eval-runner.sh (parse_disc_pattern / assert_one PRIMARY path), .claude/skills/*/examples/*.md (16 fixtures, `discriminative_pattern`), .tad/evidence/pack-eval/2026-05-31/ai-evaluation-{WITH,CONTROL}.md, COMPLETION-20260531-tad-lean-trustworthy-phase5.md (Discriminative-Gate Fix follow-up)

### `comm -12` Set-Intersection of CJK Keyword Lists Needs `LC_ALL=C` on BOTH sorts AND the comm ‚Äî 2026-05-31
- **Context**: pack-collision detector (scan-collisions.sh) keyword PRE-FILTER intersected two packs' keyword sets via `comm -12 <(... | sort -u) <(... | sort -u)` to decide which pack PAIRS to scan. Code-reviewer P1-1: under the ambient `en_US.UTF-8` locale this produced a PHANTOM shared keyword (`ËÆæËÆ°`) for `web-ui-design √ó video-creation`, two packs that share ZERO actual keywords ‚Äî admitting an un-pre-filtered pair that became the only false-positive candidate.
- **Discovery**: `comm` requires its two inputs to be sorted in the SAME collation it uses internally ‚Äî and `comm` collates by BYTE (C locale). But `sort` without `LC_ALL=C` uses the ambient locale's collation, which orders CJK/multibyte strings differently than byte order. The mismatch makes `comm` mis-pair lines and report a spurious intersection (or miss a real one). This is the same locale-collation class as the project's BSD-portability lessons. The fix is to force byte collation on EVERY participant: `LC_ALL=C comm -12 <(... | LC_ALL=C sort -u) <(... | LC_ALL=C sort -u)` ‚Äî the comm AND both sorts, not just one. With `LC_ALL=C` the `web-ui-design √ó video-creation` intersection is correctly empty and the pair is no longer scanned. (Verified: candidate set dropped the video-creation false positive; the 3 real collisions remained.) A downstream consequence worth noting: a `grep -c 'ËÆæËÆ°' <other-set>` = 0 sanity check is the cheap way to PROVE a phantom intersection ‚Äî if comm claims a shared token that grep can't find in one input, the collation is the culprit.
- **Action**: Any `comm` over text that MAY contain non-ASCII (CJK keywords, accented strings, emoji) MUST set `LC_ALL=C` on the comm and on both feeding `sort`s. When a set-intersection pre-filter admits a pair that "shouldn't share anything", first suspect locale collation, not the matching logic ‚Äî verify with `grep -c '<claimed-token>'` against the other set (0 hits ‚áí phantom intersection ‚áí add `LC_ALL=C`).
- **Grounded in**: .tad/scripts/scan-collisions.sh (shared-keyword pre-filter), .tad/evidence/yolo/pack-collision-detection/phase1-impl-review-cr.md (P1-1), COMPLETION-20260531-pack-collision-detection-phase1.md
