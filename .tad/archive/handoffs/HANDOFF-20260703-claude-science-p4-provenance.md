## 1. Task Overview

### 1.1 What We're Building
Add a structured `## Provenance` section to the completion report template. Blake fills this during implementation to track how key artifacts were produced. Gate 3 checklist updated to verify it's non-empty.

### 1.2 Why We're Building It
**业务价值**：Reproducibility — when a future session needs to regenerate or understand an artifact, the provenance section tells exactly what command, sub-agent, and environment produced it. This is Claude Science's core differentiator ("every figure carries its full history") adapted to TAD's lightweight model.
**成功的样子**：Every future completion report has a provenance table showing artifact → generation method, and Gate 3 checks it.

### 1.3 Intent Statement

**真正要解决的问题**：Current completion reports say WHAT was done but not HOW each artifact was generated. "Created audit-report.md" tells nothing about whether it was hand-written, script-generated, or sub-agent output. Provenance captures the generation method so it can be reproduced.

**不是要做的**：
- ❌ 不是 full lineage tracking system (no DAG, no dependency graph)
- ❌ 不是 retroactive provenance for existing completions
- ❌ 不是 environment snapshotting (no Docker image, no pip freeze)
- ❌ 不是改 completion report's existing sections (only ADD a new section)

## 2. Technical Design

### 2.1 New Section: `## Provenance`

Insert AFTER the `## 📋 实施总结` section (after its closing `---` separator) and BEFORE the `## 🧪 测试证据` header. Anchor on heading text, not line numbers.

**Note on Sub-Agent overlap**: The template already has `## 🤝 Sub-Agent 使用記録` which tracks per-SESSION sub-agent usage ("which agents did I invoke?"). The new Provenance section tracks per-ARTIFACT generation method ("which agent/tool produced THIS specific file?"). Both are filled; they are complementary, not redundant.

```markdown
## 🔗 Provenance (Artifact Generation Record)

> How key artifacts were produced. Fill for every created/modified file
> that is NOT a simple copy or trivial edit.

| Artifact | Generation Method | Sub-agent | Notes |
|----------|------------------|-----------|-------|
| {file path} | {command, script, or "hand-written"} | {sub-agent name or "direct"} | {env notes, version, config} |

**Examples:**
| Artifact | Generation Method | Sub-agent | Notes |
|----------|------------------|-----------|-------|
| .tad/evidence/acceptance-tests/p1/audit-report.md | `grep '^description:' .claude/skills/*/SKILL.md` + manual comparison | direct | bash one-liner + manual formatting |
| .tad/eval/pack-discovery-eval.md | Manual structured test per handoff §3.2 | direct | 12-case fixture, self-graded |
| .claude/skills/alex/references/intent-router-protocol.md | Edit tool — 3 text replacements per handoff §2.1-2.2 | direct | protocol text only |

**Rules:**
- Fill for every CREATE/MODIFY file in the File Manifest
- "Generation Method" = the actual command or tool used, not just "implemented"
- "Sub-agent" = which sub-agent produced it, or "direct" if Blake did it directly
- "Notes" = anything needed to reproduce: version flags, config params, env constraints
- Trivial copies (parity files, git ops) can be grouped: "Same as primary via `cp`"
- If a file was generated by a script, cite the script path
```
