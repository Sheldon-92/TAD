## 1. Task Overview

### 1.1 What We're Building

Three interconnected deliverables that close the "publish without manifest" gap:

1. **release-verify.sh `migration` mode** — a new case arm that detects D (deleted) and R (renamed) files between the previous git tag and HEAD, cross-references them against `.tad/migrations/` manifests, and reports any unmanifested removals.

2. **migration-draft.sh** — a standalone helper script that generates draft migration manifests from `git diff --name-status` between any two adjacent tags. Outputs schema-v1 YAML to `.tad/migrations/`. Drafts require human review before committing.

3. **Historical manifests** — 12 manifest files covering v2.19.0-to-v2.19.1 through v2.25.0-to-v2.26.0, generated by migration-draft.sh and human-reviewed.

Plus one integration point:
4. **publish-protocol.md update** — insert the migration gate call between the existing version gate (step3c) and the Confirm & Execute step (step4).

### 1.2 Why We're Building It

**Business value**: Ensures every future TAD release that removes or renames files ships with a corresponding migration manifest, so downstream projects upgrading via `tad.sh` or `*sync` never encounter "file was deleted upstream but no manifest told the engine to clean it up" drift.

**User benefit**: The publish gate catches missing manifests before they reach 14+ registered projects, preventing the retroactive cleanup that motivated this entire Epic.

**Success looks like**: When a developer deletes a framework file and runs `*publish`, the migration gate warns (or blocks, after Phase 6 activation) with a clear list of unmanifested deletions, and the developer creates the manifest before shipping.

### 1.3 Intent Statement

**The real problem**: TAD has no mechanism to ensure that file deletions/renames in a new version are accompanied by migration manifests. A developer can delete `.claude/skills/foo/bar.md`, publish a new tag, and every downstream project retains the stale file forever because no manifest told the engine to clean it.

**Not what we're doing**:
- NOT implementing hard blocking (Phase 6 activates that after the interception drill)
- NOT modifying migration-engine.sh (Phase 2/3/4 already delivered that)
- NOT modifying tad.sh (Phase 3 already delivered that)
- NOT running historical manifests against real projects (Phase 6 does that)

1. **Scope to git ls-files** (from principles.md)
   - Problem: Raw FS-walk picked up gitignored ephemera (.tad.backup.*, worktrees, bundles) — 88% noise on a real dry-run.
   - Solution: Always scope to `git ls-files` for version/migration checks. Fall back to FS-walk with --exclude-dir only when not in a git repo.

   - Problem: Silent false-negative (missed stale ref / missed unmanifested deletion) ships a broken release.

## 3. Requirements

### 3.1 Functional Requirements

**FR1: release-verify.sh `migration` mode**
- FR1.1: New case arm `migration <repo_root>` in release-verify.sh
- FR1.2: Detect previous tag automatically: `git describe --tags --abbrev=0 HEAD^` (most recent tag before HEAD). If no previous tag exists, exit 0 with "no previous tag — nothing to check".
- FR1.3: Compute D (deleted) and R (renamed) files between prev_tag and HEAD, scoped to framework-managed paths (`.tad/`, `.claude/`, `.codex/`, `.agents/`, root files `CLAUDE.md`, `AGENTS.md`, `tad.sh`).
- FR1.4: For each D entry: check if a manifest `{from}-to-{to}.yaml` exists covering that version pair AND contains the deleted path in its `delete` section. "from" = prev_tag version (strip leading `v`), "to" = the version being released (read from `.tad/version.txt` or passed as argument).
- FR1.5: For each R entry from git diff -M: check if manifest covers it in the `rename` section.
- FR1.6: For files that appear as D in git diff but might actually be renames (git did not detect with -M): run a SECONDARY rename detection pass. For each D file, check if any A (added) file in the same diff has a similar basename or is in the same directory. Report these as "POSSIBLE RENAME (needs manifest rename entry or explicit delete entry)". Prefer false-positive here.
- FR1.7: Files inside ZERO_TOUCH directories (read from `derive-sync-set.sh --zero-touch`) are EXCLUDED from the check — changes in those directories are user-owned, not framework-managed.
- FR1.8: Exit codes follow the existing contract: 0=all covered, 1=unmanifested removals found, 2=usage/wiring error.
- FR1.9: Gate behavior controlled by `TAD_RELEASE_GATE` env var. Default = `warn` (report but exit 0). When unset or set to anything other than `warn`, exit 1 on findings. This means Phase 5 ships in WARN mode by default; Phase 6 activates hard-block by removing the warn override.
  - CORRECTION on exit code semantics: In warn mode, the script STILL exits 1 (drift detected). The warn/block distinction is handled by the CALLER (publish-protocol gate step), consistent with how step3c handles version mode. The script itself always reports honestly. The publish-protocol step downgrades exit 1 to a warning when `TAD_RELEASE_GATE=warn`.
- FR1.10: Output format: list each unmanifested file with status (UNMANIFESTED DELETE / POSSIBLE RENAME / UNMANIFESTED RENAME), the file path, and which manifest file was checked.

**FR2: migration-draft.sh**
- FR2.1: Standalone script at `.tad/hooks/lib/migration-draft.sh`
- FR2.2: Usage: `bash migration-draft.sh <from_tag> <to_tag> [--output-dir <dir>]` (default output-dir = `.tad/migrations/`)
- FR2.3: Run `git diff --name-status -M <from_tag>..<to_tag>` scoped to framework-managed paths (same prefix set as FR1.3).
- FR2.4: For each D entry: emit a `delete` section entry with path, type (file/dir inferred from git status or defaulting to file), and reason placeholder.
- FR2.5: For each R entry (rename detected by git -M): emit a `rename` section entry with from/to paths, type, and reason placeholder.
- FR2.6: For each A (added) file that might be a rename partner of a D file (same basename, different directory): emit a COMMENT in the YAML noting the possible rename, but default to separate delete + no rename entry. Human reviewer decides.
- FR2.7: Always emit a `verify` section with `absent` checks for each deleted file and `present` checks for each added file that replaced a renamed file.
- FR2.8: Output filename: `{from}-to-{to}.yaml` where from/to are stripped of leading `v`.
- FR2.9: Emit `generated_by: "draft-script"` and `schema_version: 1`.
- FR2.10: Filter out paths inside ZERO_TOUCH directories (read from `derive-sync-set.sh --zero-touch`).
- FR2.11: If the output file already exists, refuse to overwrite. Print warning and exit 2.
- FR2.12: After generating, print a summary: N deletes, N renames, N possible-renames-flagged. Remind operator: "This is a DRAFT. Review and commit manually."

### 4.2 Component Specifications

#### 4.2.1 release-verify.sh `migration` case arm

**Interface**:
```
release-verify.sh migration <repo_root> [<expected_version>]
```
- `repo_root`: path to the TAD repo (must contain `.tad/` and `.tad/migrations/`)
- `expected_version` (optional): the version being released. If omitted, read from `$repo_root/.tad/version.txt` (first line, trimmed). Needed to locate the correct manifest file.

**Algorithm**:
1. Detect previous tag: `git -C "$REPO" describe --tags --abbrev=0 HEAD^ 2>/dev/null`. If no tag found, exit 0 ("no previous tag").
2. Extract prev_version from tag (strip leading `v`, normalize to 3-segment semver).
3. Read expected_version from argument or `.tad/version.txt` (add `.0` if only 2-segment like "2.27").
4. Compute framework-scoped diff: `git -C "$REPO" diff --name-status -M "$PREV_TAG"..HEAD -- .tad/ .claude/ .codex/ .agents/ CLAUDE.md AGENTS.md tad.sh`
5. Read ZERO_TOUCH dirs from `derive-sync-set.sh --zero-touch "$REPO"`.
6. Filter out lines where the path starts with `.tad/<zero_touch_dir>/`.
7. For each D entry: check manifest `$REPO/.tad/migrations/${prev_ver}-to-${exp_ver}.yaml` exists and contains the path in `delete:` section (grep for the path string in the file — not full YAML parse, since NFR4 prohibits yq).
8. For each R entry: check manifest rename section similarly.
9. Secondary rename detection: for each D without manifest coverage, check if any A entry shares the basename. Flag as "POSSIBLE RENAME".
10. Report all findings. Exit 1 if any unmanifested entries. Exit 0 if all covered or no D/R entries.

**YAML checking without yq**: Since NFR4 prohibits external dependencies, the manifest cross-reference uses `grep -F` for path presence. This is intentionally approximate — the gate is a smoke alarm, not a YAML parser. A path that appears in a comment would be a false-negative (path appears "covered" but isn't really in a delete entry). This is acceptable because: (a) manifests are small, human-reviewed files with minimal comments; (b) the consequence of a false-negative is that the gate passes when it should warn — the same outcome as not having the gate at all; (c) the alternative (requiring yq/python) violates NFR4 and the dependency-minimal philosophy of the hooks/lib/ scripts.

#### 4.2.2 migration-draft.sh

**Interface**:
```
migration-draft.sh <from_tag> <to_tag> [--output-dir <dir>]
```

**Algorithm**:
1. Validate args: both tags must exist in git (`git rev-parse --verify "$tag" >/dev/null 2>&1`).
2. Extract versions: strip `v` prefix.
3. Read ZERO_TOUCH dirs from `derive-sync-set.sh --zero-touch`.
4. Run: `git diff --name-status -M "$FROM_TAG".."$TO_TAG" -- .tad/ .claude/ .codex/ .agents/ CLAUDE.md AGENTS.md tad.sh`
5. Filter out ZERO_TOUCH paths.
6. Classify entries: D → delete, R → rename, A → track for possible-rename matching.
7. Secondary rename detection: for each D, check if any A has same basename. If yes, emit a YAML comment flagging the possible rename.
8. Emit YAML to `$OUTPUT_DIR/${from_ver}-to-${to_ver}.yaml`.
9. Print summary.

**YAML emission**: Use printf/heredoc to emit well-formed YAML. No YAML library needed — the schema is simple enough for template emission. Quote all string values (version numbers, paths, reasons).

### 4.3 Data Models

## 11. Learning Content

### 11.1 Decision Rationale: WARN mode default vs hard-block

**Chosen approach**: Default to `TAD_RELEASE_GATE=warn` in Phase 5. Phase 6 activates hard-block after the interception drill proves the gate works.

**Alternatives considered**:

| Approach | Pros | Cons | Why not chosen |
|------|------|------|-----------|
| **WARN default (chosen)** | Safe rollout; gate bugs don't block releases; real data collection before enforcement | Unmanifested deletions can still ship during warn period | Matches the v2.24 structural gate precedent; Phase 6 is specifically designed to flip to hard-block |
| Hard-block from day 1 | Immediate enforcement | A bug in the migration check (false positive) blocks all releases with no escape hatch | Too risky for a new gate; no precedent data |
| Environment-variable toggle per release | Maximum flexibility | Complexity; easy to forget to set; no clear "this is the permanent state" | Over-engineered for a binary transition |

**Core tradeoff**: Safety (catch missing manifests immediately) vs. stability (don't break the publish workflow with a new, untested gate).

**Learning point**: New gates should follow a shadow → enforce progression. Ship the detection logic first (shadow/warn), validate it catches real issues without false positives, then enable enforcement. This is the same pattern as feature flags in production deployments.
