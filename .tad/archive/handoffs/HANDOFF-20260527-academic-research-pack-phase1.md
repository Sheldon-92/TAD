## 4. Technical Design

### 4.1 Research Strategy

ScienceClaw has 5000+ files. Blake should NOT read all of them. Strategy:

**Layer 1 — Must Read (full content):**
- `SCIENCE.md` — master protocol (already partially analyzed)
- `skills/skill-evolution/SKILL.md` — how skills evolve
- `skills/skill-creator/` — all files (skill generation mechanism)
- `skills/find-skills/` — all files (skill routing/discovery)
- `extensions/memory-core/` — all files (base memory)
- `extensions/memory-lancedb/` — all files (vector storage)
- `src/memory/` — key index/interface files (not all 96)
- 15-20 representative skill SKILL.md files across different clusters

**Layer 1b — Must Read (runtime integration layer — P0 fix from expert review):**
- `src/context-engine/` — all 6 files (skill loading into agent context — determines if skills can be decoupled)
- `src/routing/` — key files: `resolve-route.ts`, `session-key.ts` (infrastructure-level request→skill routing)
- `src/agents/` — entry point files only: `index.ts`, key type definitions, skill-loader (the 847-file agent runtime is the integration layer that determines migration feasibility; read entry points, not all files)

**Layer 2 — Metadata Scan (list + first 5-10 lines):**
- All 285 `skills/*/` directories — enumerate, read SKILL.md first 10 lines for name/description
- `src/plugin-sdk/` — key files: `index.ts`, types, config-paths, channel-lifecycle (the SDK contract skills are built against — determines runtime dependencies)
- `src/hooks/` — understand hook patterns
- `packages/clawdbot/`, `packages/moltbot/` — skim for domain specialization patterns

**Layer 3 — Skip:**
- `apps/` (mobile/desktop apps — not relevant to pack architecture)
- `ui/` (web UI)
- `docs/zh-CN/` (translated docs)
- `extensions/` (non-memory channel integrations: slack, discord, telegram, etc.)
- `test/`, `scripts/`, `deploy/`, `.github/`
- `src/commands/` (CLI command definitions — not needed for pack mapping)

### 4.2 Output Documents

**Doc 1: `architecture-analysis.md`** — Full architecture analysis covering:
1. Skills System: how SKILL.md files are structured, loaded, discovered, and evolved at runtime
2. Memory System: data structures, storage backends, temporal decay, cross-session retrieval
3. Database Integration: how MCP servers and API wrappers connect academic databases
4. SCIENCE.md Protocol: enforcement mechanism, research phases, tool-call thresholds
5. Hook System: what hooks exist, what they enforce
6. Plugin/Extension System: how plugins are registered and loaded (informed by `src/plugin-sdk/`)
7. Agent Runtime + Context Engine: how skills are loaded into context (`src/context-engine/`), how requests route to skills (`src/routing/`), agent entry points (`src/agents/`). This section determines whether ScienceClaw skills can be decoupled from the OpenClaw runtime — critical for all migration decisions.
8. Runtime Coupling Assessment: which subsystems are tightly coupled to OpenClaw runtime (structurally unfixable in TAD) vs loosely coupled (portable)

**Doc 2: `skill-taxonomy.md`** — Complete enumeration with:

| # | Skill Name | Cluster | Description (from SKILL.md) | Migration Priority | Anti-Slop Score | TAD Mapping | Runtime Deps | Confidence |
|---|-----------|---------|----------------------------|-------------------|----------------|-------------|-------------|------------|

- Cluster: literature / analysis / statistics / visualization / writing / domain-natural / domain-social / meta / utility (Blake may propose additional clusters if distribution doesn't fit)
- Migration Priority: 1 (must-have) / 2 (important) / 3 (nice-to-have)
- Anti-Slop Score: H (has specific thresholds/numbers, e.g., "PRISMA 27-item checklist") / M (domain-specific but no hard numbers, e.g., "NIH grant section structure") / L (generic knowledge, e.g., "choose appropriate chart types")
- TAD Mapping: judgment-rule / executable-reference / skip (functionality covered by existing TAD pack or Claude native)
- Runtime Deps: none / memory-lancedb / plugin-sdk / mcp:{name} / code-execution / context-engine / cross-skill:{name} (what OpenClaw runtime services this skill depends on — determines migration feasibility)
- Confidence: high (deep-read) / low (metadata-only) — indicates how much Blake actually read

**Doc 3: `tad-mapping-blueprint.md`** — Concrete decisions:
1. Pack Architecture Type: reference-based / deep-skill / orchestration-router (with rationale)
2. Memory Strategy: evaluate on 5 axes (setup complexity, retrieval quality, cross-session persistence, TAD philosophy alignment, maintenance burden). Options: TAD file-based as default + optional LanceDB upgrade / LanceDB-first / NotebookLM as memory / hybrid. Note: TAD already HAS a memory system (project-knowledge + evidence + Knowledge Assessment) — adding a second creates maintenance surface.
3. Database Strategy: evaluate per-database (auth requirements, rate limits, cost). Options per DB: curl-based API wrapper (natural for Claude Code) / MCP server (for complex queries) / WebSearch fallback. Document: Semantic Scholar (free API, 100 req/5min), arXiv (1 req/3s), Google Scholar (NO API — SerpAPI $50/mo or WebSearch).
4. Skill Evolution Strategy: ⚠️ Runtime generation (ScienceClaw's skill-creator writing SKILL.md at runtime) is architecturally incompatible with Claude Code (stateless sessions, SKILL.md requires settings.json registration). Actual decision space: (a) Session-scoped generation (temporary, low value) / (b) Proposal-based via *optimize/*evolve with research-domain adapter + human gate approval (feasible, TAD-aligned) / (c) Fixed skill library from Phase 3 only.
5. Alex/Blake Role Mapping: how Alex designs research tasks, how Blake executes them, how Gates verify quality
6. Session Duration & Persistence Model: ScienceClaw uses 3600s timeout + heartbeat. Claude Code has no heartbeat — context compaction is the persistence strategy. Decision: single long session / multi-session with handoff checkpoints / Epic sub-phases for extended research.
7. Estimated effort: Phase 2-5 sizing based on actual code analysis (not README claims)
